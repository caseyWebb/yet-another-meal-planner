# note-visibility-tiers

## Why

Recipe notes today carry a binary `private` flag: a note is either author-only or visible to the entire deployment. Band 5 replaced "the entire deployment" with a visibility lens (deployment-profiles-and-visibility-lens): recipes are visible per household through friend links, with a curated floor and an anonymous bottom position. Notes were lens-bounded there only at the recipe level (`read_recipe_notes` is an enumerated D11 consumer — notes are unreachable for recipes outside the caller's lens), but the per-note flag still speaks the pre-lens language. D30-final ratifies the replacement: note tiers `public | friends | private`, default `friends`, no household tier, with visibility as a live lens. This change lands the tier model end-to-end — column, migration, read semantics, tool contract, composer UI, anonymous surface — as band 5's final change, after `households-friends-and-people-page` delivers the real friendship edges the friends tier reads.

## What Changes

- **`recipe_notes.tier`**: new column `tier TEXT CHECK (tier IN ('public','friends','private'))`; new notes default to `friends` (both deployment profiles, D30-final). Migration is a pure mapping, no surgery: `private = 1` → `tier = 'private'`, everything else → `tier = 'friends'`. The `private` column is retained but demoted to a legacy input; `tier` becomes the source of truth on every read and write.
- **Tiered read semantics through the lens** (D30-final): a note NEVER renders where its recipe isn't visible — the recipe's own lens bounds everything, in both member and anonymous positions. Within a visible recipe: `private` = author-only (member-level: the authoring member, not the tenant); `friends` = visible to the author's household and its friend households (under self-hosted, implicit all-to-all makes this everyone — reproducing today's non-private behavior exactly); `public` = anyone who can see the recipe, including the anonymous `/cookbook` surface but only where the recipe itself is anonymously visible. Public is the ONLY tier that ever reaches the anonymous surface.
- **Live-lens retroactivity** (D30/D30-final, stated in the spec): creating or severing a friendship, or changing a note's tier, re-evaluates note visibility immediately in both directions; a friends-tier note authored while friendless becomes visible to every future friend; a severed edge immediately hides.
- **`read_recipe_notes` returns `tier` and author `handle`** on every note (handle joined from the `members` table via the note's `author` member id — D10's founding-member invariant makes every legacy author value a real member row).
- **Write tools take `tier`**: `add_recipe_note` / `update_recipe_note` accept `tier` (`'public' | 'friends' | 'private'`); the legacy `private` boolean stays accepted as a deprecated alias (`true` → `private`, `false` → `friends`) so stale plugin bundles keep working. The member `/api` notes endpoints gain the same field. Note writes keep their existing D15 class (b) classification (idempotent upsert keyed by author + slug + client-minted `created_at`); a tier change on an existing note rides the same registered note-edit mutation — no new write class.
- **Three-state composer** (design request #9, local design authorized): the detail-page note composer replaces the Private checkbox with a Public / Friends / Private segmented control (the Time-filter segmented treatment), Friends pre-selected, one-line per-tier descriptions, conditional Public copy when the recipe isn't anonymously visible; the edit state gets the same control; rendered non-Friends notes carry a tier indicator chip (lock = Private, globe = Public; Friends renders unmarked).
- **Anonymous cookbook rendering**: the anonymous `/cookbook/<slug>` page — which renders no notes today — gains a notes section showing exactly the public-tier notes of anonymously-visible recipes, attributed by handle.
- **Docs/persona lockstep**: `docs/TOOLS.md` (notes tools tier param + tiered-read guarantees), `docs/SCHEMAS.md` (`recipe_notes.tier`), and the Appendix C band-5 persona edit bound to this change — notes flows mention tiers (D30) — plus `aubr build:plugin --check`.

## Capabilities

### New Capabilities

None — this change modifies existing capabilities only.

### Modified Capabilities

- `recipe-notes`: the primary delta. The `private` flag contract becomes the three-tier model: storage requirement gains `tier`; "Per-note privacy" becomes tiered visibility with live retroactivity; the cross-group read requirement becomes tier-and-lens-scoped and returns `tier` + author `handle`; the edit/delete requirement gains the `tier` param; a new requirement pins the anonymous-surface rule (public-only, recipe-lens-bounded) and the migration mapping.
- `shared-corpus`: the corpus-artifacts requirement (as rewritten by `deployment-profiles-and-visibility-lens`) still describes notes as carrying "a `private` flag" with "own-private plus everyone's shared" reads — reworded to the tier model; the lens-bound `not_found` clause is unchanged.
- `data-read-tools`: the "Group signal is readable on shared recipes" requirement (as modified by the lens change) scopes note visibility to lens households and excludes only "private" notes — under tiers, public notes from households OUTSIDE the caller's lens are visible on lens-visible recipes (e.g. a public note on a curated recipe), and friends/private semantics replace the binary rule. Favorite-count aggregation stays lens-household-scoped and unchanged.
- `data-write-tools`: the "Note write tool" requirement names the optional `private` flag and carries a "Private flag honored at write" scenario — updated to the `tier` param with the deprecated `private` alias.
- `member-app-core`: the "Recipe detail with notes…" requirement's notes clause and its "Note privacy is preserved across the group" scenario speak the binary private model — updated to the tier model plus the three-state composer control, tier chips, and tier-aware edit state.

## Impact

- **D1**: one new migration (next free number after the one `households-friends-and-people-page` claims — expected `0061`; verify at implementation time) adding `recipe_notes.tier` + backfill. No new tables, no new bindings, no new routes (`run_worker_first` untouched — `/cookbook*` is already Worker-owned).
- **Worker**: `packages/worker/src/notes-tools.ts` (tool params/returns), the notes queries in `packages/worker/src/corpus-db.ts` (tier predicate replacing `private = 0 OR author = ?2`, members join for handles and author households), the notes routes in `packages/worker/src/api/cookbook.ts`, the anonymous cookbook recipe page renderer (`packages/worker/src/cookbook.ts` — renders no notes today; the public-tier section is net-new). The friends-tier predicate consumes the SAME friend-seam provider `src/visibility.ts` exposes — no second friendship enumeration. Observed divergence folded in: the MCP tools already stamp `author` from the resolved member (`tenant.member`) while the `/api` notes routes still stamp `tenant.id` (byte-identical today under the founding-member invariant) — this change unifies both paths on the member id if the identity-split implementation has not already done so.
- **Member app**: `packages/app` recipe-detail note composer + note list (segmented tier control, chips, edit state), typed api client for the tier field.
- **Docs/persona**: `docs/TOOLS.md`, `docs/SCHEMAS.md`, `AGENT_INSTRUCTIONS.md` + regenerated plugin check.
- **Sequencing**: implementation lands AFTER `households-friends-and-people-page` (friendships table + handles UX) and after the lens change (visibility module, `recipe_imports`). Serial-surface collisions: the `recipe-notes` spec is also deltaed by band 2's `recipe-detail-tweaks` (note-tag UI — not yet landed at planning time; whichever lands second rebases its delta), and `shared-corpus`/`data-read-tools`/`member-app-core` deltas modify requirement text introduced by the lens change (this change archives after it).
- **Production data**: pre-migration `recipe_notes` rows are the implicit migration fixtures; production reads were permission-denied during planning, so an operator-gated pre-merge capture task records the observed private/shared counts (member-identity-split task 8.1 precedent).
