## Why

The Usage page already shows neurons **by model** (account-level, from the GraphQL Analytics API) and each cron job's **durations + counts** (`yamp_usage` AE) — but it cannot answer the operator's actual question when neurons spike: **which activity spent them.** The Worker uses exactly two models — `mistral-small-3.1-24b` (text-gen, the expensive, un-batchable unit) and `bge-base-en-v1.5` (embeddings, cheap/batched/cached) — across **~13 distinct activities** (facet classify, description gen, discovery import classify + taste-confirm, title clean, ingredient confirm, night-vibe naming; recipe/discovery/night-vibe/taste/ingredient/search embeds). Cloudflare's by-model view collapses all thirteen into two bars: "mistral is high" names a model, not a cause.

Two concrete blind spots make the spike diagnosable-by-nobody:

- **No per-activity attribution.** The per-job AE points carry `duration_ms` + job-specific counts, never a token or neuron figure, and one job can span several activities (discovery-sweep does classify + describe + confirm).
- **Import-time AI is invisible.** `create_recipe` seeds a description + facets inline via `env.AI` (best-effort), and that spend bypasses `job_health`/`job_runs`/AE entirely — if members import heavily, it has *no* surface.

Everything is hash-gated, so steady state ≈ 0 calls; spend spikes come from corpus churn, discovery imports, or a **cold backfill** (e.g. migration `0040` nulling every `body_hash` to force a whole-corpus reclassify, draining over many capped ticks). The operator needs to tell "a bounded backlog is draining and will finish" from "something is churning abnormally" — and today nothing connects the backlog depth already in `job_runs` to the neuron burn.

This picks up the per-run neuron-attribution thread that `usage-trends` explicitly deferred ("whether per-job neurons is obtainable is a verification task, not a commitment here").

## What Changes

- **A single AI gateway `src/ai.ts` — `runAi(env, { activity, trigger }, model, input)`** — that every `env.AI.run` call routes through (the ~7 helpers in `embedding.ts`, `description.ts`, `discovery-classify.ts`, `ingredient-classify.ts`, `night-vibe-naming.ts`). This is the wrap-once seam — the AI-call sibling of the `tool-instrumentation` registration seam — so every present and future AI call is captured with no per-call-site wiring. It sits **below the KV embedding cache**, so a cache hit spends no neurons and emits nothing.
- **A third Analytics Engine dataset `yamp_ai` (binding `AI_AE`)** — one **tenant-clean** data point per real inference call, carrying `activity`, `model`, `trigger` (`cron` | `import` | `request`), `outcome`, `duration_ms`, `calls`/items, `input_tokens`, `output_tokens`, and a derived `est_neurons`. Best-effort (a throwing/absent binding is a no-op that never affects the call), non-blocking, zero KV/D1 budget — exactly like `recordUsagePoint`/`recordToolPoint`.
- **A documented per-model token→neuron rate table** (net-new constant) → `est_neurons` per call: real `usage` tokens for text-gen, an input-length estimate for embeddings. It is a **labeled estimate anchored to the account total**, never a billing figure — the panel shows the summed estimate against the account-level by-model actual so its accuracy is self-evident.
- **`GET /admin/api/usage/ai`** — reads `yamp_ai` via the AE SQL API (reusing `CF_ACCOUNT_ID` + the analytics token), returning the per-activity neuron/token/call breakdown over the trend window, split by trigger, with the account total as the reconciliation anchor. Aggregate-only, tenant-data-free, `/admin*` Access-gated, zero KV/D1.
- **A "Neurons by activity" panel on `/admin/usage`** that ranks activities by neuron share, splits cron vs import, and **pairs each activity's series with the job backlog** (the `pending` counts already in `job_runs`) so a bounded backlog reads as "draining, will finish" rather than "broken." Degrades to an explicit not-configured / not-available state, mirroring the page's existing panels.
- **The merge test grows to assert all three AE dataset instances survive** (`yamp_usage`, `yamp_tool`, `yamp_ai`) — the allowlist already carries the `analytics_engine_datasets` type, so the third instance rides along, but the silent-drop guard should cover it.

**Out of scope / verification (not open design questions):** per-call **neurons** are not exposed by the binding (settled: `usage` gives tokens on text-gen only; embeddings expose none; `neuron` is absent from the types), so `est_neurons` is a documented estimate, reconciled against the account actual. Confirming `usage` is populated at runtime for `mistral-small` and recording the current per-model neuron rates are an **early apply-time check on a connected box** — the planning sandbox has neither installed deps nor a `CF_ACCOUNT_ID` for a live call. No new MCP tool (operator observability, not agent-facing). No alerting/forecasting (the user did not want a cap/budget view).

## Capabilities

### Added Capabilities

- `ai-usage-attribution`: every Workers AI call routes through a single gateway that emits one tenant-clean per-call Analytics Engine data point (activity, model, trigger, outcome, duration, tokens, estimated neurons) at zero KV/D1 cost, capturing cron *and* import-time *and* request-path spend below the embedding cache; and the operator Usage page gains a per-activity neuron-attribution view — ranked spend by activity, cron-vs-import split, reconciled against the account-level total and paired with job backlog to distinguish steady churn from a draining backlog — degrading gracefully when unconfigured.

## Impact

- **Code:** `src/ai.ts` (new gateway + emission + the token→neuron rate table), the ~7 `env.AI.run` call sites routed through it (`embedding.ts`, `description.ts`, `discovery-classify.ts`, `ingredient-classify.ts`, `night-vibe-naming.ts`) with `activity`/`trigger` threaded from their callers (notably `create_recipe`'s seeds → `trigger: "import"`), `wrangler.jsonc` (`AI_AE` binding), `src/env.ts` (`AI_AE: AnalyticsEngineDataset`, optional), `src/usage.ts` (the `yamp_ai` SQL mapper + the account-total reconciliation), `src/admin/api.ts` (`GET /admin/api/usage/ai`), `packages/admin-app` (the Usage screen panel + `queries.ts` entry), `packages/ui` if a new primitive is needed.
- **Tests:** the merge-allowlist test (all three AE instances survive), a `runAi` emission test (slot shape, best-effort no-op, cache-hit emits nothing, `usage`→tokens mapping), the SQL-rows → per-activity series mapper + reconciliation, and the Usage-panel decode/not-configured state.
- **Admin UI:** the panel visual is authored through the companion **Claude Design** project (not improvised here); ships with extended Playwright coverage — the `usage` page object + spec under `admin/visual/` and `aubr test:admin` — per the admin-panel-change contract.
- **Docs (lockstep):** `docs/SCHEMAS.md` (the `yamp_ai` **positional** slot contract + the `/admin/api/usage/ai` payload shape), `docs/ARCHITECTURE.md` (the AI gateway as the neuron-attribution seam; the three-dataset AE picture — job / tool / AI), `docs/SELF_HOSTING.md` (the `AI_AE` binding is code-level, no operator config; readback reuses the analytics token; `est_neurons` is an estimate anchored to the account total).
- **No new secret** (emission needs only the binding; readback reuses `CF_ACCOUNT_ID` + the analytics token). One new AE dataset **instance** (type already on the merge allowlist). **Tenant-clean by construction** — activity, model, trigger, outcome, and numbers only, never a per-tenant identifier.
