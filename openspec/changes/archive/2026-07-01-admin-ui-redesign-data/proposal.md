## Why

The Data area (`src/admin/pages/data.tsx`) is still the original generic JSON-table inspector — five sub-tabs (Recipes / Members / Corpus / Discovery / System) that mostly dump raw rows via `<pre>{json(...)}</pre>`. It predates the Basecoat kit and the redesign's purpose-built list/detail patterns already shipped for Status, Members, Logs, Usage, and (in flight) Config and Discovery. Two of its five tabs are now redundant with dedicated areas the redesign has since built: member data belongs in the Members roster/detail, and the Corpus/Discovery/System lookup tables belong in Config and Discovery. What's left — Recipes, the shared store registry, and the R2 guidance tree — deserves purpose-built explorers instead of a bare table/`<pre>` dump, and the Recipes explorer specifically should finally expose the semantic search machinery (`recipe_derived` embeddings) that already exists for the agent-facing tool and the public cookbook, but which the admin's `recipeList` has never used (keyword-only).

## What Changes

- Narrow the Data area's sub-nav to **Recipes / Stores / Guidance** — the three areas with a purpose-built explorer — dropping the generic **Members**, **Corpus**, **Discovery**, and **System** tabs. Nothing is orphaned: Members data is the Members area/detail (already redesigned); Corpus's lookup tables (`aliases`, `flyer_terms`, `feeds`, `senders`, `members`) are the Config area's shared-corpus editors (already redesigned/in flight); Discovery's tables are the Discovery area (in flight); System (`reconcile_errors`, `bug_reports`, `schema_meta`) has no redesigned home yet and is dropped with a reference empty state pending a holistic redesign, matching the mock's explicit call-out.
- Redesign **Data › Recipes**: a searchable, paginated list over the cross-tier corpus with a **Keyword** / **Hybrid** search-mode toggle, per-recipe facet chips, a relevance bar in Hybrid mode, and a projection-status badge (indexed/skipped/pending/orphaned) — composed from the kit's `Item`/`ItemGroup`/`Pager`/`Badge` and the existing search/segmented-toggle pattern. A clicked-through recipe detail assembles every tier for one slug: the AI-derived description, a pipeline-state strip (index/description/embedding), the rendered markdown body, attributed notes, a `PrettyKV` render of the R2 frontmatter and the D1 index row, and a collapsible raw-markdown panel.
- Add **hybrid (semantic + keyword) recipe search** to the admin: a new `searchRecipes(env, query, mode)` reader in `src/admin-data.ts` that, in `hybrid` mode, embeds the query and blends cosine similarity over `recipe_derived.embedding` with the existing keyword match, returning each hit's relevance score and whether it was surfaced semantically (no literal keyword hit) — reusing the Worker's existing embedding path (`src/embedding.ts` / the `recipe_derived` table already reconciled by `src/recipe-embeddings.ts`), not a new vector store or a re-implementation of `semantic-recipe-search`'s ranking.
- Redesign **Data › Stores**: a list over the shared `stores` registry, each clicking through to a store detail — identity (chain/label/address/location_id), its Kroger `sku_cache` (only for Kroger-chain locations), and its `store_notes` grouped by tag convention (layout/location/stock/general) — backed by a new purpose-built `storeDetail(env, slug)` reader rather than three separate generic-table views.
- Redesign **Data › Guidance**: a breadcrumb + folder/file browser over the R2 `guidance/**` tree (via the existing `guidanceListing`/`guidanceObject` readers), opening a file to its rendered markdown — presentation only, no new reader.
- Update `docs/TOOLS.md`/`docs/SCHEMAS.md` only if the admin-only `searchRecipes`/`storeDetail` shapes need documenting per repo convention (they are admin-internal, not MCP tool contracts, so this is expected to be a no-op beyond `docs/SCHEMAS.md` housekeeping if any).

## Capabilities

### New Capabilities

(none — this extends the existing `operator-data-explorer` and `operator-admin` capabilities)

### Modified Capabilities

- `operator-data-explorer`: **ADDS** the Recipes hybrid-search requirement, the Recipes list/detail presentation requirements, the Stores explorer requirement (list + identity/SKU/notes detail), and the Guidance browser requirement. **MODIFIES** the "Shared-corpus view browses the shared lookup tables and guidance markdown" requirement to scope it to the `stores`/`store_notes`/`sku_cache` subset consumed by the new Stores explorer (the rest of that requirement's tables move to `operator-admin`'s Config capability). **REMOVES** the "Member view aggregates a member's full per-tenant state" requirement's presentation-in-Data-area framing (the read itself, `memberDetail`, is unchanged and stays owned by `operator-admin`'s Members capability — this spec no longer routes it through `/admin/data`) and the "Discovery and System views browse the pipeline and operational tables" requirement (superseded: Discovery's tables are the `operator-admin` Discovery area's concern; System has no redesigned home and is a deferred, explicitly out-of-scope empty state, not a spec requirement).
- `operator-admin`: **MODIFIES** the "Admin panel is organized into top-level areas with client-side routing" requirement's Data-area description to name its narrowed Recipes/Stores/Guidance sub-nav (in place of the generic five-tab description), and the "Deep link to a data view" scenario's example route stays valid unchanged (`/admin/data/recipes/<slug>`).

## Impact

- `src/admin/pages/data.tsx` — rewritten: drop the `MembersListPage`/`MemberDetailPage`/`CorpusPage`/`TableViewPage` routes and components (member data path already lives at `/admin/members`; corpus/discovery/system tables have no route here anymore), replace `RecipesListPage`/`RecipeDetailPage` with the redesigned search+detail views, add `StoresListPage`/`StoreDetailPage` and a `GuidancePage` browser, and narrow `registerDataRoutes` accordingly.
- `src/admin-data.ts` — add `searchRecipes(env, query, mode)` (keyword ∪ hybrid over `recipe_derived.embedding`), add `storeDetail(env, slug)` (identity + `sku_cache` rows for the store's `location_id` + grouped `store_notes`); `recipeList`/`recipeDetail`/`guidanceListing`/`guidanceObject` are reused as-is. `readTable`'s `corpus`/`discovery`/`system` groups may shed the tables the Data area no longer routes to, if `operator-admin`'s Config/Discovery areas already read them through their own paths (confirm no orphaned dead code, but do not re-derive their queries).
- `src/embedding.ts` — reused as-is for embedding the admin's hybrid search query (the same helper `src/recipe-embeddings.ts`'s reconcile and `semantic-recipe-search`'s tool path use); no change expected.
- `src/admin/ui/kit.tsx` — no new primitives expected (`Item`/`ItemGroup`/`Badge`/`Pager`/`DataTable`/`PrettyKV` and the search/segmented pattern from the foundation change already cover this area's needs); composition only.
- `openspec/specs/operator-data-explorer/spec.md`, `openspec/specs/operator-admin/spec.md` — delta specs per the Capabilities section above.
