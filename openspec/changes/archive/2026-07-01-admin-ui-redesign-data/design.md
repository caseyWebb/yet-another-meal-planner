## Context

The Data area (`src/admin/pages/data.tsx`, backed by `src/admin-data.ts`) is the last unredesigned area of the operator panel. It is a generic five-tab JSON-table inspector (Recipes / Members / Corpus / Discovery / System) that predates the Basecoat kit and the `Item`/`ItemGroup`/`Pager`/`Badge`/`PrettyKV`/`DataTable` primitives the foundation change added (`src/admin/ui/kit.tsx`). The mock (`DataScreen.jsx`, `RecipesScreen.jsx`, `RecipeDetail.jsx`, `StoresScreen.jsx`, `GuidanceScreen.jsx`, and their `*-data.jsx` fixtures) narrows the Data sub-nav to three purpose-built explorers — **Recipes**, **Stores**, **Guidance** — and its own `DataScreen.jsx` comment states the reasoning: "Discovery/System keep reference empty states pending a holistic redesign," implying Members/Corpus/Discovery content has already moved elsewhere in this redesign sequence.

Checking that premise against the shipped and in-flight changes: the **Members** redesign (archived `2026-06-30-admin-ui-redesign-members`) already gives Members its own roster + detail area at `/admin/members`, reusing the same `memberDetail()` reader `data.tsx` currently also exposes at `/admin/data/members/<id>` — so that route is a pure duplicate today. The **Config** redesign (`openspec/changes/admin-ui-redesign-config`, in flight) already specs shared-corpus editors for `aliases`/`flyer_terms`/`feeds`/`senders`/`members` under `/admin/config/*` (see the archived `operator-admin` "Operator edits shared-corpus tables under Config" requirement) — so `CORPUS_TABLES` in today's `data.tsx` (`aliases, flyer_terms, feeds, stores, store_notes, sku_cache`) is redundant for everything except `stores`/`store_notes`/`sku_cache`, which the new Stores explorer takes over instead. The **Discovery** redesign (`openspec/changes/admin-ui-redesign-discovery`, in flight) already gives Discovery its own top-level area — so `DISCOVERY_TABLES` in `data.tsx` is fully redundant. **System** (`reconcile_errors`, `bug_reports`, `schema_meta`) has no redesigned home in any shipped or in-flight change; the mock's own comment defers it, so this change drops its Data-area route with no replacement, matching the mock.

`src/admin-data.ts`'s `recipeList`/`recipeDetail` are keyword-blind — there is no ranking, just an alphabetical slug listing and a flat per-slug assembly. The mock's `RecipesScreen.jsx` offers a **Keyword** / **Hybrid** search toggle, where Hybrid returns a relevance score and flags semantically-surfaced (non-keyword) hits. The repo already has three independent semantic-search implementations layered over the same `recipe_derived.embedding` column: the agent-facing `search_recipes` tool's ranked mode (`semantic-recipe-search` spec: facet-prefilter then cosine, plus taste/freshness/pantry re-ranks — all per-tenant signals the admin explorer has no reason to apply), the anonymous `/cookbook` keyword ranker (`cookbook-search` spec: field-weighted keyword only, no embeddings), and `/cookbook`'s "Similar Recipes" cosine-neighbor section (`cookbook-similar-recipes` spec: stored-vector cosine only, no live embed). None of the three is a drop-in fit: the tool's ranked mode carries per-tenant re-ranks the cross-tenant admin explorer must not apply; the cookbook's keyword ranker has no semantic leg at all; the similar-recipes feature ranks recipe-to-recipe, not query-to-recipe. What all three genuinely share, and what this change reuses, is the primitive pair in `src/embedding.ts`: `embedText(env, text)` (one Workers AI call, returns a 768-dim vector) and `cosineSimilarity(a, b)` (pure arithmetic, no I/O) — plus the already-reconciled `recipe_derived` table (`src/recipe-embeddings.ts`) as the vector source, with no new cron and no new AI-compute path.

## Goals / Non-Goals

**Goals:**
- Narrow the Data sub-nav to Recipes / Stores / Guidance, confirm and document that the dropped tabs' data has a real home (Members, Config, Discovery) or is a deliberately deferred empty state (System), and remove the now-dead routes/components from `data.tsx`.
- Redesign Recipes as a searchable, paginated cross-tier explorer with a genuine Keyword/Hybrid toggle, backed by a new `searchRecipes` reader that reuses `src/embedding.ts`'s primitives — not a re-implementation of `semantic-recipe-search`'s tenant-aware ranking.
- Redesign Stores as a purpose-built list + detail (identity, SKU cache, grouped notes), replacing the generic `stores`/`store_notes`/`sku_cache` table tabs with one assembled reader.
- Redesign Guidance as a breadcrumb + tree browser over the existing `guidanceListing`/`guidanceObject` readers — presentation only.
- Land this as the last area of the `admin-ui-redesign-*` sequence, closing out the generic table-inspector UI entirely (System's deferred empty state is the one remaining artifact of the old pattern, by design).

**Non-Goals:**
- No change to `semantic-recipe-search`'s tool contract, its per-tenant re-ranks (favorites/freshness/pantry-overlap), or `cookbook-search`'s keyword ranker — the admin's hybrid search is a new, narrower consumer of the shared embedding primitives, not a refactor of either.
- No new cron, no new AI-compute path, no new vector store — the admin hybrid search reads the `recipe_derived` table the existing reconcile already maintains and makes exactly one `embedText` call per search (the query only; recipe vectors are never re-embedded here).
- No redesign of System — it keeps a reference empty state (per the mock), deferred to a future holistic redesign, not specified as a requirement in this change.
- No change to `readTable`'s D1 access discipline (`src/db.ts`, allowlisted tables) — the new `storeDetail`/`searchRecipes` readers follow the same pattern (fixed queries, no operator-supplied SQL), they just assemble richer shapes than a flat table page.

## Decisions

### 1. Hybrid search: a new `searchRecipes(env, query, mode)` reader that reuses `src/embedding.ts`, not `semantic-recipe-search`'s ranked mode

**Decision**: add `searchRecipes(env, query, mode: "keyword" | "hybrid")` to `src/admin-data.ts`. Keyword mode is the existing kind of match extended with per-token substring scoring over the indexed metadata (title, slug, protein, cuisine, course, tags, `ingredients_key`) that `recipeList` already has read access to — an AND-of-tokens filter, no ranking beyond title/slug tie-break (mirrors `cookbook-search`'s coverage principle without adopting its field-weight table verbatim, since the admin index has a different field set than the cookbook's). Hybrid mode additionally: (a) embeds the query with `embedText(env, query)` — one Workers AI call per search, the same helper the `search_recipes` tool and the cookbook-similar feature both call; (b) loads every `recipe_derived` row with a non-null `embedding` (a plain `SELECT slug, embedding FROM recipe_derived WHERE embedding IS NOT NULL`, joined against `recipeList`'s already-fetched slugs — no new table); (c) scores each candidate by `cosineSimilarity` against the query vector, blended with the keyword coverage fraction (mirroring the mock fixture's `kwFrac * 0.7 + vibeFrac * 0.45`-shaped blend, exact weights left to apply); (d) flags a hit `semantic: true` when it clears the relevance floor via the cosine term without a full keyword match, so the UI's "surfaced semantically" badge is honest. Results are returned as `{ slug, score, semantic }[]` for the list page to join against `recipeList`'s rows.

This is a **new function**, not a call into `semantic-recipe-search`'s `search_recipes` ranked-mode path, because that path bakes in three per-tenant signals — nearest-favorited-recipe re-rank, never/not-recently-cooked freshness boost, and `boost_ingredients` pantry overlap — that require a `tenantId` and are meaningless for a cross-tenant operator search ("relevant to which member's taste?" has no answer here). Stripping those out of the tool's implementation would leave a facet-prefilter-then-cosine shape that is *coincidentally* close to what the admin needs, but forking it either means the admin passes a fake/absent tenant through tenant-shaped code (a footgun: a future edit to the tool's re-rank could silently start requiring a tenant the admin doesn't have) or the tool's ranking function gets an awkward "skip re-ranks" parameter serving one caller that isn't the tool contract. A small, honest, admin-owned reader over the same two primitives (`embedText`, `cosineSimilarity`) and the same table (`recipe_derived`) is cheaper to reason about and matches the "new queries only for the cross-tenant aggregates" principle `operator-data-explorer`'s existing "Data reads route through the data-access layer" requirement already establishes.

**Cost**: one `embedText` call per Hybrid search (same Workers AI budget class the tool/reconcile already use — see `src/recipe-embeddings.ts`'s comment on `env.AI` sharing the 1,000/invocation internal-services bucket) and one bulk `recipe_derived` read (small: one row per recipe, friend-group-corpus scale, same scale `recipeList` already reads in full). No new D1 table, no new migration, no new cron tick. Keyword-mode searches make no AI call at all (mirrors `semantic-recipe-search`'s "vibe-less batch makes no AI request" discipline, applied to the admin's own toggle).

**Alternative considered**: expose the raw `search_recipes` MCP tool's ranked mode to the admin explorer (call the tool internally with a synthetic/absent tenant). Rejected — the tool's ranking is specified in terms of a caller's favorites/cooking-history/pantry, none of which exist for "the operator browsing everything"; forcing a fit would mean either fabricating a null-signal tenant path through tool code that was never designed for one, or amending the tool contract for an admin-only edge case. The shared *primitives* (embedding + cosine) are reusable; the tool's *ranking policy* is not.

**Alternative considered**: keyword-only (skip Hybrid entirely, drop that part of the mock). Rejected — the task explicitly calls "build the backing now" as the intended decision, and the embedding infrastructure genuinely is a two-function reuse, not a new subsystem; shipping Hybrid is the cheap, obviously-correct move once the reuse path is clear.

### 2. Stores detail: one purpose-built `storeDetail(env, slug)` reader, not three `readTable` calls composed client-side

**Decision**: add `storeDetail(env, slug)` returning `{ slug, name, domain, chain, label, address, location_id, skus: SkuRow[], notes: StoreNoteRow[] }` — one assembled read: the `stores` row by slug (identity, with `chain`/`label`/`address` unpacked from its `extra` JSON column per `src/admin-data.ts`'s existing `stores` `TableSpec`), `sku_cache` rows filtered to that store's `location_id` (skipped/empty when `location_id` is null — a non-Kroger location), and `store_notes` rows for that store, grouped server-side by the `layout`/`location`/`stock`/`general` tag convention the mock's `StoreDetail.jsx` already encodes (first tag, falling back to `general`). This mirrors `recipeDetail`'s existing shape (one cross-tier assembly per entity) rather than making the Stores detail page compose three generic `readTable("corpus", …)` calls and re-derive the grouping client-side — the grouping-by-tag logic belongs server-side next to the query, same as `recipeDetail`'s `deriveStatus`.

**Alternative considered**: keep `readTable` as the only store-related reader and have `StoreDetailPage` filter/group in the SSR component. Rejected — `readTable`'s `stores`/`store_notes`/`sku_cache` specs are flat, un-joined, un-filtered-by-slug table dumps (see `TABLE_SPECS` in `src/admin-data.ts`); filtering `sku_cache` by a store's `location_id` and grouping notes by tag are per-entity assembly logic that belongs in `src/admin-data.ts` next to `recipeDetail`, not duplicated in the page component the way the old generic table view got away with because it never needed to join anything.

### 3. Guidance: presentation only, zero new readers

**Decision**: `guidanceListing`/`guidanceObject` already return exactly what the mock's `GuidanceScreen.jsx` needs (a prefix's immediate dir/file children; one object's markdown). The redesign is SSR route/component work only — a breadcrumb built from the `?gprefix=`/`?gpath=` route params (as today) restyled with kit primitives and folder/file icons, no new `src/admin-data.ts` surface.

### 4. Dropped tabs: routes removed, not just hidden

**Decision**: `registerDataRoutes` drops the `/data/members`, `/data/members/:id`, `/data/discovery`, `/data/system` routes entirely (not merely unlinked from the sub-nav) since their content is fully superseded (Members, Discovery) or deliberately deferred with no UI at all (System, per the mock). `/data/corpus` is dropped as a route name; the `stores`/`store_notes`/`sku_cache` subset it served moves into the new `/data/stores` explorer, and the `aliases`/`flyer_terms`/`feeds` subset is left to the Config redesign's `/admin/config/*` routes (already specified there — this change does not touch Config). `readTable`'s `TABLE_GROUPS`/`TABLE_SPECS` in `src/admin-data.ts` keep only the entries still reachable through a live route after this and the Config/Discovery redesigns land; apply should confirm with those two in-flight changes before deleting `discovery`/`system`-group entries outright, so as not to break a route that change still depends on mid-flight.

## Risks / Trade-offs

- **[Risk]** The admin's hybrid-search blend weights (keyword-fraction vs. cosine) are a new, admin-only tuning surface distinct from the tool's and the cookbook's — a future change to one ranker's feel won't automatically apply to the others, and reviewers must remember this is intentional (three independently-tunable rankers over the same vectors, not a shared policy). → **Mitigation**: `searchRecipes`'s blend constants get the same "tunable constant, not part of the contract" treatment `semantic-recipe-search`/`cookbook-search` already use for their own weights, called out explicitly in the spec so it isn't mistaken for a cross-ranker inconsistency.
- **[Risk]** `storeDetail`'s `location_id`-based SKU filter assumes `sku_cache.location_id` matches `stores.extra.location_id` exactly (both plain strings) — a mismatch (e.g. a store record with a stale or malformed `location_id`) silently yields an empty SKU list rather than an error. → **Mitigation**: match `recipeDetail`'s posture (absence is a normal, renderable state — "no cached SKUs… not a Kroger location" in the mock — not a failure); the detail page already needs to render the empty case for non-Kroger stores, so a malformed-id case degrades to the same harmless empty state instead of a new error path.
- **[Trade-off]** Dropping System's Data-area route with no replacement leaves `reconcile_errors`/`bug_reports`/`schema_meta` unreachable from the panel UI until a future change gives System a home — accepted per the mock's explicit deferral; these remain inspectable via `wrangler d1 execute` in the interim (see `CONTRIBUTING.md`'s local-D1 tooling), so nothing is silently lost, only de-prioritized.
- **[Risk]** Removing `/data/corpus`'s `aliases`/`flyer_terms`/`feeds` sub-tabs depends on the Config redesign (`admin-ui-redesign-config`) actually landing those editors — if this change's apply step runs first, there would be a gap where those three tables are reachable nowhere in the admin UI. → **Mitigation**: sequence apply after `admin-ui-redesign-config` archives (or verify its editors are already merged) before deleting the corresponding `data.tsx` routes; flagged as a task-ordering dependency, not a spec conflict (the two changes' spec deltas do not overlap on the same requirement).

## Migration Plan

No data migration — no D1 schema change, no new table, no new cron. This is UI-surface and admin-reader-only work: `src/admin-data.ts` gains two new read functions (`searchRecipes`, `storeDetail`) and `src/admin/pages/data.tsx` is rewritten to route to the three redesigned explorers and drop the four generic-table routes. Deploys as a normal panel change; no feature flag needed (the redesigned routes replace the old ones directly, matching every prior area in this redesign sequence). Apply should land after (or alongside, with route-drop sequencing) the Config and Discovery redesigns so their tables' new homes are live before this change removes the old Data-area paths to them — see the risk above.

## Open Questions

- Exact hybrid blend weights and the semantic-surfaced relevance floor (mirroring the mock fixture's `0.7`/`0.45`/`0.18` constants) — recommend apply picks initial values matching the mock's feel and treats them as the same kind of tunable constant `semantic-recipe-search`/`cookbook-search` already exempt from their specs, revisited after real corpus use.
- Whether `readTable`'s `TABLE_GROUPS.discovery`/`.system` entries in `src/admin-data.ts` should be deleted in this change or left as dead-but-harmless allowlist entries until System gets a real home — recommend leaving them (no route reaches them, so they're inert) rather than coupling this change's scope to a System redesign that doesn't exist yet; `TABLE_GROUPS.corpus`'s `aliases`/`flyer_terms`/`feeds` entries should be deleted only once confirmed unused by the Config redesign's own reader path (it may define its own, per that change's design).
- Whether `storeDetail`'s note-grouping ties break on a note with multiple tags (mock takes `tags[0]`, defaulting to `general`) — confirm this matches `store_notes`' authored tag convention exactly, or whether a note could reasonably want to appear in more than one group.
