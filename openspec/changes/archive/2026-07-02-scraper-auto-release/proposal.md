## Why

Releasing a new scraper today is a manual, drift-prone gesture: a maintainer hand-pushes a `scraper-v<version>` tag and the release workflow builds from `${GITHUB_REF_NAME#scraper-v}`. Two problems fall out of that. First, the tag is a *second* place the version lives â€” nothing forces it to equal `packages/scraper/package.json` `version`, which is the value the **running** scraper actually reports (`src/push.ts` reads it at runtime and stamps it on every batch); a mistyped tag ships an image whose GHCR label and admin skew chip disagree with what it reports. Second, the human step is exactly the kind of "did anyone remember to cut the release?" gap that lets a merged fix sit unpublished.

The companion `scraper-version-gate` change closes the *input* side â€” it makes every scraper-touching PR bump `packages/scraper/package.json` `version` or fail. This change closes the *output* side: once the human has bumped the version and the PR merges, the release should publish itself. Together they form a complete loop â€” **human bumps the version in the PR â†’ the gate enforces it â†’ the merge auto-publishes** â€” with the package.json version as the one source of truth, no hand-typed tag, no stored secret, and no commit-back to the repository.

Two latent bugs in the current workflow are fixed in the same pass, because auto-release exposes both: on any non-tag trigger the Release is tagged `${{ github.ref_name }}` (which is the *branch*, e.g. `main`, not `scraper-v<version>`); and the Docker `build-args` (`SCRAPER_VERSION`/`CONTRACT_VERSION`) are dead â€” the Dockerfile declares no matching `ARG`, and the running scraper derives both at runtime (version from `package.json`, contract from the `@grocery-agent/contract` import), so the build-args feed nothing.

## What Changes

- **Version source of truth = `packages/scraper/package.json` `version`.** The release derives the version from `package.json` (not a hand-typed tag or dispatch input) and derives the `scraper-v<version>` tag from it, so the published tag, the GHCR image label, and the version the running scraper reports can never drift.
- **Auto-publish on merge to `main`.** A version-change **detector** runs on every push to `main`, compares `packages/scraper/package.json` `version` against the previous commit (`git show HEAD^:packages/scraper/package.json`), and â€” only when it changed and CI is green â€” invokes the release. No human tag push; no commit-back to the repository.
- **`scraper-release.yml` becomes a reusable `workflow_call` workflow** (input: `version`), retaining a `workflow_dispatch` **manual fallback** that publishes whatever version `package.json` currently declares. The detector calls the reusable release **inline in the same run** â€” it does NOT push a tag and rely on a tag-triggered workflow, because a tag pushed by the built-in `GITHUB_TOKEN` does not trigger `on: push` workflows.
- **Idempotence guard.** Before building, the release checks whether the `scraper-v<version>` release/tag already exists and no-ops if so, so a workflow re-run, a manual dispatch, or an unrelated `main` push cannot double-publish a version.
- **Drop the `on: push: tags: scraper-v*` trigger** (recommended â€” see design): with package.json as the source of truth, a human-pushed tag is a redundant path that reintroduces the tag/version drift this change removes. The manual fallback is `workflow_dispatch`.
- **Fix the two latent bugs:** set the Release `tag_name` explicitly to `scraper-v<version>` (not `github.ref_name`), and remove the dead `build-args` block from the build-push step.
- **Kept intact:** GHCR + a GitHub Release, the built-in `GITHUB_TOKEN` only (no new stored secret), the multi-arch (`linux/amd64` + `linux/arm64`) manifest, the embedded build + contract version, and independence from the Worker deploy control plane.

## Capabilities

### Modified Capabilities
- `build-automation`: the scraper release is triggered automatically by a `packages/scraper/package.json` `version` change on merge to `main` (with the version as the single source of truth and the `scraper-v<version>` tag derived from it), published inline via a reusable workflow with a `workflow_dispatch` fallback, idempotent against re-publishing an already-released version â€” still GHCR + GitHub Release, `GITHUB_TOKEN`-only, multi-arch, embedded-version, and independent of the Worker deploy.

## Impact

- **Dependency / sequencing.** This change sequences **after** `scraper-multiarch-image` (which lands the multi-arch mandate in the workflow and in the `build-automation` spec â€” the modified requirement here is layered on that multi-arch text) and **after** `scraper-version-gate` (which enforces the human version bump this change assumes). The implementer should rebase onto a `main` that carries both. The `build-automation` requirement text this change modifies is the **post-multi-arch** version.
- **CI/CD:** `.github/workflows/scraper-release.yml` is refactored to `on: workflow_call` (input `version`) + `workflow_dispatch` (fallback), with the two bug fixes and the idempotence guard; `.github/workflows/ci.yml` gains a push-only version-change **detector** job and a gated `uses:` job that calls the reusable release (gated on the existing `test` + `no-open-changes` jobs, mirroring `trigger-deploy`). The `trigger-deploy` path filter is **unchanged** â€” it still excludes `packages/scraper/**`, keeping the scraper release a separate control plane from the Worker deploy.
- **No repo mutation on publish:** no version-bump commit, no committed tag on a branch â€” the `scraper-v<version>` tag is created only as part of cutting the Release, in the same run.
- **Docs:** review any maintainer-facing "push a `scraper-v*` tag to cut a release" instruction (`CONTRIBUTING.md`, `docs/SELF_HOSTING.md`, `packages/scraper/README.md`) and reword it to "bump `packages/scraper/package.json` `version`; the merge auto-publishes." Operator-facing "pull the image from GHCR" text is unaffected.
- **Spec:** `openspec/specs/build-automation/spec.md` â€” the "The scraper is published as a container image on a tagged release" requirement is modified (full restatement below).
- **No change** to the Worker, the contract package, the scraper's runtime code, D1, or the MCP tool surface. The Dockerfile is unchanged (the dead build-args live only in the workflow, not the Dockerfile).
