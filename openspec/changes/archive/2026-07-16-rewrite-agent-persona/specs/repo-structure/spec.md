## MODIFIED Requirements

### Requirement: Code repository layout

The code repository (`caseyWebb/yet-another-meal-planner`) SHALL be an **`aube` workspace monorepo** (workspaces declared in `pnpm-workspace.yaml`, packages cross-depending via the `workspace:*` protocol) whose packages are the **Worker** (the Cloudflare Worker — MCP server, member `/api`, and the Access-gated `/admin` routes — with its build tooling `scripts/`, D1 migrations `migrations/d1/`, and Worker tests), a shared **contract** package, the home-network **satellite** package, the **member app** (`packages/app`, the React SPA served at `/`), the **admin app** (`packages/admin-app`, the operator panel SPA served at `/admin`), the shared **UI** package (`packages/ui`, the shadcn/ui primitives and theme tokens both apps consume), and the **plugin** package (`packages/plugin`, owning the agent persona source `AGENT_INSTRUCTIONS.md`, the plugin generator `scripts/build-plugin.mjs`, and the generator's tests). The shared **contract** package SHALL be **workerd-pure** (no Node-only dependencies) because it is imported by BOTH the Worker (workerd runtime) and the satellite (Node runtime); it SHALL hold the recipe-parse spine (JSON-LD extraction + normalize) and the ingest wire types/validators, so the parse and the payload shape can never drift between the two runtimes. The **satellite** package (Node + a headless-browser dependency) SHALL be isolated in its own workspace so its heavy dependencies never enter the Worker build or deploy. The reference docs SHALL remain under `docs/`, the OpenSpec workspace under `openspec/`, and the agent persona source `AGENT_INSTRUCTIONS.md` in `packages/plugin/`. Reusable CI workflows that operator data repos call SHALL live under `.github/workflows/`. The code repository SHALL NOT commit a generated plugin bundle, a `.claude-plugin/marketplace.json`, or any built static-assets bundle (the Worker's merged assets root — both SPAs' output — is a gitignored build artifact); those are generated by CI/the operator's deploy, not committed.

#### Scenario: The repository is a workspaces monorepo

- **WHEN** the code repository is cloned and listed
- **THEN** it resolves as a workspaces monorepo with a Worker package, a shared workerd-pure contract package, a satellite package, the member-app, admin-app, shared-UI, and plugin packages, alongside `docs/` and `openspec/`, and no committed plugin bundle, `.claude-plugin/marketplace.json`, or built assets bundle

#### Scenario: The shared contract package is workerd-pure

- **WHEN** the contract package is built
- **THEN** it carries no Node-only dependency, so both the Worker (workerd) and the satellite (Node) import the same recipe-parse and ingest wire types

#### Scenario: The satellite's heavy deps stay out of the Worker

- **WHEN** the Worker is built or deployed
- **THEN** the satellite package's browser/Node dependencies are not included — they are isolated to the satellite workspace

#### Scenario: The plugin package owns the persona and its generator

- **WHEN** a developer looks for the agent persona source or the plugin build tooling
- **THEN** both resolve inside `packages/plugin/` (`AGENT_INSTRUCTIONS.md`, `scripts/build-plugin.mjs`, and the generator tests), and the Worker package carries no persona source or plugin generator

### Requirement: Canonical docs are committed at defined locations

The repository SHALL commit the canonical project docs. `AGENT_INSTRUCTIONS.md` SHALL reside at `packages/plugin/AGENT_INSTRUCTIONS.md` as the canonical yamp persona source from which the plugin bundle is generated — it is a build source owned by the plugin package, not repo-development guidance, so it SHALL NOT reside at the repository root. `CLAUDE.md` SHALL reside at the repository root as Claude Code development guidance for working in this repo, and the development docs SHALL point to `packages/plugin/AGENT_INSTRUCTIONS.md` for the agent persona and conversational flows. The contributor guide `CONTRIBUTING.md` SHALL reside at the repository root; the reference docs `ARCHITECTURE.md`, `SCHEMAS.md`, and `TOOLS.md` SHALL reside under a `docs/` directory. Any references to the `docs/` reference docs SHALL resolve to their `docs/` paths.

#### Scenario: Root docs are present with their distinct roles

- **WHEN** Claude Code opens the repository directory
- **THEN** it finds `CLAUDE.md` at the repository root and reads it as repo-development context, and finds the yamp persona source at `packages/plugin/AGENT_INSTRUCTIONS.md`, not at the root

#### Scenario: Development docs point to the agent instructions

- **WHEN** a reader follows the development docs looking for the agent persona or conversational flows
- **THEN** they are directed to `packages/plugin/AGENT_INSTRUCTIONS.md` rather than that prose being duplicated elsewhere

#### Scenario: Canonical docs live at their defined locations

- **WHEN** a developer looks for the architecture, contributor, schema, and tool references
- **THEN** `CONTRIBUTING.md` is present at the repository root, and `docs/ARCHITECTURE.md`, `docs/SCHEMAS.md`, and `docs/TOOLS.md` are present under `docs/`

#### Scenario: Doc references resolve

- **WHEN** a reader follows a root doc's pointer to the tool inventory (`docs/TOOLS.md`)
- **THEN** the referenced file exists at that path
