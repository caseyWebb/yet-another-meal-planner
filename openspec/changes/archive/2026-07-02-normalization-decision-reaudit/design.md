# Design — normalization-decision-reaudit

## Context

The organic-normalization pipeline captured its first ~300 identity nodes under pre-hardening rules; `ingredient-classifier-hardening` (archived) stops NEW bad decisions (the `NORMALIZE_CONFIRM_MIN = 0.72` distance guard, distinct-product prompt rules, canonical novel ids, the commit-time edge contradiction gate, embedding backfill) but explicitly excluded repairing standing rows. Production currently holds (the acceptance fixture for this change):

- **Wrong auto aliases** — two classes. *Below-guard collapses* the old rules allowed: 'flaky sea salt'→`fish sauce::type-sea-salt` (chosen-candidate cosine 0.598), 'half loaf sourdough bread'→`bread flour::type-sourdough` (0.705). *High-cosine distinct-product / prep-vs-product errors* embedding distance cannot see: 'sesame seeds'→`toasted sesame seeds::toast` (0.879), 'salmon fillets skin-on'→`salmon fillets, skin-on::species-atlantic` (0.982), 'canned salmon'→`salmon fillets, skin-on::form-canned` (0.764), 'dried medjool dates'→`dried fruit blend::type-medjool-dates` (0.745).
- **Wrong/contradictory auto edges** — a rep-resolved 2-cycle (`whole cardamom pods` ↔ `ground cardamom`), wrong-direction edges (`ground coriander` -[containment]→ `coriander seed`, `ground nutmeg` -[general]→ `whole nutmeg`), wrong-satisfies (`tuna in oil`→`tuna in water`, `semolina flour`/`dark rye flour`/`whole wheat flour`→`all-purpose flour`, `chicken tenderloin`→`whole frozen chicken`, `spaghetti`→`rigatoni`), wrong-membership (`cayenne pepper` and `serrano or jalapeño peppers`→`hot sauces (various)`, `garlic powder`→`italian seasoning`, cheese slices→`shredded cheese`, frozen fruit mixes→`dried fruit blend`).
- **Legacy `sku_cache` keys** — 13 of 20 rows keyed by raw terms ('whole milk', 'parmesan', …) that become graph nodes/aliases as capture proceeds; grocery/pantry have a key reconcile (`grocery-pantry-reconcile.ts`), `sku_cache` has none.
- **An unguarded re-confirm merge** — `reconfirmOne`'s `same` outcome merges without the confirm-distance guard (flagged in the hardening design's Risks as the first follow-up).

The operator rejected one-time manual data surgery: the pipeline must converge its own bad data. Scale: ~322 aliases (~288 self-aliases), ~85 auto edges, 20 `sku_cache` rows; the cron ticks every 5 minutes.

Constraints carried over from the siblings: all D1 through `src/db.ts`/`corpus-db.ts` (structured `storage_error`, throw-free tools); fake-D1 can't GROUP BY or subquery, so reads are full-table + JS filtering; ids are append-only (merges via `representative` only); human rows are never overwritten by auto passes; internal `env.AI`/D1 budget only; no new dependencies. Note: `corpus-db.ts` intentionally contains a raw NUL byte (`filterCommittableEdges`' pair-key separator, so ids containing spaces can't collide) — grep/Edit treat the file as binary; patch it via a script.

## Goals / Non-Goals

**Goals:**
- Re-audit every pre-hardening AUTO alias decision under the hardened rules, correcting via existing primitives only, bounded per tick, self-quiescing, with the full backlog swept within hours.
- Re-audit every AUTO edge: delete rep-resolved self-loops deterministically, resolve reverse-pair 2-cycles with one classifier direction check, validate standing edges against the satisfies semantics, drop invalid ones.
- Converge `sku_cache` keys onto the current alias/representative resolution, idempotently, every tick, no LLM.
- Apply the confirm-distance guard to the re-confirm pass's merge and specialization picks (parity with capture).
- New decisions are born audited, so the passes quiesce to standing no-ops.

**Non-Goals:**
- No node deletion, ever (wrong-mint nodes are orphan-merged or left orphaned; the admin Normalization view already surfaces orphans).
- No re-audit of `source='human'` aliases, nodes, or edges.
- No canonical-id re-home of existing nodes (ids stay append-only).
- No new MCP tools, admin routes, or UI surfaces; no HEALTH_JOBS additions.
- No re-key reconcile for `brand_prefs` (same shape, but no observed staleness; a follow-up if its keys drift).

## Decisions

- **D1 — Alias pre-filter stamps ONLY self-aliases; everything else gets one classifier confirm.** The deterministic pre-filter is `variant === alias.id` (the row every mint writes for its own node — ~288 of production's 322 rows, stamped free, no embedding and no LLM). Every other auto alias goes to the hardened classifier confirm. *Alternative rejected:* a cosine pre-filter (stamp when variant↔node cosine ≥ `NORMALIZE_CONFIRM_MIN`) — embedding distance is structurally blind to the main defect class (distinct-product / prep-vs-product errors): 4 of the 6 known-bad production aliases cosine ABOVE 0.72 (sesame seeds 0.879, salmon fillets skin-on 0.982, canned salmon 0.764, medjool dates 0.745), which is exactly why the hardening change added prompt rules instead of a threshold. The trade is ~34 one-time LLM calls at production scale (2–3 ticks at the per-tick bound) for correctness — deliberately cheap. A self-alias is definitionally consistent as a *mapping* (variant → the node of that exact name); whether the *node* should merge into a synonym is the re-confirm/co-resolution passes' job, not the alias audit's.
- **D2 — Re-decisions ride capture's own primitives: `buildResolution` + `commitResolution`.** The confirm runs with candidates from the current registry (cosine top-K over the variant's embedding, with the currently-mapped survivor appended when retrieval misses it — scored from the registry vector when it has one, unscored otherwise, which `ScoredCandidate` already models). The pick is subject to `NORMALIZE_CONFIRM_MIN` exactly as in capture (`resolveOne`'s guard): a distant pick rejects to a verbatim NOVEL mint — which corrects the flagship 'flaky sea salt' row even if the classifier repeats its old pick. `buildResolution` then constructs the outcome (same → re-point; specialization → mint `base::detail` + re-point; novel → canonical-synthesis mint + re-point) and `commitResolution` applies it: the alias upsert IS the re-point (auto source, fresh `decided_at`), the node upsert is non-destructive, edges pass the contradiction gate, the queue delete is a no-op, and the log rides the same batch. The log row carries `detail.audit = "alias"` plus `previous_id` (the pre-audit survivor) — the "distinguishable flag" mirroring `is_reconfirm`, but in `detail` so no log-schema migration is needed. A confirm that re-affirms the current mapping (`same` on the current survivor) commits identically (refreshing `decided_at`, born-stamping) and counts as `kept`. Failure handling: transient `env.AI`/D1 error → the row is skipped un-stamped (retried next tick — un-stamped IS the retry state, like re-confirm; no backoff column because there is no queue row to defer); contract-invalid confirm → **keep the existing mapping and stamp it** (log `outcome: "novel"`, `note: "confirm_failed_safe"`, `resolved_id` = the kept survivor). Conservative because a re-audit differs from capture: capture must place a NEW term somewhere (fail-safe mint), but a re-audit already has a standing resolution — destroying or replacing it on an undecidable confirm could only make things worse, while keeping it is at worst the status quo, stamped so one flaky row can't pin the pass, and still human-overridable.
- **D3 — Orphan-node cleanup on re-point.** When the applied re-decision moves variant V off node N, and N is `source='auto'`, and no alias rows still map to N (checked over the post-re-point alias set), the pass calls `mergeIdentities(N → resolve(r.id))` so the junk node (e.g. `fish sauce::type-sea-salt` after 'flaky sea salt' leaves) exits the retrieval set instead of lingering as a nonsense cosine candidate, and any stray references (old `sku_cache` keys, edges) keep resolving somewhere sane through the representative chain. Never for human nodes, never for a node that retains other aliases, never when N already resolves to the target. The merge is the existing non-destructive union-find primitive and writes its own `merge` log row.
- **D4 — `audited_at` columns via `0035_reaudit.sql`; writes are born-audited.** Two nullable INTEGER columns: `ingredient_alias.audited_at`, `ingredient_edge.audited_at` (NULL = pre-hardening backlog; epoch ms once audited). A dedicated stamp rather than overloading `reconfirmed_at` because that column lives on `ingredient_identity` and means "node re-examined for enrichment" — a different subject (node vs mapping/edge) and a different question. `commitResolution` and `commitReconfirmEdges` stamp `audited_at = now` on the alias + edge rows they write (the alias upsert also sets it on conflict), so every post-hardening decision is born outside the audit backlog and the passes converge to standing no-ops. Human writes (`addAliases`) don't stamp — `source='human'` rows are excluded by the eligibility predicate, so the stamp is irrelevant there.
- **D5 — Edge audit: deterministic deletes first, then one direction-check classifier.** Per audited edge (auto + un-stamped, oldest first, bounded): (a) endpoints resolved through `representative` equal → DELETE, no LLM (the self-loop is meaningless by construction). (b) A resolved reverse pair exists (any kind): if the reverse edge is `human`, the auto edge loses deterministically (human authority — no LLM); else ONE `confirmSatisfiesDirection(from, to)` call returns `forward | reverse | both | neither` and the pass keeps/stamps the edge(s) matching the answer and deletes the rest (`both` keeps both — mutual satisfaction is legitimate for near-synonyms; `neither` deletes both). (c) A standing edge gets the same call; `forward`/`both` → stamp, `reverse`/`neither` → DELETE. The new classifier lives in `src/ingredient-classify.ts` beside `confirmIdentity` (same model, temperature 0, strict-JSON contract, corrective retry, `storage_error`/`validation_failed` split), asks "does having FROM satisfy a request for TO?" over the readable forms, and its prompt carries the hardened distinct-product rules with fixture-derived few-shots (whole cardamom pods→ground cardamom = forward; ground nutmeg→whole nutmeg = reverse; semolina flour→all-purpose flour = neither; kielbasa→sausage = forward). Contract-invalid → keep + stamp (conservative, as D2); transient → skip un-stamped. Deleting edges is deliberately allowed here — unlike the hardening change's never-delete commit gate — because this pass's whole purpose is retroactive correction of a bounded, one-time auto backlog; edges are open-world hints, so a wrongly-deleted edge degrades to read-time world knowledge (never a wrong purchase) and every delete is logged.
- **D6 — Edge-audit decisions log as `edge_drop` / `edge_keep`, filtered out of the admin Decisions stream.** The normalization log's `outcome` gains two values (`term` = `"from -[kind]-> to"`, `detail.audit = "edge"` with the direction + reason; drops always logged, keeps logged when a classifier ran). The admin Decisions view (`normalize-admin.ts`) is term→identity-shaped (base/detail/candidates columns) and would mis-render edge rows as `novel`, so `readNormalizationPage` filters `edge_*` outcomes out of the stream in JS (no UI change; the rows stay queryable in D1 and the deletes remain fully audited). *Alternative rejected:* a new `DecisionKind` + badge — that drags in admin UI markup owned by the Claude Design project for a one-time backlog's log rows.
- **D7 — `sku_cache` re-key is pure code over the resolver, mirroring `grocery-pantry-reconcile`.** Each tick: load `readResolver` (NOT `ingredientContext` — the re-key must have **no capture side effect**, otherwise non-food/legacy cache keys would be enqueued and minted as graph nodes, breaking "non-food rows stay as-is by construction"), resolve each `sku_cache.ingredient` via `normalizeIngredient` (alias front-door, representative-baked), and plan re-keys in JS: rows whose resolution differs move to the canonical key; on a (canonical, location_id) collision — with an existing canonical row or another moving row — the row with the newer `last_used` wins whole (sku/brand/size travel together; NULL loses; tie keeps the already-canonical row). DELETEs + the winning upsert ride one batch per tick, bounded (`SKU_REKEY_MAX_PER_TICK = 200`, `truncated` in the summary — the grocery-reconcile shape). Idempotent and stampless: convergence is a pure function of current resolver state, and re-running over canonical rows plans nothing. A resolver read failure skips the tick's re-key (fail-safe: never re-key on partial state).
- **D8 — Re-confirm guard parity.** `ReconfirmDeps` gains `confirmMin` (wired to `NORMALIZE_CONFIRM_MIN`); `reconfirmOne` rejects a `same` or `specialization` pick whose chosen candidate's cosine (from the pass's own ranked list) is below it — logging the same `note: "confirm_below_min"` + `rejected {outcome, match, score}` detail shape capture uses, committing no merge and no edges (a logged no-op; the node stays stamped-novel). This closes the hardening design's flagged risk with the identical semantics capture applies to its picks.
- **D9 — Jobs, wiring, observability.** Three new pass modules (`src/ingredient-alias-audit.ts`, `src/ingredient-edge-audit.ts`, `src/sku-cache-rekey.ts`), each the sibling shape: a pure core over injected deps, a `build*Deps(env)`, and a `run*Job(env, deps)` wrapper writing `job_health` + `job_runs` (names `ingredient-alias-audit`, `ingredient-edge-audit`, `sku-cache-rekey`) and rethrowing on hard failure. Wired into `scheduled()` Phase 1 after `runReconfirmJob`/`runReconcileJob`, like their siblings. NOT added to `HEALTH_JOBS` — mirroring `ingredient-normalize`/`ingredient-reconfirm`/`grocery-reconcile`, which surface through the all-jobs run log (`/admin/logs` reads `job_runs` across every name) and the Normalization admin area rather than the public `/health` aggregate. Summaries (tenant-clean counts): alias `{ audited, self_stamped, kept, repointed, minted, merged, skipped }`; edge `{ audited, self_loops, cycles, dropped, kept, skipped }`; rekey `{ rekeyed, merged, truncated }`.
- **D10 — Per-tick budgets.** `ALIAS_AUDIT_MAX_PER_TICK = 20` rows (self-aliases stamp free, so the ~288-row bulk sweeps in ~16 ticks ≈ 80 min; the ~34 classifier confirms spread over those same ticks), `EDGE_AUDIT_MAX_PER_TICK = 10` (≈85 edges → ~9 ticks ≈ 45 min; worst case one LLM call per edge). Both passes then quiesce: eligibility selects nothing, zero model calls. Well inside "first full sweep completes within hours" on the 5-minute cron, and small enough per tick to coexist with capture/re-confirm on the internal `env.AI` budget.

## Risks / Trade-offs

- [~34 one-time LLM calls re-auditing aliases a cosine filter would have skipped] → Deliberate: embedding distance cannot see the distinct-product defect class this change exists to fix (4 of 6 fixture aliases sit above the guard). The cost is bounded by the auto-alias backlog size, spent once, and the passes quiesce permanently after the sweep.
- [The orphan merge asserts N ≈ target from indirect evidence (the classifier attested the VARIANT, not node N)] → Scoped hard: auto nodes only, zero remaining aliases, non-destructive representative pointer, its own `merge` log row. The alternative (leave orphans) keeps nonsense retrieval candidates alive — worse for every future confirm.
- [The classifier may re-affirm a wrong mapping (it produced some of these rows)] → The re-audit runs the HARDENED prompt (distinct-product rules, scored candidates) that was built against exactly these fixtures, plus the pick guard. A re-affirmed wrong row is stamped but logged with candidates + `detail.audit`, and a human override (`update_aliases`) always wins afterward.
- [Edge deletion is destructive where the rest of the pipeline never deletes] → Edges are open-world hints (a missing edge degrades to world knowledge, never a wrong purchase); scope is the bounded auto backlog + logged drops; human edges are never touched. A wrongly-dropped edge can be re-proposed by any later capture/re-confirm commit.
- [A 2-cycle against a human reverse edge deletes the auto edge without a model call] → Human authority is the established precedence rule everywhere else in the layer; asking a model to overrule an operator would invert it.
- [A transiently-failing row at the head of the audit order blocks its slot each tick (no backoff column)] → Same accepted behavior as re-confirm (un-stamped IS the retry state); the bound means one bad row costs one slot, not the tick, and a persistent AI outage already fails the sibling jobs' ticks wholesale.
- [Stamping keeps growing writes on the hot alias table] → One UPDATE per audited row, once ever; born-audited writes make the steady state zero.

## Migration Plan

1. `migrations/d1/0035_reaudit.sql` — two additive nullable columns (`ingredient_alias.audited_at`, `ingredient_edge.audited_at`); applied by the deploy's `wrangler d1 migrations apply --remote`, instantly valid for existing rows (NULL = backlog).
2. Worker deploy ships the passes; they begin sweeping on the next cron tick, converge within hours, then quiesce.
3. Rollback: revert the commit — the columns are inert without the code; stamped rows simply stop being read.
