## Why

The recipe `description` is **already** an AI-written, machine-derived field — `docs/SCHEMAS.md` annotates it "AI-written brief summary; embed source + compact candidate row" — yet it is **authored into the recipe's YAML frontmatter**, the one tier the data model reserves for *human-authored* content. That is a category error with three concrete costs:

1. **Inconsistency.** Descriptions are written by whoever (or whatever) imported the recipe, in whatever voice. Because the description is the embed source *and* the compact candidate row meal-plan retrieval reasons over, heterogeneous descriptions degrade both semantic search quality and the "why this dish" surface.
2. **A split-brain derived pair.** The description's sibling — its embedding — is *already* derived in the Worker and stored in D1 (`recipe_embeddings`, reconciled on the cron). The description it is derived *from* lives in a different tier (frontmatter) on a different cadence (authored). The two halves of one derived artifact are produced and stored inconsistently.
3. **Authoring friction & churn.** A human (soon: in Obsidian) editing a recipe sees a long AI-blob in their frontmatter. Any regeneration — a model upgrade, a prompt change — would rewrite that authored file, and (once recipes sync from a file editor) cause spurious sync churn/conflicts on a field no human owns.

This change makes the description **what it already is** — a derived field — and puts it where derived fields belong: **generated by the Worker, stored in D1, never in frontmatter.** It also establishes a small, reusable **decision rule** for *which* recipe fields are derived-to-D1 vs authored-in-frontmatter, so later fields (`perishable_ingredients` is the obvious next; the classification facets are discussed) can follow the same principle deliberately rather than ad hoc.

It is deliberately landable **on the current GitHub-backed setup** — it needs no storage migration — so it can fix the consistency pain now *and* de-risk a later R2 migration (`r2-recipe-corpus`) by proving the "Worker derives + validates metadata on the reconcile" loop first, against the existing corpus.

## What Changes

- **The recipe `description` becomes Worker-generated, not authored.** A text model (`env.AI`, the same binding the embedding reconcile already uses) writes the ~60-token description from the recipe's human-authored content, on the existing scheduled reconcile.
- **`description` is relocated from frontmatter → a D1 derived table**, co-located with its embedding (the two derived halves of one artifact, same producer, same cadence, same `slug` key). It leaves the `recipe-metadata-contract` required-frontmatter set.
- **The reconcile gains a content-hash gate** (generalizing the existing `description_hash` gate): when a recipe's *human-authored* content changes, its description is regenerated and then re-embedded (the embed step already exists downstream — it now keys off the regenerated description automatically).
- **`create_recipe` / `update_recipe` stop accepting or requiring `description`;** `read_recipe` merges the D1 description at read time (the read path already merges D1 facets + overlay + `last_cooked`, so this is one more joined field).
- **A documented decision rule** — *a field lives in frontmatter iff a human is its author or corrector of record; a purely-derived, regenerable field lives in D1 and never touches the file* — recorded in the new capability and in `docs/ARCHITECTURE.md`. Only `description` migrates in this change; `perishable_ingredients` and the classification facets are captured as candidates with their gating questions, not moved here.

## Capabilities

### New Capabilities
- `derived-recipe-metadata`: the contract for recipe fields that are **derived by the Worker into D1** rather than authored — how they are generated (`env.AI` on the reconcile), change-detected (content hash), validated, read-merged, and the frontmatter-vs-D1 placement rule. `description` is its first member; the embedding is its existing prior art.

### Modified Capabilities
- `recipe-metadata-contract`: `description` is **removed** from the required *frontmatter* set and from the non-empty-string field group — it is no longer a frontmatter field at all (it becomes a `derived-recipe-metadata` field). All other required fields are unchanged.

## Impact

- **Worker (`src/`):** a new text-generation helper beside `embedding.ts` (`env.AI` chat/instruct call); the reconcile (`src/recipe-embeddings.ts`, likely renamed to a `recipe-derived` reconcile) gains a description-generation pass ahead of the embed pass, gated by a content hash; `src/recipe-contract.js` + `src/validate.ts` drop `description`; `src/recipes.ts` / `read_recipe` (`src/tools.ts`) merge the D1 description; `src/serialize.ts` stops writing `description` to frontmatter.
- **D1:** a migration co-locating the derived description with the embedding — either a `description` (+ `content_hash`) column on `recipe_embeddings` (renamed to `recipe_derived`), or a sibling `recipe_descriptions` table; the projecting build must **not** clobber it (the same separate-producer reasoning that put embeddings in a sibling table — the build's wholesale `DELETE FROM recipes` + re-INSERT cannot own a reconcile-written column).
- **Build (`scripts/`):** `build-indexes.mjs` stops projecting `description` into `recipes` (it is no longer in frontmatter); the static cookbook (`build-site.mjs`) reads the description from the derived table.
- **Docs (lockstep):** `docs/SCHEMAS.md` (move `description` out of the frontmatter block into the derived-D1 section + the new content-hash field), `docs/ARCHITECTURE.md` (the derived-metadata principle + the reconcile's new pass), `docs/TOOLS.md` (`create_recipe`/`update_recipe`/`read_recipe` description handling).
- **Quality risk owned here:** Workers AI description quality vs. the big import-time model — see `design.md`. Mitigated by it being a low-stakes ~60-token field whose *consistency* is the goal, with a quality eval as a task.
- **Relationship to other changes:** independent and landable today; a prerequisite-in-spirit for `r2-recipe-corpus` (it proves the reconcile-derives-and-validates loop and shrinks the authored frontmatter surface before storage moves) and for `obsidian-authoring-vault` (a derived field needs no authoring dropdown, so it is correctly *absent* from the vault schema).
