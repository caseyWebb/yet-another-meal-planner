## Why

The satellite is strictly **outbound-only** (a ratified hard constraint of the `satellite` capability): the LAN box calls out, the Worker never dials in. That posture is perfect for the push job the satellite already does — scrape a walled recipe page and POST it. But the two *future* jobs the satellite exists to unlock — **scanning loyalty/in-store sale prices** and **filling a store cart** — are not things the satellite discovers on its own. They are work the **Worker** decides on: which stores/terms to scan (derived operator-wide, à la the flyer warm's `buildPlan`), and which shopping list to fill (a specific tenant's resolved order). "The Worker tells the satellite what to do" would be a push — and a push is impossible under outbound-only.

The resolution is a **pull channel**: the satellite makes an outbound request to *claim* work the Worker has queued, does it locally behind the operator's own session, and reports results back through the existing observation intake. One pull mechanism serves **both** future capabilities. This change builds that channel — the generic claim/result contract, the D1 task queue, the two key-derived auth scopes, and the endpoints — while the sale-scan and order-fill capabilities that will *populate* the queue and *consume* the results are deliberately left to changes 3 and 4. Codifying the channel now, capability-agnostic, means those later capabilities plug into a settled seam instead of each inventing a bespoke inbound path that risks the outbound-only line.

## What Changes

- **A satellite-initiated pull channel** — new outbound-only routes the satellite calls to claim queued work and to report results. No Worker-initiated connection, no websocket, no Durable Object / stateful Worker on the data path: the Worker exposes only request/response endpoints, exactly as the `satellite` capability's outbound-only requirement demands. This change adds **no** push from the Worker.
- **New `/satellite/*` paths, added alongside the retained `/admin/api/ingest`** — `POST /satellite/tasks/claim` (the satellite claims a bounded batch of work its key is scoped to) and `POST /satellite/results` (the satellite reports observations for a claimed task and transitions its lifecycle). Both are authenticated by the **same** ingest-key mechanism (`recipe-ingestion`'s SHA-256-hashed bearer key, rate-limited); both live outside `/admin*`, so they are not subject to the Access gate. `/admin/api/ingest` is unchanged and is never moved (per the `satellite` capability's decision that future `/satellite/*` routes are **added**, never a relocation).
- **A D1 task queue (`satellite_tasks`) with a claim/lease lifecycle** — `pending → claimed → done | failed`. A claim atomically leases a bounded batch to the claiming key with an expiry; a claim whose lease expires becomes re-claimable, so a dropped satellite simply leaves its work pending (graceful degradation, like a stale flyer). A new migration adds the table and an additive nullable `tenant` column to `ingest_keys`.
- **Two auth scopes derived from the key's tenant binding.** An ingest key gains an **optional** tenant binding. A key with **no** binding is **operator-global** (the existing default — every recipe-scrape key stays operator-global, unbroken) and may claim **operator-scope** work (the future cross-tenant sale-scan plan; public-derived, keyed by `locationId` like the flyer cache). A key **bound to a tenant** may additionally claim that **tenant's** tenant-scope work (the future order-list) — never another tenant's.
- **A clean, extension-open task/result contract.** The task payload is a **discriminated union keyed by `kind`** with **no concrete kinds defined here** — the seam for `sale-scan`/`order-fill`, mirroring the `satellite` capability's observation union. Results flow back as that same **observation union** (extended later with `sale`/`order-status` kinds), correlated to the claimed task by `task_id`, reusing the raw-observation intake so the Worker re-derives every conclusion (sensor-not-judge, inherited).

**Explicitly out of scope** (changes 3 and 4): the `sale-scan` and `order-fill` capabilities themselves — no scan-plan producer, no order-list resolver, no `sale`/`order-status` observation kind, no scan/order business logic, no store-session or cart mechanics. This change specifies the **channel only**: the pull/claim/result mechanism and the queue, with the union's extension point open and the queue's producers/consumers left to the later capabilities.

## Capabilities

### New Capabilities
- `satellite-pull-channel`: the Worker-side pull channel a satellite claims work from. Owns: the outbound-only claim/result posture (no Worker push, inheriting the `satellite` capability's constraint), the `/satellite/tasks/claim` + `/satellite/results` routes (key-authed, outside the Access gate) added alongside the retained `/admin/api/ingest`, the D1 `satellite_tasks` queue and its `pending → claimed → done|failed` claim/lease lifecycle, atomic-claim + result-side-idempotency correctness, idempotent per-logical-key enqueue, the operator-vs-tenant auth scopes derived from the key's tenant binding, and the capability-tagged discriminated-union task contract whose results return as the observation union correlated by `task_id`. It defines **no** concrete task kind (the seam for changes 3/4).

### Modified Capabilities
- `recipe-ingestion`: the ingest key gains an **optional tenant binding** — minting MAY bind a key to a tenant; a key with no binding is operator-global (existing behavior, existing keys unaffected); the roster surfaces the binding. The key-auth carve-out, hashing, and revocation are otherwise unchanged.
- `operator-admin`: the Config › Ingest Keys editor's **Mint key** action gains an optional **bind-to-tenant** selector (defaulting to operator-global), and the roster shows each key's binding.

## Impact

- **D1:** one new migration (next free number, e.g. `0037_satellite_pull_channel.sql`): `CREATE TABLE satellite_tasks` (+ indexes, incl. a partial-unique index enforcing idempotent enqueue) and `ALTER TABLE ingest_keys ADD COLUMN tenant TEXT` (nullable — existing rows read as operator-global, no backfill). **No data migration:** the production spike shows `ingest_keys`, `ingest_candidates`, `ingest_pushes` all at **0 rows** — no key is in flight to break.
- **Shared contract:** `packages/contract` gains the task envelope (discriminated union keyed by `kind`, no concrete arms), the claim request/response, and the `/satellite/results` request (observation union + `task_id`), defined once so the Worker and the satellite can't drift — the same runtime-agnostic boundary the ingest contract uses.
- **Worker:** new `/satellite/tasks/claim` + `/satellite/results` handlers (key-authed, before the `/admin` Access gate is even reached since they are not `/admin*` paths); a queue module reached only through `src/db.ts` prepared-statement helpers (tools/handlers never touch `env.DB` directly); the `/satellite/results` intake reuses the `/admin/api/ingest` raw-observation persistence (a shared helper, not a re-implementation) plus the lifecycle transition.
- **Admin panel:** the Ingest Keys editor's Mint action gets the tenant selector (an `src/admin/**` change — routed through the companion Claude Design project, shipped with Playwright coverage per the repo rules).
- **Docs (lockstep):** `docs/SCHEMAS.md` (the `satellite_tasks` table, the `ingest_keys.tenant` column, the claim/result wire shapes); `docs/ARCHITECTURE.md` (the pull channel beside the push ingest arm and the flyer warm); `docs/SELF_HOSTING.md` (minting a tenant-bound key for a per-tenant satellite); `docs/TOOLS.md` (confirm unchanged — no MCP tool touched).
- **Outbound-only preserved:** every channel interaction is a satellite-initiated request/response. The Worker opens no connection toward the satellite; a home box behind NAT needs no inbound port.
