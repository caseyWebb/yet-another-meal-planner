## Why

Change 1 gave the satellite a **sensor-not-judge** contract (observations-only, keyed by item `kind`; the Worker re-derives every conclusion) and forward-referenced a future capability that reports "raw measurements it derives a saving from." Change 2 built the **pull channel** — an outbound-only claim/result queue whose `TaskEnvelope` `kind` enum has **no concrete member**, "the seam a later capability (sale-scan, order-fill) extends." Both changes deliberately shipped bare seams. This change is the **first real producer + consumer** that fills them.

The initiative was scoped around one capability the Worker cannot do itself: **scan in-store / loyalty sale prices at stores the Worker has no API for** (e.g. Target) and feed those sales into menu-gen the **same way** the Kroger flyer already does. The Worker synthesizes a Kroger flyer by scanning the public API on a cron (`flyer-cache-warming`); for a walled loyalty store there is no API to scan, so the sale prices live behind the operator's own authenticated browser session — exactly the satellite's domain. The satellite claims an operator-scope `sale-scan` task, does the browser scan behind the operator's session, and reports **raw** `{ regular, promo }` price observations; the Worker re-derives the savings and the deal floor at read, identically to the Kroger path. Nothing about a sale is trusted from the wire — the satellite is a sensor.

This exercises every ratified constraint of changes 1–2 on a concrete, non-recipe observation for the first time: the observation union gains its first non-`recipe` kind (`sale`); the task union gains its first concrete kind (`sale-scan`, operator-scope); the raw-observation intake gains its first non-recipe arm; and the Kroger flyer's cross-tenant rollup — "the one deliberately cross-tenant data plane" — is generalized so Kroger and satellite sales **converge at the same raw layer** and downstream reads treat them uniformly.

## What Changes

- **A `sale` observation kind** (extends change 1's `ObservationItem` union). It carries **raw facts only** — `{ store, locationId, productId, description, size, regular, promo }` (+ optional `brand`/`categories`/`url`) — and **never** a derived `savings`/`savings_pct`. The Worker re-derives "on sale" (`promo > 0 && promo < regular`, excluding a `promo == regular` echo) and applies the `min_savings_pct` deal floor at **read**, byte-for-byte the same rule the Kroger flyer uses. This is the canonical sensor-not-judge example the change-1 spec forward-referenced.
- **A `sale-scan` task kind** (extends change 2's `TaskEnvelope` union), **operator-scope** — public-derived, cross-tenant, keyed by `locationId`, the same posture as the flyer cache (never tenant-scope, so no per-tenant sale data). Payload: `{ store, locationId, terms }`.
- **A Worker-side scan-plan producer** — a sibling cron job in the one `scheduled()` handler (not folded into the Kroger flyer tick) that **enqueues** `sale-scan` tasks for **non-Kroger** stores, derived from tenants' preferred stores × the shared `flyer_terms`, idempotent per `(store, locationId)` and refresh-gated like the flyer sweep. The Worker only **enqueues** — it never scans (it has no API); the satellite claims + scans + reports.
- **A store-namespaced flyer rollup.** The rollup key generalizes from `flyer:{locationId}` to **`flyer:{store}:{locationId}`** (Kroger writes `flyer:kroger:{locationId}`; a satellite scan writes `flyer:target:{locationId}`), reusing the existing `FlyerItem` / `isOnSale` / merge layer so Kroger and satellite sales land in the **identical raw rollup shape**. The Kroger read path falls back to the legacy `flyer:{locationId}` key until the first namespaced sweep completes, so there is **no cold gap on deploy** and **no data migration** — the ephemeral cache converges through the pipeline.
- **A store-aware read tool, `store_flyer`** — resolves the caller's primary fulfillment store and returns that store's `{ items, as_of }` rollup, whether Kroger or satellite-scanned, applying the same `min_savings_pct`-at-read floor and a staleness ceiling for scanned flyers. `kroger_flyer` is **retained unchanged** (now reading the Kroger-namespaced key with the legacy fallback) so the existing Kroger flow is byte-for-byte preserved; `store_flyer` supersedes it in the general menu-gen pre-pass so satellite sales flow into selection the same way Kroger sales do.
- **Satellite-side sale-scan.** The satellite gains a **pull-channel client** (claim → run → report over the change-2 routes — the first capability to consume it) and a **`sale-scan` adapter interface** parallel to the recipe adapter, over a shared SDK (fetch tier + session + logger). Consistent with the initiative principle "core ships host + contract + SDK, users ship the ToS-hostile drivers," **no named-retailer sale adapter ships built-in** — a store's scan adapter is operator-authored, loaded from the mounted adapters directory. The satellite validates every emitted `sale` observation against the shared contract before reporting.
- **Sensor validation for `sale`** — plausibility bounds (`0 < promo < regular`; savings ≤ a 95% ceiling; sane sizes/prices), provenance (`productId`, optional product `url`, for spot-checkability), and Worker-side re-derivation. Satellite sale data enters the **same** raw-observation layer + **equal-or-stricter** validation as Kroger, with **no privileged path**.

## Capabilities

### New Capabilities
- `satellite-sale-scan`: the first satellite producer+consumer capability. Owns: the `sale` observation kind (raw price facts, no derived saving); the operator-scope `sale-scan` task kind + payload; the Worker scan-plan producer (enqueue-only, refresh-gated, idempotent per `(store, locationId)`, `sale-scan-plan` health); the sale intake arm that validates + re-derives + converges satellite sales into the store-namespaced rollup at the same raw layer as Kroger; the `sale` plausibility/provenance validation; the store-aware `store_flyer` read; and the satellite-side sale-scan pull client + adapter interface (no built-in named-retailer driver).

### Modified Capabilities
- `satellite`: the capability enumeration admits `sale-scan` alongside `recipe-scrape`, and the observation union admits `sale` as its first non-`recipe` kind — realizing the forward-looking sensor-not-judge scenario ("a future capability needs a derived quantity, e.g. a saving → the wire carries only the raw measurements"). No privileged satellite path; the sale intake is held to equal-or-stricter validation than a first-party Kroger scan.
- `flyer-cache-warming`: the per-location rollup is **store-namespaced** (`flyer:{store}:{locationId}`); the Kroger warm writes the `kroger` namespace; the rollup/`FlyerItem` layer is store-agnostic so satellite sales converge there; a read-time fallback to the legacy key covers the deploy window (no data migration).
- `kroger-integration`: `kroger_flyer` reads the store-namespaced `flyer:kroger:{locationId}` key (legacy-key fallback), its `{ items, as_of }` contract otherwise unchanged.
- `menu-generation`: the context pre-pass flyer gate generalizes from **Kroger-only** to the caller's **primary store when it has a warmed flyer** (Kroger via `kroger_flyer`, a satellite-scanned store via `store_flyer`), and sale-steering generalizes to a scanned store's Worker-re-derived savings — so satellite sales flow into selection the same way Kroger sales do.

## Impact

- **No D1 migration.** The producer reuses the existing `satellite_tasks` queue (change 2) and its idempotent enqueue; the rollup is an ephemeral KV cache. The store-namespacing is a KV **key-shape** change that converges through the next sweep/scan, not a migration. (A KV `sale-scan:cursor` refresh marker is added, mirroring `flyer:cursor`.)
- **Shared contract:** `packages/contract` gains the `sale` observation arm (raw facts, no `savings`) and the `sale-scan` task-payload shape, defined once so Worker and satellite can't drift — the same boundary the recipe contract uses.
- **Worker:** the raw-observation intake gains a **`sale` arm** (dispatched by `kind`; recipe → `ingest_candidates`, sale → the store rollup) reached from `/satellite/results`; the scan-plan producer (a `scheduled()` sibling, `sale-scan-plan` in `HEALTH_JOBS`); the store-namespaced rollup key + `store_flyer` read; `kroger_flyer`'s namespaced read. All D1 access stays through `src/db.ts`; the producer prunes terminal sale-scan rows so the recurring queue stays bounded.
- **Satellite package:** a pull-channel client (claim/report) + a `sale-scan` adapter interface + SDK primitives + a CLI `test` verb for a sale adapter. No built-in named-retailer driver.
- **Admin panel:** the operator Status card gains the `sale-scan-plan` job row (a data-only addition to the registered-jobs render) — shipped with its Playwright seed + screenshot per the repo rule.
- **Persona:** `AGENT_INSTRUCTIONS.md` menu-gen pre-pass calls the store-aware flyer read for a non-Kroger primary store; validated conversationally.
- **Docs (lockstep):** `docs/SCHEMAS.md` (the store-namespaced `flyer:{store}:{locationId}` rollup + the `sale-scan:cursor`, the `sale` observation + `sale-scan` payload wire shapes), `docs/ARCHITECTURE.md` (the sale-scan loop beside the flyer warm + pull channel; the cross-tenant plane generalized to all stores), `docs/TOOLS.md` (`store_flyer`; `kroger_flyer` namespaced note).

**Explicitly out of scope** (changes 4 and 5): **order-fill** — no cart mechanics, no `order-status` observation, no tenant-scope order task; and the **sensor-audit / quarantine sampling** — this change ships the raw `sale` validation + provenance, and notes where change 5 would sample these sale claims, but builds no sampling/quarantine surface. This change is the scan → `sale`-observation → store-rollup → generalized-read loop only.
