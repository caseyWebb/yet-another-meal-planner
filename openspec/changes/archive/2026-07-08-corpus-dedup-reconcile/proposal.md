# Proposal ‚Äî corpus-dedup-reconcile

## Why

Issue #217: "Fresh Pasta" and "Pasta Dough" (production slugs `fresh-pasta` / `homemade-pasta-dough`) sit in the corpus as unrecognized duplicates. Dedup today runs **only at import time**: `discovery-sweep.ts`'s `findDuplicate` compares an incoming *candidate* against the corpus (description-embedding cosine ‚â• 0.9) ‚Äî recipes already **in** the corpus are never compared to each other, so any pair that predates the sweep (the ReciMe import era) or slipped past the import gate coexists forever. The `recipe-import` spec already carries the standing contract for what should happen ‚Äî *"Near-duplicate reconciliation without auto-merge: a final pass SHALL surface near-duplicate recipes for human review rather than merging them automatically"* ‚Äî but no ongoing mechanism implements it.

The production spike (see design.md) sharpened the problem: the defect pair's description-embedding cosine is only **0.7670** ‚Äî *below* the import dedup threshold and below every plausible cosine-only threshold (the corpus has **zero** pairs at ‚â• 0.90 and the pair ranks outside the top-50 at 0.80+). Cosine alone cannot surface this pair without drowning the operator in false positives. But the pair's `ingredients_key` overlap is maximal ({flour, eggs} ‚äÜ {flour, eggs, olive oil} ‚Äî Jaccard 0.67, containment 1.0), and a corroborated rule (moderate cosine AND strong ingredient overlap) surfaces it with only ~7 candidate pairs corpus-wide, several of which are genuine near-duplicates (including `butter-chicken` / `pressure-cooker-butter-chicken` ‚Äî the spec's own canonical example).

## What Changes

- **A corpus-wide near-duplicate reconcile (`dup-scan`).** A new scheduled job in the one `scheduled()` handler (Phase 5, beside `reconcile-signals` / archetype derivation) compares corpus recipes' description vectors, corroborated by `ingredients_key` overlap ‚Äî both already exist on the index (`recipe_derived.embedding`, `recipes.ingredients_key`). Detection is the **calibrated two-arm rule**: cosine ‚â• 0.90 alone (the import-dedup analog, catches paraphrase twins), OR cosine ‚â• 0.72 AND ingredients-key Jaccard ‚â• 0.5 AND ‚â• 2 shared ingredients (the corroborated arm the fixture pair needs). No model calls, no external subrequests ‚Äî pure arithmetic over D1 rows.
- **Bounded + watermarked, never O(n¬≤) per tick.** Each recipe carries a scan stamp (a new tiny `dup_scan` D1 table: slug ‚Üí hash of its embedded-description hash + effective `ingredients_key`). A tick scans at most `DUP_SCAN_MAX_PER_TICK` (25) unstamped/stale recipes, each against the full current vector set, then stamps them. The backlog (205 recipes today) drains in ‚â§ 9 five-minute ticks; a converged corpus is a no-op tick; a re-described or re-faceted recipe re-queues itself organically. Pair coverage holds because whichever member of a pair is scanned later is compared against a vector set that contains the other.
- **Pairs surface as operator proposals ‚Äî no auto-merge.** Each detected pair enqueues one `merge_recipes` proposal into the existing `pending_proposals` queue (`reconcile-db.ts`), addressed to the **operator tenant** (`OWNER_TENANT_ID` ‚Äî corpus curation is operator-trusted, consistent with the reconcile's operator-privileged surface). The existing stable `(kind, target)` id + `INSERT OR IGNORE` gives idempotent re-detection and permanent suppression of dismissed pairs for free. Payload carries the slugs, titles, cosine, and shared-ingredient evidence. With no operator tenant configured the job no-ops **without stamping**, preserving the backlog.
- **Confirming is an agent-guided merge via existing tools ‚Äî non-destructive.** Accepting a `merge_recipes` proposal applies **no automatic corpus write** (`applyProposal` records the decision only). The merge itself happens conversationally: the agent reads both recipes + notes, folds anything worth keeping into the survivor via `update_recipe`, re-points `pairs_with` referrers, then marks the duplicate with a `duplicate_of: <survivor-slug>` frontmatter field (rides `update_recipe`'s pass-through frontmatter). The **projection** learns to exclude `duplicate_of`-marked files from the index deliberately (not an error) ‚Äî the duplicate leaves search/menu-gen/list on the next tick, its `recipe_derived` row prunes via the existing orphan prune (so it can never re-trigger detection), and removing the marker restores it. The file, its notes, and cooking-log history are never destroyed.
- **The member app renders the new kind honestly.** The profile screen's reconcile queue gains a `merge_recipes` branch: pair-title + rationale + **Dismiss** only (the merge is chat-guided; no synthetic accept button, per the app's kind-specific-actions rule).
- **Acceptance fixture:** after deploy, within the drain window the operator's queue MUST hold a pending `merge_recipes` proposal for `fresh-pasta` + `homemade-pasta-dough` (verified with a read-only production SELECT).

## Capabilities

### New Capabilities

- **`recipe-dedup`** ‚Äî the corpus-wide near-duplicate reconcile: the bounded, watermarked scheduled scan over description vectors + `ingredients_key`; the calibrated two-arm detection rule; the `merge_recipes` operator proposal (surfacing, idempotence, dismissal permanence, no-auto-merge); the agent-guided non-destructive merge flow (`duplicate_of` tombstone); job health.

### Modified Capabilities

- **`recipe-import`** ‚Äî the standing "Near-duplicate reconciliation without auto-merge" requirement becomes the **ongoing scheduled reconcile's** contract (surfaced as operator proposals), not a one-time import pass.
- **`recipe-index`** ‚Äî the projection excludes a recipe whose frontmatter carries `duplicate_of` from the rebuilt index **deliberately** (no `reconcile_errors` row; counted in the projection summary); removal of the marker restores the row.
- **`profile-reconciliation`** ‚Äî the pending-proposals queue is explicitly kind-extensible beyond profile diffs: corpus-curation kinds (`merge_recipes`) are addressed to the operator tenant, and their **accept records the decision without applying a profile/corpus write** (the action is agent-guided); reject still suppresses permanently.
- **`member-app-core`** ‚Äî the reconciliation queue's kind-specific rendering covers `merge_recipes` (pair title, rationale, Dismiss; no synthetic merge action).
