# Proposal — member-app-core

## Why

The member web app plan (`docs/plans/web-app.md` on the plan branch, §10 phase P1) calls for the member-facing core: every screen a member touches weekly — cookbook browse/search/detail, favorites, meal plan, grocery list, pantry, cooking log, notes, profile, and the night-vibe palette + reconciliation queue — as a UI over **existing deterministic Worker operations**. The Claude.ai plugin stays the capture-rich surface; the app is the retrieve/act surface, deep-linking out ("Cook with Claude") rather than embedding a model. P0 (member-web-app foundations: invite-code session auth, the `/api` mount + shared error/ETag middleware, the `packages/app` + `packages/ui` scaffold, the app Playwright harness) is assumed landed; this change fills the mount with the member core.

It also lands **W3**, the grocery `status` lifecycle enforcement: `update_grocery_list` today exposes `status: active | in_cart | ordered` completely unguarded (`packages/worker/src/grocery-tools.ts`), so any caller — and, once the app ships, any member tap — can mint `ordered` from thin air, bypassing the order flow. The lifecycle intent lives only in prose (a module header comment and `docs/TOOLS.md`'s lifecycle notes). Grounding against the actual code corrected the plan's premise: `place_order` never writes `ordered` (it advances to `in_cart` via `advanceInCartRows`); `ordered` is reached by the **user-asserted** "I placed the order" advance *via `update_grocery_list`* (documented in `docs/TOOLS.md`) and by the satellite receipt flush (`advanceOrderedRows`). So W3 is a **transition guard**, not a flat reject: `active ⇄ in_cart` stays freely writable, `in_cart → ordered` (the user-asserted advance) is kept and now stamps `ordered_at`, and every other path into `ordered` is rejected with a structured error — stated as a guarantee in the tool description and `docs/TOOLS.md`.

Production grounding (read-only D1 spike, 2026-07-07, see design.md): 200 recipes, 337 pantry rows, a grocery list that is 100% `active` (the guard converges trivially — no defect rows exist), an **empty night-vibe palette with 47 pending `add_vibe` proposals** across 2 tenants (the reconciliation queue ships with real content and the palette's empty state is the actual first render), and a fresh, healthy `archetype-derive` job (the suggest endpoint's ~20h gate would throttle today, as designed).

## What Changes

- **Per-area `/api` JSON route groups** (chained Hono sub-apps under the P0 mount, all session-gated, typed for `hc`): `cookbook` (index, keyword search, new-for-me, recipe detail, similar, notes), `overlay` (favorites), `plan`, `grocery`, `pantry`, `log`, `profile` (assembled profile, preferences merge-patch, taste/diet-principles markdown, retrospective, Kroger login URL), and `vibes` (palette CRUD, pending proposals + confirm, gated suggest). Every endpoint calls a **named existing `src/` operation**; the handful of ops that today live only inside MCP tool closures (`read_recipe` assembly, `log_cooked`, `update_preferences` apply, `read_user_profile` assembly, night-vibe add/update, `confirm_proposal`) are **extracted into shared operations** the tool and the route both call — zero tool-behavior change, existing Worker tests stay green.
- **W3 — grocery `status` transition guard** in the shared op layer (`updateGroceryRow` / pure `grocery.ts` check): `active ⇄ in_cart` free; `in_cart → ordered` allowed (the user-asserted advance, now stamping `ordered_at`); any other write of `ordered` → structured `validation_failed`, row unchanged. The order-flow advance functions (`advanceInCartRows`, `advanceOrderedRows`) are separate code paths and are untouched. Tool description + `docs/TOOLS.md` state the guarantee. The member route additionally accepts only `active | in_cart` for `status` at its boundary.
- **`update_meal_plan` gains a `set` op** (exposed on the MCP tool and the route in the same pass, per the operator's one-contract decision): replace a planned row's `sides` wholesale (today's `add` is union-only — a side can never be removed) and set or **explicitly clear** `planned_for` (today's `add` cannot clear a date). Required by the meal-plan page's side-chip removal and date clearing; `docs/TOOLS.md` + the `meal-planning` spec update in lockstep.
- **Cooking-log list + delete ops** (thin, new, web-only — no new MCP tool): a bounded most-recent-first read of the caller's `cooking_log` (recipe rows enriched with title/facets via the existing `recipes` join idiom) and a tenant-scoped delete by `id` (the table already has an `INTEGER PRIMARY KEY`). The plan's §4 table listed "delete" as if it existed; it does not — see design.md.
- **Gated vibe suggestion endpoint**: `POST /api/vibes/suggest` calls the existing `runDerivation`, but **only after** a `readJobHealth("archetype-derive")` freshness check (~20h, mirroring the cron's `DERIVE_INTERVAL_MS`) — fresh-and-healthy returns a throttled response without touching `env.AI`. The MCP tool `suggest_night_vibes` is agent-mediated and stays ungated (today's behavior; grounded finding — the plan assumed the gate existed on the tool).
- **`packages/app` pages** (React 19 + TanStack Router/Query + shadcn/ui per the design bundle `docs/plans/web-app-design/project/cookbook/`, matched pixel-perfectly in the target stack): app shell + sidebar, cookbook browse/search, recipe detail (notes, similar, Cook-with-Claude deep link, add-to-plan, log-cooked, favorite), favorites, meal plan, grocery (category grouping; **no** substitutions/aisles/store-picker/derived-to-buy — later phases), pantry (with the needs-verification section), cooking log, profile (taste read from `retrospective`, preferences, dietary, brands, stores, Kroger link), night-vibe palette + reconciliation queue. Shared primitives land in `packages/ui`.
- **Two-writer posture applied**: every P1 write endpoint is classified class (a) whole-document If-Match (412 on lost race) or class (b) idempotent canonical-id upsert (offline-replay safe, never If-Match) — the classification table in design.md is normative. Toggle-shaped interactions (favorite, in-cart) become **explicit set** operations so replay converges.
- **Playwright coverage** for every page on the P0 app harness, blocking in CI, screenshots surfaced — same rule as the admin panel.

## Capabilities

### New Capabilities

- **`member-app-core`** — the member app's P1 surface: the per-area session-gated `/api` JSON route groups over named existing ops (with the tool-closure extractions), the pages listed above, the write-class discipline (If-Match vs canonical-id idempotent), the gated vibe-suggest endpoint, and the per-page Playwright gate.

### Modified Capabilities

- **`grocery-list`** — the CRUD-tools requirement gains the W3 status-transition guard: `active ⇄ in_cart` free, `in_cart → ordered` user-asserted (stamps `ordered_at`), all other `ordered` writes rejected with structured `validation_failed`.
- **`meal-planning`** — `update_meal_plan` gains the `set` op (wholesale `sides` replace incl. removal; explicit `planned_for` set/clear) beside the existing union-only `add` and `remove`.
- **`cooking-history`** — gains member log corrections: a bounded list read and a tenant-scoped delete-by-id over `cooking_log`, serving the web surface (no new MCP tool).
- **`night-vibe-archetype-derivation`** — the on-demand requirement now distinguishes surfaces: a member-tappable app trigger is gated by the archetype-derive job's recorded health (~20h) so it cannot spend `env.AI` unboundedly; the agent-mediated tool path is unchanged.

## Impact

- **No D1 migrations.** All new reads/writes fit existing tables (`cooking_log` already has its `id` PK; conditional requests use hash-of-representation ETags, not new columns). `docs/SCHEMAS.md` is untouched.
- **Worker** (`packages/worker/src/`): op extractions (behavior-preserving) in `tools.ts`/`cooking-write.ts`/`write-tools.ts`/`notes-tools.ts`/`night-vibe-tools.ts`/`reconcile-tools.ts` → shared functions; the W3 guard in `grocery.ts` + `session-db.ts`; the `set` op in `meal-plan.ts` + `session-db.ts` + `cooking-tools.ts`; new `readCookingLog`/`deleteCookingLogRow`; the new `/api/*` route modules mounted by the P0 app; per-route analytics to the existing `TOOL_AE` dataset (or a sibling).
- **Frontend**: `packages/app` pages + query/mutation hooks; `packages/ui` shared components (Basecoat → shadcn/ui mapping per the committed design bundle).
- **Docs (lockstep)**: `docs/TOOLS.md` (`update_grocery_list` guard + lifecycle notes; `update_meal_plan` `set` op), `docs/ARCHITECTURE.md` (the member app's P1 page/API areas). `AGENT_INSTRUCTIONS.md` needs no change: the documented agent flows (user-asserted `in_cart → ordered` advance) remain valid under the guard.
- **Tests**: Worker unit tests for the guard, the `set` op, the log ops, the extractions, and the route groups; Playwright page objects + specs for every page.

## Dependency

**Requires P0 (member-web-app foundations) to have landed**: the invite-code session (cookie + middleware yielding the resolved tenant), the `/api` Hono mount with shared error→HTTP and ETag/`X-App-Build` middleware, the `packages/app`/`packages/ui` workspaces with the build wiring, and the app Playwright harness. Tasks in this change are written against that baseline and mount into it; if P0's shapes differ in detail (middleware export names, harness paths), the tasks name the *role*, not a hardcoded path, and the implementer binds them to P0's actuals.
