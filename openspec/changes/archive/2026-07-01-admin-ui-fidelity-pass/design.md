## Context

The 8-area admin-panel redesign (change `admin-ui-redesign-*`, archived, merged as `#159`) shipped the Basecoat-based Status, Members, Data (Recipes/Stores/Guidance), Usage, Discovery, Logs, and Config areas. The operator (repo owner) reviewed the live panel against the approved design mock (`StatusScreen.jsx`, `MembersScreen.jsx`, `RecipesScreen.jsx`, `RecipeDetail.jsx`, `StoresScreen.jsx`, `GuidanceScreen.jsx`, `UsageScreen.jsx`, `DiscoveryScreen.jsx`, `LogsScreen.jsx`, `ConfigScreen.jsx`, plus the shared `hover-tip.jsx`, `icons.jsx`, `pretty-kv.jsx`) and filed a punch-list. Investigation (this change) confirmed the punch-list splits into two kinds of work: **pure visual fidelity** against an already-approved design (no spec change — captured as tasks only), and a handful of **genuine bugs** whose root cause needed diagnosis before scoping the fix. It also surfaced **cross-cutting primitives** the mock already factors out (a shared sparkline+tooltip, a `PrettyKV` renderer) that the current shipped code duplicates or omits per-area — the implementation should build each once in the shared kit (`src/admin/ui/kit.tsx` for presentational pieces, a new `src/admin/client/*` module for the interactive tooltip) rather than per-page.

## Goals / Non-Goals

**Goals:**
- Close the fidelity gap between the shipped panel and the approved mock, area by area, per the punch-list.
- Root-cause and fix (or conclusively explain as "working as designed, config gap") the five reported bugs: Stores-empty, Members-status, dropdown-navigates, slider-fill, Usage-colors.
- Extract the shared primitives the mock already models as reusable (`useTip`/`PrettyKV`-equivalent, trash-icon remove button, frontmatter-stripping markdown helper, list-value badge/pill renderer) so each area consumes one implementation.
- Extend the two spec-relevant behaviors called out by the operator: Status stat-tile navigation (RSS Feeds, Cached SKUs) and Recipes' page size.

**Non-Goals:**
- No new capability domain and no new D1 tables — this is a fix/fidelity pass over the already-approved, already-specified redesign.
- Not re-litigating the redesign's information architecture (area boundaries, routing) — only the punch-list's items.
- Not building a generic slider/tooltip component library beyond what these areas need.

## Decisions

**Decision 1 — Bugs get root-caused before tasks are written, not fixed blind.**
Read-only investigation (this change's exploration) against the live `src/` code, cross-referenced with `test/tenant.test.ts` and a local D1 query, established:

- **Members status ("awaiting claude.ai connection")**: `listTenants` (`src/admin.ts:235-268`) derives `active`/`pending` from presence of a `tenant_activity` row; the write path (`touchTenantActivity`, `src/tenant.ts:121-152`) is correctly wired with `recordSeen=true` on the real MCP hot path (`src/index.ts:41`). The table (`migrations/d1/0024_tenant_activity.sql`) is new — it landed in the same PR as the redesign. **Diagnosis: most likely legitimately "no activity recorded yet"** (neither member has completed an MCP tool call against the redeployed Worker since the migration landed) — **not a logic bug**. One latent risk found in passing: `touchTenantActivity` swallows all storage errors in a bare `catch {}`, so a genuinely failed write (e.g. the migration not yet applied `--remote`) would present identically and silently. Task: confirm the migration is live in the operator's remote D1 and that a real MCP call updates the row before concluding "no bug"; if it doesn't update, surface the swallowed error via a structured log so this failure mode is distinguishable in the future.
- **Members dropdown navigates instead of opening**: the whole roster row is an `<a href="/admin/members/:id">` (`src/admin/client/members.tsx:198`); the `RowMenu`'s action buttons correctly call `preventDefault()`+`stopPropagation()` via a `stop()` wrapper (lines 143-148, used at 164/168/172), but the **trigger button itself** (`onClick={() => setOpen((v) => !v)}`, lines 151-162) has neither — a plain toggle with no event guard. **Root cause: the trigger's onClick is missing `preventDefault`/`stopPropagation`**, unlike its sibling action buttons. Fix: apply the same `stop()`-style guard to the trigger (toggle `open`, but stop the event before it reaches the ancestor `<a>`).
- **Data > Stores empty**: `storeList`/`storeDetail` (`src/admin-data.ts:611-690`) query `stores`/`store_notes`/`sku_cache` correctly (no incorrect tenant filter — stores are shared/cross-tenant per spec) against the schema in `migrations/d1/0006_shared_corpus.sql`. A local D1 query confirms `stores` is genuinely empty. **Diagnosis: legitimately empty — no `add_store` call has been made in this environment**, not a query bug. Task: verify against the *operator's* environment (not local dev) whether the same holds; if the operator's D1 has store rows and the explorer still renders empty, the bug is elsewhere (e.g. a route/env mismatch) and needs re-diagnosis.
- **Slider fill fixed regardless of position**: `Slider` (`src/admin/ui/kit.tsx:250-262`) renders a bare `<input type=range>` with no `style` prop; Basecoat's Vega stylesheet fills the track via a `--slider-value` CSS custom property with a **hardcoded fallback of `20%`**, and nothing in `kit.tsx` or its caller (`KnobRow`, `src/admin/client/knob-console.tsx`) ever computes `(value - min) / (max - min) * 100` and sets `--slider-value` accordingly. **Root cause confirmed: a genuine bug** — every slider falls back to the same hardcoded 20% fill. Fix: `KnobRow`'s `onInput` handler (or a wrapper around `Slider`) sets `style="--slider-value: {pct}%"` on the input, recomputed on every change (SSR-rendered initial value included, so first paint is also correct, not just post-hydration).
- **Usage namespace colors all grey**: `resolveNamespaceLabel` (`src/usage.ts:149-158`) requires the operator-pasted `KV_NAMESPACE_LABELS` env var (documented as optional in `docs/SELF_HOSTING.md`); absent, every namespace resolves to the `unlabeled`/grey fallback by design. **Diagnosis: not a lookup/CSS bug — a config gap** (the var is genuinely unset in this environment). This is working as specified, but is a rough operator experience: on a fresh deploy, KV namespace ids ARE known (they're in `wrangler.jsonc`'s `kv_namespaces` block) — the Worker just has no runtime API to map its own binding's id back to a name. Task (fidelity, not a spec change): consider defaulting `KV_NAMESPACE_LABELS` at deploy time from the wrangler config's own namespace ids so a fresh deploy shows correct colors without a manual paste step; at minimum, confirm this operator's env is missing the var and, if so, treat it as a docs/deploy-defaults gap rather than a code fix.

**Decision 2 — Cross-cutting primitives are built once in the shared kit, consumed everywhere.**
The mock already factors these out; the shipped code does not consistently reuse them. This change extracts:
- **A shared sparkline + hover-tooltip primitive** (mirroring the mock's `useTip()` + the `uptime-bar`/`spark-seg` markup): one interactive tooltip helper island pattern, consumed by the Status job-run sparkline AND the Usage 30-day KV sparkline (both currently missing hover detail).
- **A trash-icon `Remove` button** (mirroring `I.trash`-only buttons in the mock's `CorpusEditor`/`AlwaysImport`): replaces the current text "remove" `btn[data-variant=destructive]` in `src/admin/client/corpus.tsx`, applied to every corpus-editor row (feeds, flyer terms, aliases, always-import) — one component, not four copies.
- **A frontmatter-stripping markdown-render helper**: today the Guidance renderer does not strip the YAML frontmatter fence before rendering (leaving stray whitespace + a leading `<hr>`); the Recipe detail view already has this logic (per the `operator-data-explorer` spec's "Recipe body is provided frontmatter-stripped" requirement, computed via `parseMarkdown`). Reuse that same body-extraction path for Guidance instead of a second implementation, and reuse the Recipe detail's pretty frontmatter render for Guidance's extracted frontmatter block.
- **A list-value badge/pill renderer**: today `PrettyKV`-equivalent rendering exists for the R2 frontmatter but the D1 index row's list-valued fields (tags, dietary, etc.) render as raw text/JSON in the Recipe detail. Route the D1 row through the same pretty-array-as-chips renderer the frontmatter panel uses.
- **A job/candidate property pill**: Status's `job.stats` (today plain `jstat`/`jstat-k`/`jstat-v` text spans per the mock, but currently unstyled key:value pairs in the shipped code) render as Basecoat `badge` pills.

**Decision 3 — Spec deltas are scoped to genuinely behavioral changes only.**
Per the proposal, only four items get spec deltas (MODIFY on the already-folded `operator-admin`/`operator-data-explorer` requirements): Status's RSS-Feeds/Cached-SKUs tile navigation targets, Logs' sidebar removal, Recipes' configurable page size, and Discovery's match-score surfacing. Everything else (icons, borders, shadows, slider CSS, sparkline tooltips/axis labels, trash icons, markdown rendering, padding/width, badge/pill styling, the Members/Stores/Usage bug fixes once diagnosed as "config gap" or "guard bug" rather than a data-model change) is implementation/CSS/rendering fidelity against an already-approved design — captured as tasks, not new spec requirements, per the "don't invent spec requirements for CSS" instruction.

## Risks / Trade-offs

- [The Members-status and Stores-empty items may turn out to be genuine data-absence in every environment, not bugs] → Tasks explicitly ask the implementer to re-verify against the *operator's* live/remote environment (not just local dev) before closing them as "working as designed"; if the operator's remote D1 actually has rows/activity and the panel still shows empty, that is new information requiring re-diagnosis, not a closed non-issue.
- [Swallowed errors in `touchTenantActivity`'s `catch {}` could mask a real write failure indefinitely] → Task adds a structured (non-throwing) log/metric on that failure path so a future "why is this member still pending" investigation doesn't repeat this same blind read-only diagnosis.
- [`KV_NAMESPACE_LABELS` auto-defaulting from `wrangler.jsonc` at deploy time touches the deploy script, not just `src/`] → Scoped as an optional task; if it proves out of scope for this change's deploy-tooling boundary, the fallback is documenting the manual step more prominently rather than automating it.
- [Extracting shared primitives (tooltip, trash-button, frontmatter-strip, pill renderer) touches multiple areas at once] → Each primitive is added to the shared kit first with one call site migrated and verified, then rolled out to the remaining call sites — avoids a single big-bang refactor PR.
