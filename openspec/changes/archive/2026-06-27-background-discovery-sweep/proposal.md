## Why

Discovery today is **pull, synchronous, and frontier-model-in-chat**: the only time recipes are discovered is when a member runs `meal-plan`, which calls `fetch_rss_discoveries` + `read_discovery_inbox` and then has Claude-in-the-conversation triage → `parse_recipe` → classify → `create_recipe`, 1–2 at a time, with a human watching. That shape has four concrete costs:

1. **Discovery only happens at plan time.** A great recipe published Monday sits unseen until someone happens to plan a menu. There is no continuous intake — the freshest, most intentional signal in the system is gated behind a human starting a conversation.
2. **Every planner re-does the same triage.** The candidate pools are shared, but the *judgment* over them is redone — on the frontier model, in context — by each member who plans. Two members planning the same week both burn tokens triaging overlapping pools.
3. **The expensive step runs in the most expensive place.** Classifying a recipe into frontmatter (protein/cuisine/course/season/perishables/description) is the costly leg, and it runs on the frontier model inside a long planning context — when `env.AI` already does the sibling work (descriptions, embeddings) for a fraction of the cost, on a budget that doesn't compete with the conversation.
4. **A just-imported recipe isn't retrievable the same session** (its embedding reconciles a tick later), so the meal-plan flow carries a special-case "work from the parse you hold, don't re-search" wart — and the whole triage/parse/classify/import phase rides the planning hot path.

The opportunity is that **every primitive this needs already exists.** `env.AI` is bound and already generates descriptions + embeddings; the `scheduled()` handler already runs three capture jobs; `flyer-warm.ts` is a textbook tenant-enumerating, budget-bounded cursor sweep; `favoriteAffinity()` in `semantic-search.ts` is *already* "cosine a recipe against a member's taste"; and the corpus's vectors already sit in `recipe_derived`, ready to serve as a duplicate detector. This change is **composition, not new infrastructure**: it makes discovery the **fourth scheduled capture job**, relocating discovery's "capture" from the frontier model in-chat to a small model on-cron — exactly the capture→retrieve→narrow pattern the architecture already runs for the flyer and the recipe index.

## What Changes

- **A new background discovery sweep** — a fourth job in the existing `scheduled()` handler that polls the shared `feeds` and drains the email `discovery_candidates` inbox, runs each new candidate through a progressive-narrowing pipeline (cheap triage → classify+describe → taste-match → dedup), and **auto-imports the matches** into the shared corpus. Budget-bounded on both the external-subrequest axis (feed/page fetches, flyer-style cursor sweep) and the internal `env.AI` axis (classify/describe/embed, reconcile-style per-tick caps), under the one existing cron trigger.
- **Recipes are auto-imported when they match *any* member's taste** (the shared-corpus, opt-out model: a member who doesn't want one is a `toggle_reject` away). Matching is a **hybrid**: a cheap embedding-cosine recall filter (favorites + an embedded distillation of the member's `taste` text) followed by a single small-LLM "genuine fit?" confirm on survivors — so the matcher respects negation ("I hate cilantro") that bare cosine cannot, while keeping cost proportional to near-matches, not feed volume.
- **Per-member match attribution.** The sweep records which member(s) each import matched. This one piece of state is both the import gate *and* the per-member surfacing filter, so the shared corpus never floods any one member with the whole group's combined discovery.
- **Semantic deduplication at import.** Beyond the existing canonical-URL check, a candidate's description embedding is cosined against the corpus (`recipe_derived`) — `max cosine ≥ δ` ⇒ skip as a near-duplicate — plus an intra-sweep pass so two feeds surfacing the same dish in one tick collapse to one import. Same cosine machinery as the matcher, aimed at the corpus instead of at favorites.
- **A bloat governor** — a calibrated cosine threshold plus a per-window auto-import rate cap (excess deferred, never silently dropped) — because the autonomous matcher is now the *only* gate where the frontier model + human used to stand.
- **`meal-plan` reads, instead of discovers.** The discovery-first triage/import phase is removed; meal-plan gains a **new-for-me** read: recipes `discovered_at` after the member's last plan, that matched *their* taste, that they haven't already dispositioned or cooked. Because the sweep does the full capture, these are already classified and embedded — the "not retrievable this session" wart disappears.
- **An autonomous error surface.** A candidate that fails classification/validation is retried then parked in a `discovery_errors` table with a `read_discovery_errors` tool + ntfy push — the same out-of-band feedback the reconcile uses, because a cron has no conversation to surface a failure into.
- **An operator discovery-import log in the admin panel.** The sweep records an auditable per-candidate outcome (imported / skipped-duplicate / skipped-no-match / dietary-gated / parked-error, with attribution and detail), and the admin SPA gains a new **Logs** top-level area — a left submenu (first item: **Discovery**) with the entries on the right (the MCP-inspector master/detail shape), individual entries expandable into a detail dialog — served by an Access-gated `/admin/api/logs/discovery` read. So the operator can see what the autonomous pipeline did, since no member watches it run.
- **BREAKING (agent surface):** `fetch_rss_discoveries` and `read_discovery_inbox` are retired as agent tools (their logic becomes sweep internals); `reject_discovery` is reframed from pre-import candidate suppression to source/import suppression. The manual import path (`parse_recipe` + `create_recipe`, for "import this URL I'm handing you") and the config tools (`update_feeds`, `update_discovery_sources`) are unchanged.

## Capabilities

### New Capabilities
- `discovery-sweep`: the autonomous background discovery pipeline — intake (feed poll + email drain + the evaluated/seen set), the progressive-narrowing stages (cheap embedding triage, `env.AI` classification + description, the hybrid cosine+LLM taste matcher, semantic + intra-sweep dedup), auto-import with per-member match attribution, the governor (threshold + rate cap), the `env.AI`-derived per-member taste vector, and the `discovery_errors` surface. The fourth scheduled capture, modeled on `flyer-warm` (budget-bounded sweep) and the recipe-derived reconcile (`env.AI`, change-driven, health-tracked).

### Modified Capabilities
- `recipe-discovery`: the autonomous intake/dedup/import requirements move to `discovery-sweep`; `fetch_rss_discoveries` and `read_discovery_inbox` are **removed** from the agent tool surface; `reject_discovery` is **reframed** to suppress a source/import group-wide; `parse_recipe`/`create_recipe`/`update_feeds` (the manual-import + config surface) are retained and clarified as the non-discovery path.
- `menu-generation`: the **discovery-first** requirement (meal-plan triages and imports discoveries before retrieval) is **replaced** by a **new-for-me read** requirement (meal-plan reads recently-imported, taste-matched, undispositioned recipes and folds them into retrieval) — the triage/parse/classify/import obligations leave the flow.
- `newsletter-discovery`: the email `discovery_candidates` inbox becomes the sweep's push intake rather than an agent-read inbox; the `email()` handler intake itself is unchanged, but its consumer moves from `read_discovery_inbox` to the sweep.
- `operator-admin`: a new **Logs** top-level area (left submenu, master/detail entries, detail dialogs) whose first source is the discovery-import log, plus an Access-gated `GET /admin/api/logs/discovery` read; the top-level-areas requirement grows from three areas (Status/Members/Dev) to four.

## Impact

- **Worker (`src/`):** a new `discovery-sweep.ts` (the sweep core over injected deps, mirroring `flyer-warm.ts`'s testable shape) wired as a fourth job in `scheduled()` (`src/index.ts`); a classifier helper beside `description.ts` (`env.AI` instruct call producing validated frontmatter, structured-error mapped); a per-member taste-vector derivation (hash-gated like `recipe_derived`); reuse of `buildNewRecipe`/`validateFile`/`seedRecipeDescription` (`discovery.ts`/`validate.ts`) for the write, `favoriteAffinity`/`cosineSimilarity` (`semantic-search.ts`/`embedding.ts`) for matching + dedup, and `recipeSourceMap` (`recipe-index.ts`) for URL dedup; retirement of the `fetch_rss_discoveries`/`read_discovery_inbox` registrations in `discovery-tools.ts`; a `list_new_for_me` + `read_discovery_errors` tool.
- **D1 (migrations):** promote `discovered_at` from the `recipes.extra` JSON to a queryable column + index; new tables for match attribution (`(recipe, tenant)`), the evaluated/seen set (`discovery_evaluated`), the per-member taste vector, the per-candidate sweep **log** (the admin Discovery view's data source; `discovery_evaluated`/`discovery_errors` may be facets of it), and `discovery_errors`; a per-tenant `last_planned_at` (a `profile` column). All reconcile-/sweep-owned; the projection's wholesale `recipes` rebuild must not clobber them (the same sibling-table discipline as `recipe_derived`).
- **Admin SPA (`admin/`, Elm):** a new **Logs** routed area (`/admin/logs`, `/admin/logs/discovery`) with a left submenu, a `RemoteData` master/detail list, and a detail dialog (modeled per `admin/CLAUDE.md`'s impossible-states standard); a `GET /admin/api/logs/discovery` handler in `src/admin.ts` reading the sweep log; the committed `admin/dist/` bundle rebuilt via `aubr build:admin`.
- **Agent persona (`AGENT_INSTRUCTIONS.md` → `plugin/`):** `meal-plan` step 2 (triage + import) collapses to the new-for-me read; the `grocery-discovery` skill's triage/disposition section retires (its parse/classify mechanics survive only for the manual `import-recipe` flow).
- **Docs (lockstep):** `docs/ARCHITECTURE.md` (the determinism boundary — capture relocates to the cron; a new §discovery-sweep beside the other crons; "three crons" → "four"; the discovery + disposition section rewritten), `docs/TOOLS.md` (tools retired/added), `docs/SCHEMAS.md` (the new tables/columns).
- **Quality risk owned here:** the unattended `env.AI` **classifier** is the load-bearing unknown (frontier-model judgment → small model, no human to catch a bad call) — gated by a Phase-0 spike with an eval set, exactly like the description-gen spike. The cosine thresholds (taste τ, dedup δ) and the rate cap are calibrated in the same spike. See `design.md`.
- **Relationship to other changes:** builds directly on `ai-derived-recipe-metadata` (the `recipe_derived` table + `env.AI` describe/embed loop) and `semantic-meal-plan`/`unify-recipe-search` (the cosine + `favoriteAffinity` machinery); independent of the R2 corpus move.
