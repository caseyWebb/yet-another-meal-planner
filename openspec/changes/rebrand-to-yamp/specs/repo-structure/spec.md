## MODIFIED Requirements

### Requirement: Code repository layout

The code repository (`caseyWebb/yet-another-meal-planner`) SHALL be an **`aube` workspace monorepo** (workspaces declared in `pnpm-workspace.yaml`, packages cross-depending via the `workspace:*` protocol) whose packages are the **Worker** (the Cloudflare Worker — MCP server, member `/api`, and the Access-gated `/admin` routes — with its build tooling `scripts/`, D1 migrations `migrations/d1/`, and Worker tests), a shared **contract** package, the home-network **satellite** package, the **member app** (`packages/app`, the React SPA served at `/`), the **admin app** (`packages/admin-app`, the operator panel SPA served at `/admin`), and the shared **UI** package (`packages/ui`, the shadcn/ui primitives and theme tokens both apps consume). The shared **contract** package SHALL be **workerd-pure** (no Node-only dependencies) because it is imported by BOTH the Worker (workerd runtime) and the satellite (Node runtime); it SHALL hold the recipe-parse spine (JSON-LD extraction + normalize) and the ingest wire types/validators, so the parse and the payload shape can never drift between the two runtimes. The **satellite** package (Node + a headless-browser dependency) SHALL be isolated in its own workspace so its heavy dependencies never enter the Worker build or deploy. The reference docs SHALL remain under `docs/`, the OpenSpec workspace under `openspec/`, and the agent persona source `AGENT_INSTRUCTIONS.md` at a defined location. Reusable CI workflows that operator data repos call SHALL live under `.github/workflows/`. The code repository SHALL NOT commit a generated plugin bundle, a `.claude-plugin/marketplace.json`, or any built static-assets bundle (the Worker's merged assets root — both SPAs' output — is a gitignored build artifact); those are generated by CI/the operator's deploy, not committed.

#### Scenario: The repository is a workspaces monorepo

- **WHEN** the code repository is cloned and listed
- **THEN** it resolves as a workspaces monorepo with a Worker package, a shared workerd-pure contract package, a satellite package, the member-app, admin-app, and shared-UI packages, alongside `docs/`, `openspec/`, and `AGENT_INSTRUCTIONS.md`, and no committed plugin bundle, `.claude-plugin/marketplace.json`, or built assets bundle

#### Scenario: The shared contract package is workerd-pure

- **WHEN** the contract package is built
- **THEN** it carries no Node-only dependency, so both the Worker (workerd) and the satellite (Node) import the same recipe-parse and ingest wire types

#### Scenario: The satellite's heavy deps stay out of the Worker

- **WHEN** the Worker is built or deployed
- **THEN** the satellite package's browser/Node dependencies are not included — they are isolated to the satellite workspace

### Requirement: Canonical docs are committed at defined locations

The repository SHALL commit the canonical project docs. `AGENT_INSTRUCTIONS.md` SHALL reside at the repository root as the canonical yamp persona source from which `plugin/` is generated. `CLAUDE.md` SHALL reside at the repository root as Claude Code development guidance for working in this repo, and SHALL point to `AGENT_INSTRUCTIONS.md` for the agent persona and conversational flows. The contributor guide `CONTRIBUTING.md` SHALL reside at the repository root; the reference docs `ARCHITECTURE.md`, `SCHEMAS.md`, and `TOOLS.md` SHALL reside under a `docs/` directory. Any references to the `docs/` reference docs SHALL resolve to their `docs/` paths.

#### Scenario: Root docs are present with their distinct roles

- **WHEN** Claude Code opens the repository directory
- **THEN** it finds `CLAUDE.md` at the repository root and reads it as repo-development context, and `AGENT_INSTRUCTIONS.md` at the root as the yamp persona source

#### Scenario: CLAUDE.md points to the agent instructions

- **WHEN** a reader opens `CLAUDE.md` looking for the agent persona or conversational flows
- **THEN** it directs them to `AGENT_INSTRUCTIONS.md` rather than containing that prose itself

#### Scenario: Canonical docs live at their defined locations

- **WHEN** a developer looks for the architecture, contributor, schema, and tool references
- **THEN** `CONTRIBUTING.md` is present at the repository root, and `docs/ARCHITECTURE.md`, `docs/SCHEMAS.md`, and `docs/TOOLS.md` are present under `docs/`

#### Scenario: Doc references resolve

- **WHEN** a reader follows a root doc's pointer to the tool inventory (`docs/TOOLS.md`)
- **THEN** the referenced file exists at that path

### Requirement: README and gitignore

The repository SHALL include a `README.md` explaining the project and how to use the repo, and a `.gitignore` covering Node artifacts, OS files, editor files, Cloudflare Worker secret files, and the Cloudflare **`.wrangler/`** local-state directory (which caches the account id and the account name). The operator data repository **and its template** SHALL likewise ignore `.wrangler/`, so local `wrangler` use never commits the account cache into a repository that may be made public.

#### Scenario: README explains the project

- **WHEN** a developer opens `README.md`
- **THEN** it explains what yamp is and how to use the repository

#### Scenario: gitignore excludes secrets and noise

- **WHEN** Worker secret files, `node_modules/`, OS files (e.g. `.DS_Store`), editor files, and a `.wrangler/` directory are present in the working tree
- **THEN** `.gitignore` prevents them from being committed, in the code repository, the operator data repository, and the data-repo template

### Requirement: Data template tracked as an independent repository

The public data-repo template (`caseyWebb/yet-another-meal-planner-deployment-template`) SHALL be tracked as its own independent repository, NOT vendored into this code repo. This repository SHALL NOT carry a git submodule (or `.gitmodules` entry) for the template, and SHALL NOT keep hand-maintained copies of the data-repo caller workflows under `docs/`. The template repo's own `.github/workflows/` is the single canonical reference for the thin data-repo callers.

The explanatory content describing the data-repo callers — the mapping of each data-repo caller workflow to this repo's reusable workflow, and the rationale for running them in the data repo — SHALL reside in `docs/SELF_HOSTING.md`, pointing at the [`yet-another-meal-planner-deployment-template`](https://github.com/caseyWebb/yet-another-meal-planner-deployment-template) repo as the canonical example.

#### Scenario: No template submodule is present

- **WHEN** the repository is inspected for submodules
- **THEN** no `.gitmodules` file exists and no `docs/data-template/` submodule is tracked

#### Scenario: No duplicated caller-workflow copies remain

- **WHEN** the repository is searched for hand-maintained data-repo workflow copies
- **THEN** `docs/data-repo-workflows/` does not exist, and the caller→reusable-workflow mapping and rationale are found in `docs/SELF_HOSTING.md`

#### Scenario: CI does not depend on the template

- **WHEN** `ci.yml` runs
- **THEN** typecheck and the test suites pass without fetching the template, because no built or tested source imports it
