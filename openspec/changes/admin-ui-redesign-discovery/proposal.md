## Why

`/admin/discovery` today is a routed placeholder (`admin-ui-redesign-foundation`) — the nav slot exists, the body says "coming soon" and points the operator at `/admin/logs/discovery` for the per-candidate log. That log is a flat outcome list (id/url/title/source/outcome/slug/detail), useful for an audit trail but silent on the question an operator actually has about a stuck or interesting candidate: *how far did this get, and exactly where and why did it stop?* The discovery sweep (`discovery-sweep`) is a seven-step pipeline (triage → acquire → classify → describe → dedup → match → import) with a rich, structured halt taxonomy (content park vs. infra failure vs. taste/dietary/dedup rejection vs. rate-cap deferral) and a retry lifecycle (bounded backoff, attempt cap, a manual operator override) — none of which the flat list visualizes. This change fills the Discovery area with the candidate-pipeline view the foundation change reserved space for: a stage-by-stage progression track per candidate, a plain-language halt summary, and the manual retry affordance, replacing the placeholder body.

## What Changes

- Replace the `/admin/discovery` placeholder body with the **candidate-pipeline view**: page-level stat tiles (Candidates / Imported + import rate / Parked-failed / In-retry-queue), a filter-pill row (All / Imported / Retrying / Parked / Failed / No-match / Duplicate / Dietary / Deferred) with per-pill counts, and a paginated list of per-candidate cards. Each card renders a 7-stage progression track (the real pipeline: triage → acquire → classify → describe → dedup → match → import) showing the candidate's furthest stage reached and its halt point colored by outcome, a one-line plain-language summary of where/why it stands, and — for a retryable (parked `error` or `failed`) candidate — a **Retry now** action plus an attempts/auto-retry-clock readout. Expanding a card reveals a per-stage detail (passed / stopped here / not reached, with a blurb) and the raw `discovery_log` row via `PrettyKV`.
- **Migrate the candidate log to live at `/admin/discovery`:** this area becomes the single home for the per-candidate discovery view. `/admin/logs/discovery` is **retired as a destination** (redirects to `/admin/discovery`); its existing Retry/Delete mutations and the `client/logs.tsx` island that hosts them move to the Discovery area (a new `client/discovery.tsx` island, or a renamed/relocated one — implementation's call in design.md) — they are genuine row mutations, so the area keeps an island for them, while the page's stat tiles, filter pills, pipeline track, and stage detail stay SSR per `admin/CLAUDE.md` rule 8.
- **Repoint the Logs all-jobs link:** the Logs area's all-jobs run log (`admin-ui-redesign-logs`, in flight) renders a `discovery-sweep` run entry's expanded detail with a "View discovery candidates →" link; that link is repointed from `/admin/logs/discovery` to `/admin/discovery`.
- **Add a backend read** that exposes, per candidate, what the flat log row does not: the furthest pipeline stage reached, the halt stage + outcome-colored kind, and (for retryable rows) the next-retry countdown — derived from the existing `discovery_log` columns (`outcome`, `detail`, `attempts`, `next_retry_at`) via a halt-stage mapping table, not a schema change.
- Stat tiles, filter counts, and pagination are computed over this same read; the manual retry action reuses the existing `processCandidate(..., { bypassCap: true })` path unchanged (`POST /admin/api/discovery/:id/retry`), and delete reuses the existing rejection-and-remove path (`DELETE /admin/api/discovery/:id`) unchanged.

## Impact

- `src/admin/app.tsx` — the `/discovery` route body is replaced with the pipeline view's SSR composition; `/logs/discovery` becomes a redirect to `/discovery`; the discovery row-action API routes (`/api/discovery/:id/retry`, `DELETE /api/discovery/:id`) are unchanged in contract, only their UI caller moves.
- `src/discovery-db.ts` — a new reader (e.g. `readDiscoveryCandidates` or an enrichment over `readDiscoveryLog`) that maps each row's `outcome`/`detail.stage`/`detail.reason` to a halt stage index, exposing furthest-stage + halt + retry-clock fields the view needs; no migration (existing columns only).
- `src/admin/pages/discovery.tsx` (new) — the SSR page: stat tiles, filter pills, the pipeline-track + card list, pagination.
- `src/admin/client/discovery.tsx` (new, or relocated from `client/logs.tsx`) — the Retry/Delete island, reusing the existing typed routes.
- `src/admin/pages/logs.tsx` / `src/admin/client/logs.tsx` — lose the Discovery-specific submenu destination and its island usage; the all-jobs run log's discovery-sweep entry link is repointed to `/admin/discovery`.
- `src/admin/ui/kit.tsx` — likely a small new presentational primitive for the stage-progression track (or composed from existing `Item`/`Badge`/icons — implementation's call), kept generic enough to live in the shared kit per the foundation's intent.
- `docs/TOOLS.md` — no change (no MCP tool surface touched). `docs/SCHEMAS.md` — no change (no D1 schema change). `docs/ARCHITECTURE.md` — no change (no architectural shift; same `discovery_log` table and `processCandidate` pipeline).
- Depends on `admin-ui-redesign-foundation` (archived — the placeholder route + kit primitives this fills in) and coordinates with `admin-ui-redesign-logs` (in flight — the link repoint and the Discovery-destination retirement touch its just-landed Logs work; this proposal documents the reconciliation explicitly so the two changes don't talk past each other).
