# Proposal — clean-discovery-import-titles

## Why

Issue #219: the discovery sweep imported Bon Appétit's "A Better Beer Can Chicken" **verbatim** — title `A Better Beer Can Chicken`, slug `a-better-beer-can-chicken`. The corpus naming contract already exists: `recipe-import/spec.md` ("Clean titles and globally-unique slugs") requires SEO suffixes, marketing qualifiers, and editorial framing stripped, with the slug derived from the clean dish name. But that contract was only ever applied by the one-time ReciMe naming pass and the in-chat agent import (where Claude does the cleaning conversationally). The **unattended** discovery leg never got it: `discovery-sweep.ts` threads the raw feed/page title straight through (`acquireContent` → `classifyRecipe`, where `title` is deliberately excluded from `CLASSIFIED_FIELDS` and pipeline-set verbatim), and `buildNewRecipe` derives the slug mechanically from that raw title. The production spike confirms the systemic result: of 92 discovery-origin recipes, **~25 carry flowery titles** ("Super Soft and Tender Lemon Yogurt Loaf", "Our Go-To Side Salad", "Homemade BBQ Sauce Recipe", "Summer Dinner Recipe: Fresh Strawberry Chicken Salad", …) — and their slugs are permanently fossilized marketing copy.

Two convergence problems, per the repo's pipeline-not-surgery rule:

1. **Import-time**: new discovery imports must be named to the existing cleaning contract before slug derivation.
2. **Existing data**: the ~25 flowery titles in production must converge organically through a shipped pipeline pass — never a hand-edit — with `a-better-beer-can-chicken` as the observed-defect acceptance fixture. Existing **slugs stay stable**: a slug is the recipe's id (the R2 object path and the join key for favorites, cooking logs, meal plans, notes, discovery attributions), so renaming one breaks references. Titles converge; slugs do not.

## What Changes

- **Title cleaning rides the sweep's existing classify call.** `discovery-classify.ts` gains `title` as a model output: the system prompt instructs the model to also emit the clean dish name (strip SEO suffixes, marketing qualifiers, editorial framing; preserve foreign dish names; keep identity-bearing words like "Vegan"/"Slow Cooker" and informative parenthetical glosses), anchored by few-shot exemplars. The pipeline swaps the cleaned title in behind a **deterministic word-subset guard** (the cleaned title may only remove words from the raw title, compared case/punctuation-insensitively) and **fails open** to the raw title — cleaning can never invent a new dish identity and can never park an import. Zero marginal `env.AI` calls: it is one more key on a call the import already makes. The facet-derivation consumers of the same classifier ignore the new key (`title` is NOT added to `DERIVED_FACET_FIELDS`), so authored titles are never overridden by the facet cron.
- **Slugs derive from the dish name, not the gloss.** `buildNewRecipe` derives the slug from the cleaned title with any parenthetical gloss stripped (`"Jatjuk (Pine Nut Porridge)"` → `jatjuk`), matching the ReciMe precedent (`sundubu-jjigae`). Applies to both the sweep and `create_recipe` (one shared naming funnel).
- **Cleaned-title slug collisions are disambiguated, not parked.** Cleaning increases collision odds ("The Best Strawberry Icebox Cake" cleans onto an existing `strawberry-icebox-cake`). The sweep already semantically dedups same-dish candidates; a residual collision is a same-name-different-dish, so the sweep's import leg resolves `slug_exists` with a bounded deterministic numeric suffix (`-2`, `-3`, …) instead of parking. `create_recipe` keeps its `slug_exists` structured error (a conversational agent should decide, not auto-suffix).
- **A bounded, one-shot-stamped title re-audit converges the existing corpus** (the `ingredient-normalize`/`audited_at` pattern): a new scheduled pass reads a bounded batch of un-audited recipes per tick, runs the same guarded title-clean judgment, rewrites only the R2 file's frontmatter `title` when it differs, and stamps a new `title_audit` D1 row (slug PK, `audited_at`, kept/cleaned outcome, before/after). New imports are **born-stamped**, so the backlog is exactly the pre-existing corpus (205 rows) and the pass quiesces to a ~0-LLM no-op once drained (~2 hours at 10/tick on the 5-minute cron). Slug/file path are never renamed. A rewritten title flows downstream organically: the projection re-indexes it next phase, and the recipe-derived `content_hash` gate (title is in its domain) regenerates the description + embedding.
- **Acceptance fixture**: after deploy, `a-better-beer-can-chicken`'s title converges to "Beer Can Chicken" through the re-audit (slug unchanged), and a fresh flowery discovery import lands with a clean title AND clean slug (`beer-can-chicken`-shaped). The full expected-rewrite list (~25 rows) is enumerated in design.md from the production spike.

## Capabilities

### New Capabilities

- **`recipe-title-audit`** — the corpus title-hygiene convergence pass: a bounded, one-shot-stamped scheduled re-audit that converges existing recipe titles to the naming contract through the pipeline (guarded LLM title-clean, R2 frontmatter rewrite, `title_audit` stamp, born-stamped new writes, job-health record), with slug immutability as an explicit requirement.

### Modified Capabilities

- **`discovery-sweep`** — gains the import-time naming requirement: the unattended import leg SHALL clean the candidate title (riding the existing classify call, subset-guarded, fail-open) before slug derivation, derive the slug from the cleaned title's dish name (parenthetical gloss excluded), and resolve residual cleaned-title slug collisions deterministically rather than parking. This extends the `recipe-import` "Clean titles and globally-unique slugs" contract to the discovery path; the `recipe-import` spec itself is unchanged.

## Impact

- **Worker (`packages/worker/src/`)**: `discovery-classify.ts` (prompt + exemplars + guarded title swap in `toFrontmatter`), `discovery.ts` (slug basis = parenthetical-stripped title), `discovery-sweep.ts` (collision suffix in `importRecipe`; born-stamp on import), `discovery-tools.ts` (born-stamp in `create_recipe`), new `title-audit.ts` (the re-audit job, logic/deps split for unit tests), `index.ts` (`scheduled()` phase-1 wiring), `health.ts` (`HEALTH_JOBS` + "title-audit" — the admin Jobs/Health views pick it up generically from the registry; **no admin-app/ui/Playwright surface changes**).
- **One D1 migration**: `migrations/d1/0042_title_audit.sql` (next available) — the `title_audit` stamp table. All access via `src/db.ts` helpers (throw-free `storage_error` discipline).
- **No tool contract changes**: no new/changed MCP tool params or returns. `create_recipe`'s described behavior gains one nuance (slug derives from the dish name before any parenthetical) — its tool description and `docs/TOOLS.md` note it, same pass.
- **Docs (lockstep)**: `docs/SCHEMAS.md` (the `title_audit` table), `docs/ARCHITECTURE.md` (the title-audit pass in the cron-job list), `docs/TOOLS.md` (`create_recipe` slug nuance). No persona change (`AGENT_INSTRUCTIONS.md` untouched — the agent-facing import flow already owns conversational title cleaning; nothing new to instruct).
- **No member/admin app changes**; no `wrangler.jsonc` changes (no new binding, no new route); deploy-merge allowlist untouched.
- **Production data**: converges organically post-deploy — ~25 title rewrites, 0 slug renames, verified against the fixture rows (design.md).
