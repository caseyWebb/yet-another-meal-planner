## Requirements
### Requirement: Derived recipe fields are produced by the Worker into D1

A recipe field that is **purely derived** from the recipe's human-authored content — reproducible by re-running its generator, with no human as its author or corrector of record — SHALL be produced by the Worker and stored in D1, and SHALL NOT be authored into recipe frontmatter. The recipe `description` SHALL be such a field: it is generated by the Worker from the recipe's human-authored content (using the Worker's AI binding, `env.AI`) and stored in D1, not in frontmatter. The placement rule SHALL be: a field lives in frontmatter **iff** a human is its author or corrector of record; a purely-derived, freely-regenerable field lives in D1.

#### Scenario: The description is generated, not authored

- **WHEN** a recipe is imported or its human-authored content changes
- **THEN** the Worker generates the `description` from that content and stores it in D1, and no `description` is required or written in the recipe's frontmatter

#### Scenario: A purely-derived field is not placed in frontmatter

- **WHEN** a field is purely derived from recipe content and has no human corrector of record
- **THEN** it is produced into D1 by the Worker and is absent from the authored frontmatter contract

### Requirement: Derived fields regenerate on a content-change gate

The Worker SHALL detect changes to a recipe's human-authored content via a content hash and SHALL regenerate the recipe's derived fields when that hash changes, on the scheduled reconcile. The content hash SHALL cover only the authored frontmatter facets the description is derived from (title, ingredients_key, course, protein, cuisine, time_total, dietary, season — not the recipe body; the reconcile stays pure D1 + AI and does not read the body) and SHALL NOT include any derived field, so that regenerating a derived field cannot itself trigger further regeneration. A steady corpus (no content changes) SHALL perform no regeneration work.

#### Scenario: Editing recipe content regenerates the description

- **WHEN** a recipe's body or authored frontmatter changes
- **THEN** the content hash changes, the reconcile regenerates the `description`, and (because the description changed) the recipe is re-embedded on the same or a following tick

#### Scenario: A steady corpus does no work

- **WHEN** the reconcile runs and no recipe's human-authored content has changed
- **THEN** no description is regenerated and no embedding is recomputed

#### Scenario: A derived-field change does not loop

- **WHEN** the reconcile writes a regenerated `description`
- **THEN** the content hash (which excludes derived fields) is unchanged by that write, so the next tick does not regenerate it again

### Requirement: The derived description is co-located with its embedding

The recipe `description` and its embedding SHALL be stored together as the two derived halves of one artifact — same producer (the reconcile), same cadence, same `slug` key — in a reconcile-owned D1 table that the recipe-index build SHALL NOT overwrite. The index build's wholesale rebuild of the `recipes` table SHALL NOT clobber the derived description or embedding.

#### Scenario: A recipe-index rebuild preserves the derived fields

- **WHEN** the build rebuilds the `recipes` table (wholesale delete + re-insert)
- **THEN** the reconcile-owned derived table (description + embedding) is untouched, and the derived fields survive the rebuild

### Requirement: Reads merge the derived description and tolerate its absence

Recipe read tools SHALL merge the D1-resident `description` into the recipe they return, alongside the existing read-time merges (overlay, cooking-log `last_cooked`). A recipe whose description has not yet been generated (e.g. just imported, before the next reconcile tick) SHALL read with an empty/absent description treated as "not yet derived" — never an error.

#### Scenario: read_recipe returns the derived description

- **WHEN** `read_recipe` is called for a recipe whose description has been generated
- **THEN** the returned recipe carries the D1 description, merged at read time

#### Scenario: An undescribed recipe reads cleanly

- **WHEN** `read_recipe` is called for a just-imported recipe before its description is generated
- **THEN** the recipe is returned with an empty/absent description and no error, the same tolerance the embedding index applies to an unembedded recipe
