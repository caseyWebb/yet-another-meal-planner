- **Claude.ai** is the conversational surface and the reasoning. Each chat starts fresh; state lives in D1 (profile + session state + cooking log + recipe index + attributed records) and the R2 corpus (authored recipes + guidance markdown), not in chat history. The agent reads what it needs through MCP tools at the start of a conversation.
- **The Worker** (this repo, root `src/`) is a Cloudflare Worker hosting the `grocery-mcp` MCP server тАФ the domain tool surface (pantry, recipes, Kroger, cart) тАФ plus an OAuth 2.1 provider members connect their Claude.ai to. It is the locus of determinism and the multi-tenant gate.
- **The R2 corpus bucket** (`CORPUS`, bound to the Worker, operator-owned) holds the human-authored markdown: `recipes/*.md` and the `guidance/**/*.md` umbrella (`ingredient_storage/` + `cooking_techniques/` + `purchasing/`), read/written through `src/corpus-store.ts` (`createR2CorpusStore`: getFile/listDir/list/put/delete) тАФ there is no GitHub App or GitHub API on the data path. Authors edit an Obsidian vault synced to the same bucket over its S3-compatible API. Everything else тАФ profile, session state, shared corpus, cooking log, recipe index, attributed records тАФ is in D1. See [`SELF_HOSTING.md`](SELF_HOSTING.md).

There is no database, no scheduler, no CLI, and no stateful agent runtime тАФ the `agents` SDK is present only for its stateless `createMcpHandler` MCP transport (no Durable Objects, no Workflows). Everything else is glue.

## The determinism boundary

The whole design turns on putting the LLM only where genuinely-fuzzy judgment is needed and keeping everything else as plain deterministic code inside the Worker's tools.

**Three places the LLM earns its keep:**

1. **Message understanding and tool orchestration.** Claude reads the request, decides which tools to call in what order, asks follow-ups, interprets freeform constraints ("comfort food one night," "I'm feeling lazy"). This is Claude's native strength тАФ there is no custom routing logic.
2. **Menu-generation reasoning.** Given assembled context (pantry, flyer, candidates, ready-to-eat, preferences), Claude proposes a plan honoring multiple soft and hard rules. The genuinely-fuzzy step.
3. **Fuzzy-matching fallback** inside the Worker's Kroger pipeline, only when deterministic narrowing leaves ambiguity, plus the **discovery sweep's classify + taste-match** тАФ which run on a small Workers AI model on the cron, not the frontier model in chat (see *the discovery sweep* below).

**The discovery LLM lives on the cron, not the hot path.** Unprompted discovery's "capture" тАФ classifying a new candidate into the controlled-vocab facets and judging taste fit per member тАФ runs on the **cron** via a small classifier (`@cf/mistralai/mistral-small-3.1-24b-instruct`) plus the embedding cosine, not in the frontier chat model. It is *capture тЖТ retrieve тЖТ narrow* run on a schedule: the sweep captures (classify + describe + embed + match) into the corpus and the `discovery_matches`/`discovery_log` tables; the agent retrieves the result deterministically (`list_new_for_me`) and narrows it into the menu. The frontier model in chat reasons only over what the sweep has already imported; it does not classify unprompted discoveries itself.

**Everything else is deterministic code with no LLM in the loop:** corpus (R2) reads/writes, frontmatter parsing, recipe filtering and scoring, Kroger API calls, RSS/JSON-LD parsing, cart writes, index projection, validation, the discovery sweep's dedup/match cosine arithmetic.

The MCP tool boundary is where this is enforced. Tools are **coarse and opinionated** тАФ they wrap multi-step pipelines so the LLM can't bypass them. Raw building blocks (`kroger_raw_search`, `corpus_raw_write`, `cart_add_by_name`) are deliberately **not** exposed, because they would let the LLM skip the cache, the validation, or the SKU-matching pipeline. See [`TOOLS.md`](TOOLS.md) for the design philosophy and the full inventory.

**The deeper pattern тАФ capture тЖТ retrieve тЖТ narrow.** Where the system needs LLM-derived *knowledge* (not just orchestration), it does not re-derive it on every read; it **captures** the model's judgment once into persistent data, **retrieves** it deterministically, and lets the LLM **narrow** with live context. Recipe import already works this way: Claude classifies `protein` / `cuisine` / `perishable_ingredients` / `course` once at import (capture), the scheduled projection (`src/recipe-projection.ts`) reads the R2 corpus and projects them into the D1 `recipes` table and `search_recipes` filters them (retrieve), and menu-gen reasons over the filtered set (narrow). The LLM sits at the two ends тАФ the one-time capture and the contextual narrowing тАФ while determinism owns the middle (identity, indexing, lookup) and the gates (validation). Caching beats re-reasoning at scale because the hot path stays deterministic, reserving the model for genuine novelty and genuine judgment тАФ which is also what keeps the agent viable on a smaller, faster model. **The same pattern also runs on a schedule:** a cron-driven sweep warms a per-store Kroger *flyer* into KV (capture), `kroger_flyer` reads that rollup (retrieve), and menu-gen reasons over it (narrow) тАФ keeping the multi-second sale fan-out off the agent's hot path and under the free-tier per-request subrequest cap (a synchronous tool call has one invocation's budget; a background sweep has unlimited invocations over time). The **discovery sweep** is the same shape for *recipe* discovery: a cron classifies + taste-matches + imports new candidates (capture), `list_new_for_me` reads each member's fresh imports (retrieve), and menu-gen folds them into the plan (narrow). See *the flyer warm* and *the discovery sweep* below.

**Direction тАФ thin tools, recipe-side retrieval, LLM reasoning.** The scaling pressure is on the recipe side, and it's already solved there: recipes are first-class entities with captured metadata and an index, and you never reason over the whole corpus at once тАФ `search_recipes` (membership filtering plus, in ranked mode, semantic retrieval) with the makeability gate and each member's **rejects** narrows it to a small candidate set that loads into context. So the system leans into LLM reasoning at the *narrow* end (substitution, freshness, pantry matching, the to-buy list тАФ all read-time over the loaded pantry + chosen recipes) and keeps deterministic *retrieval* only over what's unloadable (the Kroger catalog) or large (the corpus index). Tools shrink тАФ `substitutions.toml`/`propose_substitutions` and `verify_pantry` (with its recipe-ingredient parser) fall out тАФ and the reasoning moves into the skills. The realized lever is **recipe faceting**: an open-vocabulary `course` field (`main | side | dessert | breakfast | тАж`, classified at import, shape-validated only тАФ no controlled set) so one faceted `search_recipes` call returns the available mains+sides with metadata and `meal-plan` reasons holistically over the whole plate (menu + sides + expiry-matching + pantry subs) before cost/confirm. Sides are **two-tier**: corpus sides (`course: side` recipes, remembered in `pairs_with`) and **open-world** sides (trivial preparations with no recipe file) that ride on the main's `meal_plan` row and flow to the cart by world-knowledge ingredient enumeration. Whether a main needs a side is inferred in that same pass. Ingredients stay strings; the `aliases` table stays as the matcher's small normalization table. A self-growing *ingredient knowledge graph* was considered and deferred (the feature that justified it тАФ expiry-driven cooking тАФ is read-time-solvable). See [`adr/0001-determinism-boundary-capture-retrieve-narrow.md`](adr/0001-determinism-boundary-capture-retrieve-narrow.md) for the decision, the locked choices, the deferred graph, and the rollout (the `thin-pantry-and-substitution-path` change is Phase 0).

## Multi-tenant identity

One self-hosted Worker serves a small friend group; each member connects their own Claude.ai. The code is a separate upstream self-hosters deploy without forking. The R2 corpus bucket (`CORPUS`) holds the authored corpus (`recipes/*.md`, `guidance/**/*.md`); all operational data (profile, session state, cooking log, shared corpus, recipe index) lives in D1, tenant-isolated by the `tenant` column.

- **OAuth 2.1 provider.** Claude.ai custom connectors authenticate via OAuth, so the Worker hosts an OAuth provider (`@cloudflare/workers-oauth-provider`, KV-backed тАФ no SQL). `src/index.ts` constructs the provider; `src/authorize.ts` renders the invite-code consent page.
- **Identity is an operator-issued invite code** against a curated allowlist тАФ members need no GitHub account. The issued access token's grant carries the member's `tenantId`.
- **"Which tenant" is a D1 column.** Per-tenant data is isolated by the `tenant` column on every D1 table. Each request resolves token тЖТ tenant *before* any tool runs, so no tool can reach another member's data.
- **The authored corpus is the R2 `CORPUS` bucket**, bound to the Worker and read/written through `src/corpus-store.ts` тАФ no credential, GitHub App, or installation token on the data path. The bucket is shared by the friend group (the corpus is single-source, read by all); per-tenant data never lands in R2.
- **Kroger split:** `client_credentials` product/price reads are shared at the app level; cart writes use a **per-tenant** `authorization_code` refresh token (`kroger:refresh:<tenant>`). The only isolate-cached Kroger state is that app-level access token тАФ it carries no tenant context, so sharing it across the requests an isolate serves is sound. The resolved `locationId` is per-tenant store context and is **not** isolate-cached (a cached location would leak one tenant's store to another, since isolates are reused across tenants); `resolveLocationId` resolves per call, and the tool path memoizes it per request instead. The product/price client bounds its own concurrent in-flight requests to a small fixed cap (default 6), so fan-out callers (`kroger_prices`, the background flyer warm, etc.) use plain `Promise.all` тАФ the cap prevents 429 storms without callers needing a concurrency primitive.
- **Shared flyer cache (the one deliberately cross-tenant data plane).** The warmed Kroger flyer is keyed by `locationId` (`flyer:{locationId}` in `KROGER_KV`), so tenants at the **same store share one rollup** and tenants at **different Krogers get independent ones**. This is the single shared data-plane cache; everything else is strictly per-tenant. It is sound because store-wide sale prices are **public-derived, not tenant-private** тАФ no member's state leaks. The cron sweep that fills it (`src/flyer-warm.ts`) runs *without* an OAuth session, so it enumerates the tenant directory and reads each tenant's D1 `profile` row for its `preferred_location`.

A **solo operator** is simply the degenerate case: one tenant.

### Operator admin surface

Member lifecycle тАФ **onboard / revoke / rotate-invite / list** тАФ is an **in-Worker** operator surface at `/admin`, not a GitHub Action. A **Hono** app (`src/admin/`, mounted under the Worker's `fetch` тАФ no second Worker) server-renders each page by calling these `src/` operations directly and hydrates the interactive bits as **islands** that call a same-origin, typed `/admin/api/*` JSON surface тАФ **both** paths writing the `TENANT_KV` allowlist + `invite:*` codes and purging per-tenant D1 through `src/db.ts` (one source of truth per operation). The islands + stylesheet are built by `scripts/build-admin.mjs` into `admin/dist/` тАФ a **gitignored build artifact** produced fresh at CI/deploy time (the esbuild bundles embed environment-specific module paths, so a committed copy isn't reproducible) тАФ served via the Workers `assets` binding; SSR pages render in the Worker at request time. It is the operator **surface that runs with no per-tenant OAuth session** (alongside the scheduled cron jobs and `email()`): deliberately cross-tenant, gated instead by **Cloudflare Access** on `/admin*`. The Worker verifies the injected `Cf-Access-Jwt-Assertion` (signature via the team JWKS + audience) as defense-in-depth and sets `workers_dev:false` to close the `*.workers.dev` bypass; the surface is **opt-in** (404 when `ACCESS_TEAM_DOMAIN`/`ACCESS_AUD` are unset). Two further guards close the "mis-set-up Access" gaps: an optional `ACCESS_ALLOWED_EMAILS` allowlist re-checks the verified `email` claim (so a too-loose Access policy or a wrong `ACCESS_AUD` can't admit a stranger), and the local-dev bypass (`ADMIN_DEV_BYPASS`) only engages on a **loopback** host so it is inert on any deployed Worker тАФ and `/health` surfaces the gate posture, screaming `exposed` (тЖТ 503 / red badge) if a deployment ever leaves that bypass as the surface's only safeguard. This is what keeps the minted invite code out of any git-hosted log тАФ it is shown **once** in the authenticated UI. The carve-out is surface-specific: the **MCP** surface keeps its own OAuth provider (Access is not its identity), consistent with multi-tenancy's "no Access on the MCP-surface identity." Revoke is **complete**: it scans `invite:*` to delete the member's code(s) without one being pasted, and deletes the per-tenant `kroger:refresh:<id>` token. The `assets` binding is on the deploy-merge allowlist (like `ai`), or it would be silently dropped from operator deploys.

The panel is **server-rendered, navigated by full page loads** тАФ each area is its own route, so it grows by adding pages, not cards. Top-level areas: a **Status** home (the `/health` service-health view: jobs, D1, admin-gate posture), **Members** (member management + the per-member Kroger consent-link mint), **Data** (the read-only data explorer), **Usage** (resource + per-job + per-tool dashboards), **Logs** (operator audit logs), and **Config** (the discovery-sweep calibration console тАФ a cheap no-AI threshold analysis and a no-write full-pipeline dry-run тАФ plus the ranking/flyer weights and the shared-corpus editors). Read-heavy areas (Status, Data, Usage) are **pure SSR** тАФ the page calls the `src/` reader and renders, no client fetch, the data in the first paint; interactive areas (Members, the Logs row-actions, the Config forms) add an **island** (`hono/jsx/dom`) that calls the typed `/admin/api/*` routes via `hc`, each route calling the same `src/` function the SSR path does. The **Data** explorer is cross-tenant introspection over D1 and the R2 corpus across five entity views тАФ a per-slug **Recipe** view joining the R2 source тЖФ the `recipes` projection тЖФ `recipe_derived` тЖФ `reconcile_errors` into one projection status (`indexed | skipped | pending | orphaned`), a per-tenant **Member** 360, and flat browsers for the shared-corpus / discovery / operational tables (incl. `bug_reports`) plus the `guidance/**` markdown; behind `src/admin-data.ts`, SSR'd with query-param navigation, crossing every member's domain data with **no redaction** but no KV secret, reusing the canonical `profile-db`/`session-db`/`recipe-index` readers. The home route `/admin` is the Status view; members at `/admin/members`, the data explorer under `/admin/data*`, config under `/admin/config*`. The **Logs** area's first source is the **Discovery** log тАФ the discovery sweep's per-candidate outcome stream, served by the Access-gated `GET /admin/api/logs/discovery` reading the shared `discovery_log` table (most-recent-first, bounded) with the per-member attribution the rest of `/admin` can see; its rows hydrate an island for the per-row retry/delete actions. Any `/admin/*` request that matches no SSR route falls through to the `assets` binding (already past the Access gate, since `/admin*` is routed worker-first), so the islands + stylesheet are served gated too.

## The data model

D1 plus the R2 corpus is the system's memory. It splits two ways тАФ shared vs per-tenant тАФ and within a tenant, into a small set of intents that must not be conflated. Field-level schemas live in [`SCHEMAS.md`](SCHEMAS.md); this is the conceptual map.

### Storage tiers (the three-tier boundary)

Per `cloudflare-storage-architecture`, persistent state lives across three tiers chosen by the *nature* of the data, not by convenience:

- **R2** (`CORPUS`) тАФ authored **markdown** only: `recipes/*.md` (recipe *content*) and the `guidance/**/*.md` umbrella (`guidance/ingredient_storage/` тАФ curated, read-only put-away advice; `guidance/cooking_techniques/` тАФ agent-writable technique memories; `guidance/purchasing/` тАФ agent-writable buy-side selection advice; the writable two go through `save_guidance`). The source of truth for the human-authored corpus, read/written through `src/corpus-store.ts` and hand-edited via an Obsidian vault synced to the same bucket over R2's S3-compatible API. This is the one tier a human edits directly; everything else (shared corpus, attributed notes, profile, session, cooking log, recipe index) is D1. Git history for the corpus is a deliberately-accepted loss of this move off GitHub.
- **D1** (`env.DB`) тАФ all **domain/operational data and derived projections**: the queryable, relational, admin-editable, strongly-consistent (read-after-write) tier. The recipe index, profile, session state, cooking log, notes, registries, config, caches, and the `reconcile_errors` + `bug_reports` tables land here. Tools never touch `env.DB` directly тАФ they go through `src/db.ts` (prepared-statement helpers + structured-error mapping; tools never throw).
- **KV** тАФ **ephemeral infrastructure only**, no domain data: `KROGER_KV` (Kroger tokens, PKCE verifiers, the TTL flyer cache), `OAUTH_KV` (OAuth provider state), `TENANT_KV` (tenant directory / invites). The Worker binds only `KROGER_KV`, `TENANT_KV`, and `OAUTH_KV`. (Background-job health is operational data and lives in D1's `job_health` table, not KV тАФ persisting per-job liveness every cron tick is standing write load that belongs in D1's far larger budget.)

### Shared vs per-tenant

- **Shared corpus (D1)** тАФ objective, single-source, read by everyone: `aliases`, the location-tagged `sku_cache`, `stores` (registry identity), `flyer_terms`, and the discovery sources (`feeds`, `discovery_candidates` inbox, `discovery_senders`/`discovery_members` allowlist, and `discovery_rejections` тАФ the group-wide source-suppression set). Written + validated at the Worker write tools, read by query. (The curated `guidance/**/*.md` umbrella stays R2 markdown.) The recipe index is the derived D1 `recipes` table; the discovery sweep's group-wide `discovery_log` (audit + dedup + parked errors) and per-member `discovery_matches` (import attribution) + `taste_derived` (taste embedding) are sibling derived/operational tables.
- **Attributed records (D1, `recipe_notes` / `store_notes`)** тАФ each member's attributed recipe/store notes, in D1 tables with an `author` column + `private` flag (own-private + group-shared at read time). No per-tenant domain data lives in R2.
- **Per-tenant D1 (session state)** тАФ each member's working state in D1 row tables: `pantry`, `meal_plan`, `grocery_list` (keyed by normalized name or recipe slug). Adds are row upserts, removes/status changes are targeted row statements тАФ strong read-after-write consistency, no whole-array rewrite. The Worker read path has **no** GitHub/KV fallback (a miss returns empty).
- **Per-tenant D1 (records + profile)** тАФ the relational tier. The `cooking_log` table is per-tenant realized cook history; the `recipes` table is the shared objective recipe index; the **profile** tables (`profile`, `brand_prefs`, `kitchen_equipment`, `staples`, `overlay`, `ready_to_eat`, `stockup`) hold each member's preferences/taste/diet/kitchen/staples/overlay/ready-to-eat/stockup. Tenant-scoped on every read; written via `log_cooked` (cooking events тАФ which also clears the cooked recipe from `meal_plan` in the **same transaction**), the scheduled recipe-index projection, the profile write tools, and the session-state tools (`update_pantry`, `update_meal_plan`, `add_to_grocery_list`, тАж).

### Three-category recipe model

A recipe splits three ways so a shared corpus is safe to share:

- **Content** тАФ objective frontmatter + body, shared and single-source.
- **Overlay** тАФ the caller's two mutually-exclusive disposition marks `favorite` (loved) + `reject` (hidden-from-me), per-tenant in the D1 `overlay(tenant, recipe, favorite, reject)` table. One member's disposition never changes another's. Visibility is **opt-out**: a recipe with no overlay row is **neutral (available)**, so the candidate set is the whole shared corpus minus the caller's rejects. `reject` is a **hard gate** тАФ a rejected recipe is dropped from that member's `search_recipes` results entirely (both membership and ranked modes). The group-favorites signal (`read_recipe_notes`) is a single indexed query тАФ `SELECT tenant, favorite FROM overlay WHERE recipe=?` scoped to the caller's group, counted.
- **Notes** тАФ per-tenant, attributed, append-mostly (D1 `recipe_notes` table, keyed by `(tenant, recipe)`).

`last_cooked` is **not stored** тАФ it's derived per-tenant from that member's D1 `cooking_log` rows (`MAX(date)` per recipe). Read tools merge shared content + the caller's overlay + cooking-log `last_cooked` at read time; the shared D1 `recipes` table carries objective fields only.

**Notes are the spin-capture mechanism that makes sharing safe.** A tweak ("sub gochujang for the sriracha") is an attributed note, never an edit to shared content; only a genuinely *different dish* warrants importing as a separate corpus recipe. The shared body changes only for an objective correction. Group notes/favorites aggregate across members at read time (`read_recipe_notes`).

### The intent model (per-tenant)

Five intent kinds тАФ don't conflate them:

| Key / backing | Kind of intent |
| --- | --- |
| `pantry` (D1, tenant-scoped) | **observation** тАФ what's physically in the kitchen |
| `stockup` (D1, tenant-scoped) | **conditional intent** тАФ buy IF it drops below a threshold |
| `grocery_list` (D1, tenant-scoped) | **committed buy intent** тАФ buy on the next order (ingredient-level, SKU-free) |
| `meal_plan` (D1, tenant-scoped) | **committed cook intent** тАФ recipes agreed to cook next (transient) |
| `cooking_log` (D1, tenant-scoped) | **realized history** тАФ append-only log of meals actually cooked |

D1 row tables are freely mutable with strong read-after-write consistency (appropriate for operational data); the `cooking_log` D1 table is the durable, queryable cook history (`id`-addressable, read-after-write consistent). The agent **captures intent into the grocery list continuously**, and **flushes to the cart once**, at order time. Capture is store-agnostic (the list is SKU-free); the flush is not.

### The flush branches (`shop-groceries`)

Capture is identical regardless of where the user shops; only the flush differs, detected by the `shop-groceries` skill from the D1 `profile` row's `stores.primary` preference and trip context:

- **Kroger online** (`primary: kroger`, no in-store trip) тАФ `place_order` resolves the whole D1 `grocery_list` against current Kroger availability, surfaces ambiguous/unavailable items as one batch, writes the Kroger cart, and appends learned SKU mappings to the D1 `sku_cache`. D1 is the mutable store (capture continuously); the cart is append-only (flush once).
- **Kroger in-store** (`primary: kroger` + in-store trip, or named Kroger location) тАФ uses the Kroger Products API's `aisleLocation: { number, description, side? }` field (returned by `kroger_prices`) to order the list by aisle number automatically тАФ no pre-mapped layout required. After the first visit, the store's slug and Kroger `locationId` are registered in the D1 `stores` table (`location_id` field); `resolveLocationId` in `src/kroger.ts` detects a no-space `location_id` string and returns it directly, bypassing the Locations API on every subsequent walk. Items with `inStore: false` are surfaced before the walk (not silently dropped); `location`-tagged store notes are seeded silently and idempotently after each walk.
- **In-store walk** (`primary` is a non-Kroger store slug, or named non-Kroger store) тАФ reads the same list and groups it for the store, walked hands-free one aisle at a time. Degrades gracefully: no map тЖТ a department-grouped list from world knowledge; a mapped store тЖТ aisle-by-aisle from its `layout` notes, with `location` notes pinpointing the tricky items. On completion, picks received directly from `active` (no `in_cart`/`ordered` stage) тАФ removing them and restocking the pantry, the same end-state as a Kroger pickup. A first visit to an unmapped store offers to record the layout (as `layout`-tagged store notes) *while* shopping.

The D1 `stores` table holds store **identity** per *location*, including the optional `location_id` (chain-specific external id тАФ for Kroger, the `locationId` that bypasses the Locations API); the **layout** lives in attributed per-tenant D1 `store_notes` rows тАФ aisle order (`layout` tag), where-it-hides hints (`location`), and not-carried entries (`stock`) тАФ so mapping a store once helps the whole group, and an author can correct their own notes (`update_store_note` / `remove_store_note`). Each grocery-list item carries a `domain` facet (default `grocery`) so a non-grocery run (e.g. Lowe's) filters the list for free.

## Kroger product matching (ingredient тЖТ SKU)

The hardest deterministic problem: turning a recipe ingredient string ("extra virgin olive oil, 1 tbsp") into a specific Kroger SKU. It lives entirely inside `match_ingredient_to_kroger_sku`. The pattern is **progressive deterministic narrowing, with LLM fallback only when ambiguity remains.**

1. **Normalize** тАФ strip quantity/units, lowercase, apply the D1 `aliases` table. Alias-driven, *not* an aggressive qualifier-stripper: the `aliases` table is the curated source of truth for which variants collapse to which canonical term.
2. **Cache lookup тЖТ revalidate** тАФ if a normalized term тЖТ SKU mapping exists in the D1 `sku_cache`, take that SKU and revalidate it with one targeted lookup (current price + curbside/delivery availability at the preferred location). Available тЖТ use it with fresh price/promo; unavailable тЖТ treat as a miss and fall through to search (self-healing). Every hit is revalidated, so there is no TTL. The cache short-circuits the expensive search, not the price check. The LLM may pass `bypass_cache` when a cached generic doesn't fit the recipe context.
3. **Kroger search** тАФ `filter.term` + `filter.locationId` (+ fulfillment) тЖТ candidate products with price, size, brand, `aisleLocation`, and `inStore`. `resolveLocationId` is called to resolve the `locationId` from the user's `preferred_location` label тАФ if the D1 `stores` row has a `location_id` with no spaces, it is treated as an already-resolved `locationId` and returned directly without a Locations API round-trip.
4. **Score candidates** (rule-driven scoring, *not* hard filters) тАФ brand preference from the D1 `brand_prefs` rows; dietary as a soft score. Two near-hard constraints govern *which product*: **availability** (must be fulfillable via curbside/delivery) and **identity relevance** (how many query tokens appear in the product description/categories). A confident pick comes only from the top relevance tier, so "anaheim peppers" resolves to the Fresh Anaheim Peppers PLU, never a cheaper unrelated fulfillable item. If nothing shares any query token, the matcher returns ambiguous rather than guess. Scoring (not filtering) means a missing preferred brand can't empty the set. This step does **not** substitute.
5. **Deterministic tiebreaker** (within the top-scoring set) тАФ prefer on-sale, then best price-per-unit (deterministic arithmetic; the LLM only normalizes messy size strings, never does the math); "don't care" commodities take the smallest package covering the quantity hint, then cheapest.
6. **Confidence gate тЖТ LLM only when ambiguous** тАФ **confident** (auto-pick): a cache hit, or a defined brand preference resolves it (including `[]` = "don't care, cheapest acceptable"). **Ambiguous**: no cache hit *and* no defined brand preference тЖТ return narrowed candidates and let Claude pick from context or ask.
7. **Cache result** (persisted at order time) тАФ the resolved mapping is upserted into the D1 `sku_cache`; the matcher itself only resolves, and the cache write rides `place_order`'s flush.

**Confidence is legible and self-extinguishing.** It comes entirely from the D1 `brand_prefs` rows, which are **tri-state**: row absent тЖТ ask; `[]` тЖТ "don't care," cheapest acceptable; `["A","B"]` тЖТ ranked preference. Every answered question caches, so it asks less over time тАФ after a few weeks of use, most common ingredients are cached and never hit the LLM. **Substitution is a separate, confirmed step** тАФ LLM reasoning, not a tool: inventory subs are judged over the loaded pantry, sale/unavailable subs are enumerated from world knowledge and resolved as ordinary Kroger searches, and either is surfaced for the user to confirm. The matcher itself never substitutes. Quantity translation is intentionally coarse ("3 cloves garlic" тЖТ buy a bulb); pantry tracking absorbs the slack.

## The flyer warm (scheduled capture)

The public Kroger API has no flyer/circular endpoint, so the "what's on sale" list is **synthesized** by searching curated broad terms (the D1 `flyer_terms` table) and keeping the genuine discounts. Doing that live inside `kroger_flyer` fanned one search per term and ran into the Cloudflare Workers **free-tier cap of 50 external subrequests per invocation** as the term set grew тАФ plus multi-second latency on the user's hot path. So the fetch runs in a scheduled **cron** (`src/flyer-warm.ts`, the `scheduled()` handler in `src/index.ts`), and `kroger_flyer` is a pure KV read.

- **One trigger, a cursor sweep.** A single cron fires on a short cadence (every few minutes). Each tick reads a small `flyer:cursor`, processes the **next bounded batch** of `(location, term)` units (sized to stay under the 50-subrequest *and* ~10ms-CPU per-invocation caps), advances the cursor, and **no-ops** once the sweep is complete тАФ until the daily refresh window re-arms it. The total term set is unbounded: more terms just mean more ticks, never a bigger invocation. The murky free-tier cron-*count* limit never bites because there is exactly one trigger.
- **Plan built once, persisted.** Enumerating the work (the tenant directory + each tenant's D1 `profile` row + the D1 `flyer_terms` rows) happens **once at sweep start** and the plan is persisted in `flyer:plan`; every later tick reads the plan from KV (a CF-services read, not an external subrequest) and spends its budget only on Kroger scans.
- **Per-location rollup, noise floor at warm / deal floor at read.** Results are materialized as one `flyer:{locationId}` rollup of fulfillable, on-sale candidates (raw `regular`/`promo` kept). `kroger_flyer` applies the caller's `min_savings_pct` at read, so the deal threshold stays tunable without a re-fetch. A cold cache reads as empty (graceful), and an `as_of` timestamp conveys age тАФ staleness is low-stakes because the order path re-prices live.

## The recipe-index projection (scheduled capture)

With the corpus in R2, the recipe **index** is projected by the Worker on a schedule: a job in the `scheduled()` handler (`src/recipe-projection.ts`, `runProjectionJob`) that holds the whole corpus each pass.

- **Whole-corpus read, validate, project.** Each tick reads the **entire** R2 corpus through `src/corpus-store.ts`, validates every recipe тАФ the shared `recipe-contract.js` required-field/vocab contract, the body `## Ingredients` / `## Instructions` sections, a duplicate-slug guard, and **cross-corpus `pairs_with` slug resolution** (possible because the reconcile holds the whole corpus at once) тАФ then rebuilds the D1 `recipes` table **wholesale** (`DELETE` + batched `INSERT` in one D1 transaction). A *derived* projection, rebuilt each pass, not a stateful migration.
- **Invalid recipes are skipped, not fatal.** A recipe that fails validation is **left out of the index** (not committed) and recorded to the D1 `reconcile_errors` table, so one malformed file can't break the whole rebuild. This is the system's eventual-consistency model тАФ see *eventual human-edit feedback* below.
- **Runs before the recipe-derived reconcile** so that pass reads a fresh index; the flyer warm is independent of the index and runs alongside. A `recipe-index` health job reports the run (its skipped-invalid count in the `summary`).

## The recipe-facet classify pass (scheduled capture)

The same `scheduled()` handler runs a **classify pass** (`src/recipe-classify.ts`, `runFacetJob`) that generalizes the discovery sweep's classifier (`src/discovery-classify.ts`) over the **whole corpus**: it derives the descriptive recipe facets тАФ `protein`, `cuisine`, `course`, `season`, `tags`, `ingredients_key`, `perishable_ingredients`, `side_search_terms`, `meal_preppable` тАФ from each recipe **body** into the `recipe_facets` table (migration 0018). It runs **first in the tick, before the index projection**, so the projection can materialize the **effective** facet (authored override ?? classified; `tags` unioned; the two hard gates `dietary`/`requires_equipment` stay authored) into `recipes` and every reader is unchanged. It reads R2 bodies directly (a binding call, off the 50-subrequest budget; the internal `env.AI`/D1 bucket like the describe pass), is **`body_hash`-gated** (reclassify only on a body or override change тЖТ steady state тЙИ 0 work) and **bounded per tick by both a small count cap and a wall-clock budget** тАФ classify is the tick's first phase and blocks the projection/embed/discovery jobs after it, so a large backlog (e.g. a bulk re-derive) is worked down a few recipes per tick rather than overrunning the scheduled invocation's resource budget and starving those jobs тАФ and writes a `recipe-classify` health record. **Failures are handled by kind:** a **transient** `env.AI`/storage error (including Workers AI quota exhaustion) leaves the gate **un-advanced** тАФ the recipe retries on a later tick and the projection keeps merging its authored frontmatter meanwhile тАФ while only a **permanent** contract `validation_failed` parks the recipe (gate advanced with empty facets) so it isn't re-spent; a quota (4006) error stops the tick early and raises the `/health` quota signal (below). This applies the **placement rule** (below) to the facets: most were AI-classified-at-import and frozen into frontmatter; now they are regenerable D1-derived data, with an optional authored override for the Tier-B set. See the `recipe-facet-derivation` capability.

## The recipe-derived reconcile (scheduled capture)

The same `scheduled()` handler runs another job each tick (`src/recipe-embeddings.ts`), reconciling the `recipe_derived` table (migration 0013) in **two passes**, after the index projection and before the discovery sweep (which matches against the embeddings it produces). The **placement rule** behind it (and behind the classify pass above): a recipe field lives in frontmatter only if a human authors or corrects it; a purely-derived, regenerable field lives in D1. The `description` is exactly that тАФ AI-written, not authored тАФ so it leaves frontmatter for the reconcile-owned `recipe_derived` table, co-located with the embedding it feeds (the two derived halves of one artifact, one producer, one cadence, one `slug` key). The describe pass reads the recipe's **effective facets from the D1 index** (authored-or-derived), still **not the body** тАФ the classify pass is the one derive step that reads the body.

- **(1) Describe.** Generate the `description` from the recipe's authored **facets** (title, ingredients_key, course, protein, cuisine, time_total, dietary, season) via `env.AI`, gated by a `content_hash` over those facets тАФ pure D1 + AI, **no body read**. Steady state тЙИ 0 work.
- **(2) Embed.** Embed any description whose vector is new/changed (a `description_hash` gate) and prune rows whose slug no longer exists in `recipes`. A freshly-described recipe flows into the embed pass the same tick.

Both passes need `env.AI`, so they run Worker-side. The `description` is also **seeded synchronously at import** (`create_recipe`) so a new recipe reads well before the next tick (the reconcile stays the authority and refreshes on a facet change). This is *capture тЖТ retrieve тЖТ narrow*: capture the description + embedding once, `search_recipes` (ranked mode) retrieves by cosine, the planning skill narrows.

- **It is *not* the flyer's free-tier problem тАФ a different subrequest budget.** The flyer became a cron because Kroger fetches exhaust the **50 external-subrequest** cap. `env.AI` is an **internal Cloudflare-services** call (the **1,000**/invocation bucket, shared with D1), a different budget entirely тАФ so the reconcile coexists with the flyer in one tick without competing for the 50. It is also far lighter than the flyer it rides beside: embedding **batches** (`embedTexts` = one subrequest for a whole chunk, where each Kroger term is an irreducible separate fetch) and is **change-driven** (the hash gate тЗТ steady-state тЙИ 0 work). It still **bounds work per tick** (`RECONCILE_MAX_PER_TICK`, deferring the rest to later ticks) тАФ for the 1,000 cap, Workers AI's own rate limit, and tidy wall-clock тАФ keeping the flyer's bounded-batch discipline under the **one** cron trigger.
- **Separate table, not a `recipes` column.** The vector has a different producer and cadence than the rest of the row (the recipe-derived reconcile vs the recipe-index projection, though both run Worker-side in `scheduled()`); a sibling table keyed by `slug` lets each rebuild independently, so the projection's wholesale `DELETE FROM recipes` + re-INSERT can't clobber a vector it doesn't own. Search JOINs the two тАФ facet-prefilter on `recipes`, cosine over the joined vectors. The cost is a bounded reconcile lag: a just-imported recipe is unembedded until the next tick (treated as "not yet indexed," not an error).

## The discovery sweep (scheduled capture)

Unprompted recipe discovery is the **fourth** scheduled job in the one `scheduled()` handler (`src/discovery-sweep.ts`, `runDiscoverySweepJob`) тАФ autonomous, not an in-chat pull. It turns "find me something new" into a background pipeline that classifies, taste-matches, and **auto-imports** candidates into the shared corpus, attributed per member, so the agent reads ready-made results (`list_new_for_me`) at plan time instead of fetching/triaging/parsing in-flow. It runs **last in the tick** (phase 3, after the index projection and the recipe-derived reconcile) so its dedup and matching see a **fresh index *and* fresh embeddings**. The key pipeline knobs тАФ `tasteThreshold`, `triageThreshold`, `dedupThreshold`, `classifyMaxPerTick`, `rateCap` тАФ are read at job start from the `discovery_config` D1 singleton (migration 0017), a sparse operator override merged over `DEFAULT_CONFIG`, so the compiled defaults are the safe starting point and the operator calibrates from there via the **Config** area of the admin panel (a cheap no-AI threshold analysis + no-write full-pipeline dry-run тАФ the sweep's built-in E2E verification).

- **Intake тЖТ dedup.** Poll the shared `feeds` and read the email `discovery_candidates` inbox (extracting recipe links from each body), then **dedup** every candidate URL against three sets at once: the corpus `source_url`s, the `discovery_rejections` suppression set, and the `discovery_log` **"already evaluated"** set (any prior outcome for that URL). The recorded outcomes are the sweep's candidate progress state тАФ there is no separate candidate cursor, and a re-run never reprocesses a handled candidate. (Feed *polling* is separately bounded by a small rotation cursor тАФ see *two budgets* below тАФ but that governs only how many feeds are fetched per tick, not candidate dedup.)
- **Cheap triage before the expensive legs.** Embed `title + summary` and cosine it against each member's favorites + taste vector; a candidate **near nobody** (below a loose triage threshold) is dropped *before* a fetch or a classify is spent. The blurb vector is low-fidelity, so the triage threshold is deliberately looser than the real match threshold. The pool is first clamped to `maxCandidatesPerTick` (the overflow defers), then its triage embeds run as **batched `env.AI` calls chunked at `EMBED_INPUT_BATCH`** (`embedTexts`, the same input ceiling the recipe-derived reconcile uses) тАФ `ceil(N/BATCH)` calls, not one subrequest per candidate (which would exhaust the budget on triage alone) and not one oversized call (which would exceed the model's input limit). A chunk failure fails the tick (nothing logged тЖТ the pool re-gathers next run).
- **Classify (env.AI) with a contract backstop.** Fetch + parse the survivor to structured content via the shared `acquireRecipeContent` helper (`src/recipe-acquire.ts`) тАФ the **same** fetch тЖТ JSON-LD тЖТ `Recipe` тЖТ normalize pipeline `parse_recipe` uses, returning a discriminated `{ ok } | { reason }` rather than a bare null, so a park records the **specific** reason (`unreachable` with the HTTP status / `no_jsonld` / `not_a_recipe` / `incomplete`) and a walled/dead source is distinguishable from a feed entry that simply isn't a recipe. (Every server-side fetch here тАФ the sweep's, `parse_recipe`'s, and the operator probe's тАФ runs through one hardened egress primitive, `fetchWithBrowserHeaders` (`src/http.ts`): public-`http(s)`-only with no userinfo or private/loopback/link-local host, manual per-hop redirect re-validation, a per-request timeout, and a capped body read; a refusal surfaces as a plain `unreachable` with no status, so untrusted feed/email content can't steer an internal fetch or use the outcome to probe reachability.) Then classify it into the controlled-vocab facets `create_recipe` requires on the small classifier model, validated against the shared `recipe-contract.js` with a **corrective retry** (echo the validator's complaints back to the model). A classification that can't pass the contract within the retry budget **parks** the candidate (no human is present to fix it).
- **Describe + embed, then semantic dedup.** Generate the `description` from the classified facets (the same tuned `generateDescription`) and embed it тАФ the **authoritative** vector for dedup and match. Drop the candidate as a **duplicate** when its description vector is within ╬┤ (cosine) of any corpus `recipe_derived` vector, or of a recipe imported earlier **this same tick**.
- **Hybrid taste match тЖТ small-LLM confirm.** Score the description vector against each member (max cosine over their favorites and their `taste_derived` vector); keep members clearing the match threshold ╧Д, **repel** a near-duplicate of something a member rejected, and apply each member's **hard dietary gate**. Then a **negation-aware confirm** on the classifier model asks which of the cosine-matched members *genuinely* fit given their stated dislikes (a member with no taste text is kept; the confirm **fails open** to the cosine matches on an AI error тАФ a missed negation is one `toggle_reject` away).
- **Governed auto-import + attribution.** A candidate with тЙе1 confirmed member is imported (assemble body + frontmatter, full file validation, R2 write, seed the description) under a per-tick **rate cap** (the corpus-bloat governor); candidates beyond the cap (or the per-tick classify cap) are **deferred** and re-gathered next tick (still un-evaluated, so no dedup needed). Each import records a `discovery_matches(recipe, tenant, score)` row per confirmed member тАФ the per-member attribution that is **both** the import gate (тЙе1 row тЗТ import) and the `list_new_for_me` filter (so the shared corpus never floods one member with the group's combined discovery).
- **Every outcome is logged.** Each terminal outcome тАФ `imported | duplicate | no_match | rejected_source | dietary_gated | error | failed` тАФ is appended to `discovery_log` (one table: the operator audit log, the dedup evaluated-set, and the parked/failed surface read by `read_discovery_errors`), pruned under a retention window. `error` is a **content park** (an un-importable page тАФ its `detail.reason` names *which* of `unreachable`/`no_jsonld`/`not_a_recipe`/`incomplete`/a validation message; expected steady state); `failed` is an **infrastructure failure** (a transient env.AI/D1 error тАФ a subrequest-limit hit, an outage). The split is what lets health degrade on `failed` without crying wolf on every benign `error`.
- **Automatic retry for transient parks.** The two retryable outcomes тАФ `error` with `reason = 'unreachable'` and `failed` тАФ are re-admitted into each cron tick as a bounded retry sub-stream (capped at `retryFetchMaxPerTick`, independent of fresh intake). Rows carry `attempts` (how many acquisition passes so far) and `next_retry_at` (when the row next enters the retry stream); `NULL` is terminal. Each re-failure bumps `next_retry_at` along an exponential backoff schedule (`retryBackoffMinutes: [60, 360, 1440, 4320]`); at the cap (`retryMaxAttempts = 5`) the row terminalizes to `outcome = 'error'`, `next_retry_at = NULL` so `/health` clears (the `countDiscoveryFailures` query counts `outcome = 'failed'` only). Rows in `discovery_rejections` are excluded from the retry stream, so a rejected URL is never re-admitted.
- **Admin per-row retry and delete.** The admin **Logs тА║ Discovery** view shows per-row **Retry** and **Delete** buttons for `error`/`failed` rows. `POST /admin/api/discovery/:id/retry` runs the full acquisition pipeline once (bypassing the attempt cap тАФ operator intent overrides), then returns the updated row. `DELETE /admin/api/discovery/:id` writes a permanent group-wide rejection (`discovery_rejections`) for the URL then removes the row тАФ the rejection gates future retries and fresh intake for that URL.
- **Operator feed-probe.** Because a feed can fetch `200` yet have every entry walled, the operator can test a feed **from the Worker's edge** before/after adding it: `POST /admin/api/discovery/test-feed` (`src/discovery-probe.ts`, surfaced as a *Test* button in the admin Config тА║ Feeds editor) fetches the feed and runs `acquireRecipeContent` against a sample of its entry pages, reporting each page's outcome in the **same** taxonomy the sweep parks with тАФ so "is this actually a viable source from the edge" is answerable without waiting for a cron tick.
- **Two budgets, like the other crons.** Feed/page fetches are **external subrequests** that share the **same 50-cap per-invocation budget as the flyer** (both run in one `scheduled()` tick), so the recipe-page fetches are bounded directly by `fetchMaxPerTick` тАФ counted on every fetch, since a parse failure spends one *without* ever reaching classify (which is why the fetch cap, not `classifyMaxPerTick`, is the page-fetch governor; `fetchMaxPerTick тЙе classifyMaxPerTick`). **Feed** fetches are bounded the same way: rather than fan out one fetch per feed every tick (which, as the add-only feed set grows, blew the same 50-subrequest budget the flyer was redesigned to escape), the sweep polls a **rotating bounded batch** of at most `feedFetchMaxPerTick` feeds advanced by a persisted KV cursor (`discovery:feed-cursor`), so the feed set grows without ever enlarging one invocation тАФ the residual budget is `flyer(~25) + recipe-page(fetchMaxPerTick) + feed(feedFetchMaxPerTick) тЙд ~50`. `feedFetchMaxPerTick` is a constant guardrail (like the retry sub-budget), not an operator knob. Email intake also drops obvious non-recipe links and caps links-per-email so a newsletter's junk tail can't flood the fetch budget. The classify/describe/embed/confirm calls plus the chunked triage embeds and the per-candidate log writes draw on the separate **internal `env.AI`/binding** bucket (the 1,000-cap shared with D1/KV) тАФ which `maxCandidatesPerTick` + `EMBED_INPUT_BATCH` chunking keep bounded per invocation (the original overrun was an unbounded pool spending one embed + one log write apiece). Sizing the caps against the *combined* budget is the deferred calibration task (0.3). The `discovery-sweep` health record carries the counts summary and sets **`ok: false` while any `failed` row stands unresolved** (`countDiscoveryFailures`), so `/health` shows the degradation even on a later idle tick тАФ not just on the tick that failed; a hard (thrown) tick rethrows (cron isn't retried), and a best-effort ntfy heads-up fires on standing failures (and, separately, when candidates park).

## Semantic recipe selection

Recipe **selection** is the retrieval analog of the classify-once pattern above. Instead of loading the whole corpus into the model and reasoning over it ("dump-and-reason"), the meal-plan flow **distills** the request into a few search specs, **retrieves** by meaning, and **composes** the plan from the much smaller returned set. A vibe-bearing `search_recipes(specs[])` spec is the retrieve leg: per spec it **facet-prefilters** the index (the same `filterRecipes` gate the membership mode uses тАФ diet, makeability, anti-recency), then **cosines** the query embedding against the surviving recipes' vectors, then **re-ranks** with two small Worker-side nudges тАФ *nearest-liked* (max cosine to a favorited recipe; taste **direction**) and *freshness* (never-cooked up, recently-cooked down; **rotation**, tuned by the `rotation` preference). Hard constraints stay in the prefilter, so semantic rank only reorders survivors тАФ it can never admit a recipe a gate rejected.

**The determinism boundary doubles as a token boundary.** The embedding and the cosine run *in the Worker*, not the model: the skill ships a **query string**, never vectors, and gets back **compact candidate rows** (`slug`, `title`, the ~60-token `description`, key facets, score) rather than the full metadata of the whole corpus. So the same boundary that keeps the hot path deterministic also bounds the *token* cost of selection тАФ it scales with the number of specs and K, not with corpus size. The AI-written `description` is the linchpin: generated by the reconcile into `recipe_derived` (and seeded at import), it is simultaneously the embed source, the compact per-candidate context rep, and the user-facing "why this dish."

**Brute-force cosine over a D1 column, not Vectorize тАФ a measured, deferred promotion.** The vectors live in the reconcile-owned `recipe_derived` table; retrieval loads them through the Worker and cosines the facet-prefiltered survivors. At friend-group scale (hundreds, low-thousands of recipes) this is **exact** (not ANN-approximate) and a single store тАФ Vectorize's sub-linear ANN, server-side metadata filter, and managed scale buy nothing yet while costing a second, eventually-consistent copy to keep in sync. The contract is **backend-agnostic** (`search_recipes(specs) тЖТ ranked slugs + score`), so the swap is a tool-internal change skills never see. **Promote when *measured*:** a search that is measurably slow, OR loading embeddings through the Worker getting heavy (тЙИ low-thousands ├Ч 768 ├Ч 4 B). Runway before that bites: **int8-quantize** the stored vectors, or only ever cosine the **facet-prefiltered** subset (already the design). The written-down trigger keeps the decision data-driven, not guessed.

## Operator runtime configuration

Several knobs are tunable at runtime by the operator without a deploy, via the admin Config panel. Two singletons carry these overrides:

**`operator_config`** (D1 singleton, migration 0019) тАФ ranking weights and flyer settings. Loaded by `loadOperatorConfig(env)` (`src/operator-config.ts`), which reads the single row and merges over `DEFAULT_OPERATOR_CONFIG`: any null column falls back to the compiled default. Two subsystems consume it:

- **Ranking weights** тАФ applied in `resolveRankParams(prefs, operatorDefaults?)` (`src/semantic-search.ts`) with a three-tier precedence: *compiled defaults* тЖТ *operator_config* (the group baseline, set here) тЖТ *per-tenant `profile.rotation`* (individual override for `noveltyBoost` and `resurfaceAfterDays`). The six weight fields (`favoriteWeight`, `noveltyBoost`, `pantryWeight`, `perishWeight`, `keyWeight`, `overlapCap`) come from `operator_config`, falling back to the compiled constants. Loaded once per ranking tool call alongside the per-tenant preferences.
- **Flyer knobs** тАФ `minFlyerDiscount` sets the default savings threshold for `kroger_flyer` (a per-call `min_savings_pct` override sits above it); `flyerRefreshHours` and `flyerBatchUnits` gate the flyer warm's cursor plan. Both are loaded once at cron start.

**`discovery_config`** (D1 singleton, migration 0017 + 0020) тАФ discovery sweep calibration thresholds plus processing limits. Loaded by `loadDiscoveryConfig(env)` at job start; follows the same sparse-override pattern. Two groups of fields:

- **Calibration thresholds** (`tasteThreshold`, `triageThreshold`, `dedupThreshold`, `classifyMaxPerTick`, `rateCap`) тАФ tuned via the admin Config тА║ Calibration console with a cheap no-AI analyze preview and a no-write dry-run.
- **Processing limits** (`fetchMaxPerTick`, `maxCandidatesPerTick`, `retryMaxAttempts`, `logRetentionDays`) тАФ replace previously hardcoded constants in `src/discovery-sweep.ts`; tunable in the same Calibration console's "Processing limits" section.

The two singletons are independent: `operator_config` is read on the synchronous tool path (rankings, flyer reads); `discovery_config` is read only by background jobs. Neither is read speculatively тАФ each subsystem loads the config once, merges, and uses a typed `OperatorConfig` / `DiscoveryConfig` object with no further null-checks.

The five scheduled jobs (flyer warm, recipe-facet classify, recipe-index projection, recipe-derived reconcile, discovery sweep) and the inbound `email()` handler are the system's **background processes** тАФ they run with no user attached, so a failure has no in-band consumer (every synchronous tool failure surfaces to the user via Claude.ai; a 3am cron failure surfaces to no one). And the platform won't fill the gap: Cloudflare Cron Triggers have no retries and no failure alerts. The keystone failure тАФ *a stopped job emits nothing* тАФ is only detectable from **outside** the Worker.

- **Each background job upserts a `job_health` row** (keyed by job name) in **D1** per run (`{ ok, last_run_at, summary }`, tenant-data-free), through `src/db.ts`. The flyer warm, the recipe-facet classify pass (`recipe-classify` тАФ its `summary` carries the classified/parked/errored counts + a `quota_exhausted` flag), the recipe-index projection (`recipe-index` тАФ its `summary` carries the skipped-invalid count), the recipe-embedding reconcile, the discovery sweep (`discovery-sweep` тАФ its `summary` carries the per-outcome counts), and the email handler are the registered jobs (`HEALTH_JOBS`); a future cron rides the same convention with no new wiring. The five scheduled jobs share the one trigger and each writes its **own** row, so `/health` shows them independently even though one tick drives all of them. Health lives in D1 (not KV) because persisting per-job liveness on every cron tick is standing write load тАФ over the free-tier KV write budget on its own тАФ that belongs in D1's far larger budget.
- **`/health` aggregates them** on the **fetch** path тАФ deliberately, because `fetch` is independent of `scheduled`, so the endpoint stays answerable when the cron is dead, and an external monitor catches a stopped job via stale `last_run_at`. It is **open and tenant-clean** (no per-tenant data; the D1 probe is coarsened to a boolean so no raw `storage_error` string leaks; 200 when ok, 503 when a job is failing). Reading the `job_health` rows goes through `src/db.ts`; if D1 is unreachable the rows degrade to "unavailable" rather than throwing out of the health path тАФ the live D1 probe already reports `d1.ok: false`, which degrades overall `ok`. The payload also carries the **admin gate posture** as tenant-clean booleans (`access_configured` / `email_allowlist` / `dev_bypass_set` / `exposed`, never the allowlisted emails), computed from the same gate logic `requireAccess` uses; an `exposed` gate (dev bypass set without Access) degrades overall health (503 / red badge), the loud backstop for a mis-set-up deployment. The payload also carries an explicit **`ai_quota_exhausted`** boolean тАФ aggregated from the AI jobs' tenant-clean summaries (an explicit `quota_exhausted` flag or a 4006/"neurons" error string) тАФ that **names** Workers AI's daily-allocation exhaustion rather than leaving a generic job-fail; it degrades overall health, `/health.svg` renders an explicit `ai  quota exhausted` row, and the admin Status view banners it with the daily-reset / Workers-Paid remedy. The payload carries nothing secret, so restricting *who* may read it is an **edge** concern (Cloudflare Access / WAF), not Worker code тАФ the same "emit truthful state, decide policy outside" stance as the alerting. A sibling **`/health.svg`** renders the same payload as a README status-badge card тАФ **also open** (a public README badge must be anonymously fetchable), always 200 + `image/svg+xml` so an image proxy renders it (degraded shows by color), stamped into the data-repo README by the deploy. (There is no `HEALTH_TOKEN` тАФ health is non-sensitive, so both endpoints are public and the badge is a normal anonymous badge URL.) A third open, tenant-clean route тАФ **`/source`** тАФ serves the AGPL section 13 source offer: a static HTML page linking to the corresponding source (the operator's `SOURCE_URL` when they run a genuine fork, else the upstream code repo, which the standard template deploy runs unmodified). It is gateless by the same reasoning as `/health` тАФ it carries no tenant data, only the license and a source link.
- **The Worker stays alerting-agnostic** тАФ it *emits* truthful state; *what is alarming and who to notify* lives in an external monitor (point it at `/health`, route to ntfy). The one in-Worker exception is an **optional** secret-gated ntfy push (`NTFY_URL`) тАФ a failure-domain-independent backstop that fires from the edge even if the operator's monitor is offline. Both default off; unset means `/health` is disabled and no push, i.e. unchanged behavior.
- **`scheduled()` rethrows** a failed tick (cron is not retried) so Cloudflare's native Cron-Events status reflects failures rather than always-green. Rich diagnosis lives in Workers Logs тАФ queryable via the Cloudflare Workers Observability MCP.
- **Resource-usage observability** (`/admin/usage`, operator-only): the Usage view reads account-wide **KV-operation** and **Workers AI neuron** usage for the current UTC day тАФ against the daily free-tier limits тАФ from the Cloudflare **GraphQL Analytics API** (`src/usage.ts`, SSR'd into the Usage view). It is an outbound `fetch` that costs **zero KV** тАФ observing the budget must not consume it (the `KVNamespace`/`Ai` bindings expose no usage counters, and `KV.list()` key-counting would itself burn the list quota). Opt-in like the Access gate / ntfy: with `CF_ACCOUNT_ID` / `CF_ANALYTICS_TOKEN` (a read-only Analytics token) unset, the view reports **not configured** and makes no request. Tenant-clean тАФ account/namespace aggregates only, never a per-tenant identifier. KV rows are keyed by namespace **id** (a Worker can't map ids to binding names at runtime).
- **Usage trends тАФ the history tier** (`/admin/usage` trends panel, `src/usage.ts`): where the snapshot above is *today's account-level* number and the `job_health` D1 row is *current liveness*, the trends panel is *per-job history over time*. Each registered background job emits **one tenant-clean data point per run** to a **Workers Analytics Engine** dataset (`grocery_usage`, binding `USAGE_AE`) via `recordUsagePoint` (`src/health.ts`) тАФ job name, outcome, run duration, and the job's own summary counts, additive to the `job_health` upsert. The emission is **best-effort** (an unbound binding or a throwing `writeDataPoint` is a swallowed no-op, like the ntfy push) and **non-blocking**, drawing on **neither the KV nor the D1 budget** тАФ so it never re-introduces the standing write load that moved `job_health` out of KV. The dataset's blob/double slot layout is a **positional contract** (`docs/SCHEMAS.md`); a later change must not reorder existing slots. The panel reads the points back through the **AE SQL API** (`POST /accounts/<id>/analytics_engine/sql`) тАФ a *different* surface from the GraphQL Analytics API the snapshot uses (built-in datasets are GraphQL; custom AE datasets are SQL) тАФ reusing the same `CF_ACCOUNT_ID` / `CF_ANALYTICS_TOKEN`, performing no KV or D1, opt-in to the same not-configured degradation, and tenant-clean (per-job/per-day aggregates only). The AE dataset binding is **code-level** (no operator-owned id, like `ai`/`assets`): the deploy merge propagates it via the `scripts/merge-wrangler-config.mjs` allowlist тАФ a **new binding type**, added explicitly so it is not silently dropped from operators' deploys.
- **Tool usage trends тАФ the request-path history tier** (`/admin/usage` tool panel, `src/usage.ts`): the sibling of the per-job trends above, but for the **MCP tool surface** тАФ *which tools are called, how often, and how they perform*. Every tool call emits **one tenant-clean data point** (tool name, outcome `ok`/`error`, call duration) to a **second** Workers Analytics Engine dataset (`grocery_tool`, binding `TOOL_AE`) via `recordToolPoint` (`src/health.ts`). The emission is wired at **one seam**: a decorator (`src/tool-instrumentation.ts`) wraps `server.registerTool` once in `buildServer`, so every tool тАФ the inline registrations and those added by the `register*Tools` helpers, plus any future tool тАФ is instrumented with no per-tool wiring. The outcome is read from the tool's own MCP result (`runTool`'s `fail()` sets `isError`); a raw throw is recorded as `error`. Like the per-job tier it is **best-effort and non-blocking** тАФ fired after the result is computed, so it never changes a tool's result or adds latency тАФ and costs **neither KV nor D1 budget**. The panel reads it back through the **AE SQL API** as per-tool aggregates (call count, error count, p50/p95 latency), reusing the same `CF_ACCOUNT_ID` / `CF_ANALYTICS_TOKEN`, opt-in to the same not-configured degradation, and tenant-clean (per-tool aggregates only тАФ the tool name is a fixed enum, never a tenant id or a call argument). `TOOL_AE` rides the same `analytics_engine_datasets` allowlist line as `USAGE_AE` тАФ the merge copies the whole array, so the **second** dataset propagates without a new line; a regression test pins that both survive. The blob/double slot layout is the same kind of **positional contract** (`docs/SCHEMAS.md`), with `blob3` reserved for a future error code.

## Discovery and disposition

Recipe discovery is **autonomous**: the background **discovery sweep** (above) imports new candidates into the shared corpus on the cron, so by plan time the work is already done. Two recipe-discovery sources feed the sweep:

- **RSS feeds** тАФ recipe candidates from trusted blogs in the shared D1 `feeds` table; the sweep polls them each tick.
- **Newsletter email** (optional) тАФ a *push* source that reaches the bot-walled/paywalled sites RSS can't. The Worker exports an `email()` handler; Cloudflare Email Routing points a forwarder address at it. Emails are captured (body text, not pre-extracted URLs) in the D1 `discovery_candidates` inbox, and the **sweep reads that inbox** тАФ extracting recipe links from each body and running them through the same classify/match/import pipeline as a feed candidate. See [`SELF_HOSTING.md`](SELF_HOSTING.md) step 8.

The agent **reads the sweep's results** for the caller via `list_new_for_me` at plan time тАФ recipes already classified, embedded, and attributed to that member тАФ and folds them into the menu. There is no in-chat fetch/score/parse/import on the discovery path.

A separate, **buy-time** discovery rides the Kroger flyer: `kroger_flyer` surfaces ready-to-eat candidates during the sale pre-pass (deduped against the caller's `ready_to_eat` catalog, captured via `add_draft_ready_to_eat`). This is unchanged and independent of the recipe sweep.

**Disposition is a decision, not a lifecycle.** Once a recipe is in the corpus (imported by the sweep, or manually via `parse_recipe` + `create_recipe`), each member's standing toward it is the two-mark overlay: **`toggle_favorite`** (loved) or **`toggle_reject`** (hidden-from-me, a hard gate). An imported recipe a member doesn't want is just a personal `toggle_reject` away тАФ one member's disposition never changes another's.

**Group-wide source suppression is the collective curation surface.** A *source* being **not corpus-worthy for the group** (junk, broken, not a recipe, a feed producing off-base results) is a shared judgment, so it has its own surface: `reject_discovery(url)` writes the canonical URL to the shared `discovery_rejections` table, and the **sweep's intake dedup folds it in** тАФ a rejected source is never re-evaluated or re-imported for anyone. This is deliberately asymmetric with the per-tenant marks: group `reject_discovery` is collective curation of the noisy *intake* stream (pre-import, by source URL); per-tenant `toggle_reject` hides an already-imported corpus recipe for one member; favoriting is personal taste.

## Indexes and validation

The scheduled recipe-index projection (above) regenerates the derived tables from the R2 corpus:

- **The D1 `recipes` table** тАФ the recipe index, the shared objective projection of all recipe frontmatter (no per-tenant `favorite`/`reject`/`last_cooked`). The projection validates `recipes/*.md` and **projects** the valid set into D1, replacing the table wholesale in one transaction (`DELETE` + batched `INSERT`) тАФ a *derived* projection rebuilt each pass. The Worker reads it from D1 (`src/recipe-index.ts`) and filters in memory; discovery's source-URL idempotency check is an indexed lookup (`idx_recipes_source_url`). Recipe *content* (`recipes/*.md`) stays in R2; the derived index is in D1.

- **The D1 `cooking_log` table** тАФ per-tenant realized cook history. `last_cooked` and `retrospective` are SQL aggregations (the latter a `cooking_log LEFT JOIN recipes`). New events are appended via `log_cooked`, which validates the entry and resolves a recipe slug against `recipes` **at write time**.

Ready-to-eat is per-tenant, in the D1 `ready_to_eat` table; the Worker reads each member's catalog from D1.

**Validation runs through one validator.** Both the agent write tools (`src/validate.ts`, on `workerd`) and the projection share `src/recipe-contract.js` тАФ the sole recipe contract. The projection checks every recipe frontmatter is well-formed, `pairs_with` references resolve cross-corpus, and the body has its `## Ingredients` / `## Instructions` sections; `guidance/**/*.md` is structurally checked. The D1 profile tables + the D1 session-state tables (`pantry`/`meal_plan`/`grocery_list`) + the D1 `cooking_log` + the shared corpus (stores, aliases, sku_cache, feeds, discovery) are **not** corpus-validated тАФ they live in D1, not in R2; the Worker is their sole validator, at write time (`update_preferences`тАЩ merge-patch validation for preferences, `log_cooked` for the cooking log with real recipe-slug resolution against `recipes`).

**Eventual human-edit feedback.** A malformed Obsidian/R2 edit cannot fail a push тАФ there is no push. Instead the projection **skips** the invalid recipe (it stays out of the index) and surfaces it four ways: the D1 `reconcile_errors` table, the `/health` `recipe-index` job's skipped count, an agent-readable `read_reconcile_errors` MCP tool, and an ntfy push for each **new** invalid recipe. This is the system's accepted eventual-consistency model: a bad edit degrades that one recipe's discoverability until corrected, never the whole index, and the author learns out-of-band rather than from a build log.

**Client-side validation at the editing surface.** The eventual reconcile feedback is backstopped by a *fast* one at authoring time. The **authoring vault** is a generated, committed artifact (`scripts/build-vault.mjs` тЖТ `vault/`, from `vault-template/` + `src/vocab.js`, with a `--check` drift gate тАФ the same generated-from-source discipline as the plugin and admin-island builds, which are likewise rebuilt rather than hand-edited). Its Metadata Menu `recipe` fileClass binds each controlled-vocabulary facet (`protein` / `cuisine` / `season` / `requires_equipment`, plus the open `course` set) to a dropdown **generated from `src/vocab.js`** тАФ the same module `src/validate.ts` and `recipe-contract.js` enforce тАФ so the values an author can pick are, by construction, exactly the values the reconcile accepts, and an off-vocab token like `poltry` is never offered. This is a **convenience and a fast-feedback aid, not the gate**: an author can edit with plugins off or outside the vault, so the reconcile stays authoritative and is never bypassed by the dropdowns. The vault is an **author** tool (operator + co-authors); the friend read-path is the cookbook, not this vault. See [`SELF_HOSTING.md`](SELF_HOSTING.md) and the `recipe-authoring-vault` capability.

The hosted **cookbook** is served by the Worker (`src/cookbook.ts`, route `/cookbook`): the index lists from the D1 `recipes` table, each `/cookbook/<slug>` renders that recipe's R2 body followed by a **Similar Recipes** section (`src/cookbook-similar.ts`) тАФ the nearest recipes by cosine over the stored `recipe_derived` embeddings, computed at request time from those already-stored vectors (recipeтЖТrecipe, so no query embedding and no Workers AI call), behind a tunable similarity floor and omitted entirely when the recipe is unembedded or nothing clears the floor тАФ and `recipe_site_url` resolves `<origin>/cookbook`. Open + read-only, publishing only the shared objective corpus. A `?q=` runs **keyword search** ranked in the Worker (`src/cookbook-search.ts`): a field-weighted token scorer over the indexed metadata (title тЙл tags/`protein`/`cuisine` > `course` > ingredients/`dietary`/`season` > `description`), scaled by query coverage, with a whole-query title-prefix typeahead bonus тАФ score-descending, tie-broken on title then slug. Keyword search makes no query embedding and no Workers AI call тАФ the only recipe vectors the cookbook reads are for the Similar Recipes section above, itself AI-free; the agent-facing `search_recipes` keeps its own semantic ranking. The surface is anonymous, so none of `search_recipes`' per-tenant favourite/freshness/pantry boosts apply. Search is a **debounced client-side enhancement**: `/cookbook/search?q=` returns the ranked rows as JSON and the first-party `/cookbook/search.js` updates the list in place as you type, layered over a server-rendered `?q=` page that still works with JavaScript disabled (shareable URL, no-JS fallback) and uses the same ranker. The **CSP is split**: the recipe-body page keeps the strict no-script policy (untrusted markdown render), while the index/search page relaxes only to `script-src 'self'; connect-src 'self'` for the first-party script and its fetch.

## Security posture

- **The repo is public тАФ but only code lives here.** This repo is the Worker, the agent's persona/skills source, and build tooling; all personal data тАФ the operator's included тАФ lives in D1 (private, tenant-isolated, the operator is just another tenant), with the shared authored corpus in the operator's private R2 bucket. A public *code* repo collapses the auth story: the MCP read path leaks nothing not already public, so the security boundary moves cleanly to the **write + Kroger** path. The one genuinely-public read surface is the Worker-hosted cookbook (`src/cookbook.ts`, `/cookbook`), and it publishes only the **shared, objective recipe corpus** (`recipes/*.md`) тАФ never any per-tenant data or the per-tenant `favorite`/`reject` overlay. Eating habits, grocery cadence, and each member's `preferred_location` stay private in D1 with the rest of each member's state.
- **Secrets never touch the repo.** Because it's public, this discipline is load-bearing: the Kroger OAuth tokens live as Cloudflare Worker secrets only (encrypted at rest, never logged, gitignored locally via `.dev.vars`).
- **OAuth protects writes, not reads.** Claude.ai's custom-connector UI requires OAuth (no "no auth" / bearer option), and that OAuth guards the write/cart surface.
- **The cart is write-only.** The Kroger Cart API can add but cannot remove or check out тАФ so the agent literally cannot read the cart or check out for the user. A useful safety property: reconciliation reports what *should* change and tells the user to fix it in the Kroger app, never silently pretends items are gone.

## Tech stack

- **Claude.ai** (web + mobile) тАФ conversational surface, subscription auth, fresh-context conversations.
- **Cloudflare Workers** (TypeScript / `workerd`) тАФ hosts the MCP server + OAuth provider. Free tier handles personal-scale load. **Wrangler** for deploys; **KV** for OAuth/tenant/Kroger token state.
- **GitHub** тАФ the code (this repo) and each operator's data repo, which doubles as their **public plugin marketplace** (the deploy publishes the bundle there); CI/CD via Actions. Not on the data path (the authored corpus is R2).
- **Cloudflare R2** (`CORPUS`) тАФ the authored markdown corpus, read/written through `src/corpus-store.ts`; Obsidian authoring rides its S3-compatible API.
- **Kroger Developer API** тАФ product search, prices, cart writes (write-only).
- **Pure-JS parsers** that run on `workerd`: `js-yaml`, JSON-LD via `HTMLRewriter`, RSS/Atom via `fast-xml-parser`. (No `recipe-scraper`/`cheerio` тАФ they assume Node internals unavailable on `workerd`.)
- **Obsidian** (optional) тАФ recipe authoring + mobile viewing during cooking, pointed at a vault synced to the R2 corpus bucket.

## What this is тАФ and isn't

A personal automation experiment targeting a real friction point тАФ the time and willpower of grocery planning тАФ tuned to one person's tastes, freezer, and grocer, and shareable with a small friend group. Not a product, not a startup. The architecture is intentionally minimal: Anthropic provides messaging and reasoning, the Worker provides a domain interface, R2 holds the authored recipe corpus, D1 provides the operational data layer. Recipe files and narrative markdown are inspectable by humans (edited through an Obsidian vault) and outlive the agent if anyone stops using it.
